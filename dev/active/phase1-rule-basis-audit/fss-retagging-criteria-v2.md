# FSS 감리지적사례 재태깅 기준서 v2 (초안 — 사용자 승인 대기)

> 목적: `fss_case_combo_tagging.md`(구판, 폐기 예정)의 실패 원인이었던 "무엇이 추측인지 룰별
> 정의 부재"를 제거한다. 모든 룰 태깅을 **확정(E)/추정(I) 2등급**으로 분리하고, 룰별로
> E 기준·I 허용 조건·태깅 금지 예를 명문화한다. 이후 모든 집계는 "확정만 / 추정 포함"
> 두 수치를 병기한다(단일 수치가 검증 때마다 뒤집히는 문제 차단).

## 1. 2등급 태깅 원칙

| 등급    | 정의                                      | 요건                                  |
| ------- | ----------------------------------------- | ------------------------------------- |
| E(확정) | 원문 발췌 문언에 직접 근거가 있는 태깅    | 발췌 안에 근거 문구가 실제로 존재     |
| I(추정) | 문언엔 없으나 사실관계상 개연성 있는 태깅 | 추론 근거 1줄 필수 + §3의 I 허용 조건 |

- I 허용 조건이 §3 표에 없는 룰은 I 태깅 금지(E만 가능).
- 금감원 원문은 회계처리 위반 중심 서술이라 전표 메타데이터(승인자·입력시각·수기여부)를
  거의 쓰지 않는다. 메타데이터 계열 룰(L1-05·L3-02 등)이 구판에서 추론으로 오염된 이유이며,
  이 계열은 E가 드물고 I 조건이 엄격해야 한다.

- 태깅 대상(현행 PHASE1-1 canonical): L1-04, L1-05, L1-06, L1-07, L1-07-02, L1-08,
  L2-01, L2-02, L2-03, L2-04, L2-05, L3-02, L3-03, L3-04, L3-05, L3-06, L3-07, L3-09,
  L3-10, L3-11, L3-12, L4-01, L4-03, L4-04
- **사용 금지**: L3-01·L3-08·L1-09(폐기), L4-02·L4-05·L4-06(PHASE1-2 이관),
  L1-01·L1-02·L1-03(데이터 정합성 트랙 — 부정 조합 재료 아님). 구판은 폐기 룰
  L3-01·L3-08을 사용했으므로 이 항목만으로도 전면 재태깅이 필요하다.
- 순환거래 구조(A→B→C→A)는 룰 칸에 `PHASE1-2 family(graph)`로 기재(룰 id 아님).
- 전표와 무관한 사안(공시·주석·연결범위·감사절차)은 `N/A(전표무관)` + 사유. 억지 매핑 금지.

## 3. 룰별 태깅 기준표

표기: "E 기준" = 원문 발췌에 이 유형의 문언이 있을 때만 E. "I 허용" = 이 조건에서만 I(－는 금지).
"금지 예"의 ★는 구판에서 실제 발생한 오류 유형.

## 4. tier 판정 기준 (조합과 독립 — 순환논법 차단)

구판은 "기존 HIGH 조합에 해당하면 HIGH"로 tier를 매겼다. 그 tier로 조합을 검증하면
순환이므로, 이번 tier는 **원문 사실관계의 고의성·중대성만으로** 독립 판정한다.

| tier          | 기준                                                                                     |
| ------------- | ---------------------------------------------------------------------------------------- |
| HIGH          | 고의 문언 명시(허위·가공·조작·위장·은폐·횡령·위조·임의로) + 재무제표 왜곡 또는 자금 유출 |
| MEDIUM        | 고의 문언은 없으나 중대한 추정 왜곡·비합리적 가정·내부통제 중대 결함                     |
| LOW           | 단순 오류·회계정책 적용 오류·과실                                                        |
| N/A(전표무관) | 공시·주석·연결범위·감사절차 문제(전표 조작 없음)                                         |

## 5. 산출 스키마

사례 1건 = 1행. 컬럼(고정):

```
case_id | source_file | 위반 핵심사실(원문 발췌) | 6대패턴 | 룰(E) | 룰(I)+추론근거 | 연결주제 | tier | 근거
```

- 6대패턴: 가공전표/결산수정/횡령은폐/순환거래/승인SoD/비정상시점 (복수 가능, 구판과 동일).
- 연결주제: 기존 5주제(approval_control·closing_timing·account_logic·duplicate_outflow·
  revenue_statistical) 매칭, 없으면 `NEW:<이름>` (구판과 동일).
- 룰(I) 칸의 각 룰에는 괄호로 추론 근거 1줄 필수. 근거 없는 I는 검증에서 삭제된다.

## 6. 절차·검증 (태깅과 검증 주체 분리)

1. 태깅: 배치 에이전트가 230파일 전수(개별 PDF 84 + hwp 텍스트 146). 행마다 발췌 필수.
2. 기계 검증: (a) 230파일 전수 등장(파일별 grep, 누락 0) (b) 금지 룰 id 등장 0건
   (c) I 태깅 전건 추론근거 존재.
3. 에이전트 검증(태깅 주체와 분리): E 태깅 전건 — 발췌 안에 §3 기준 문언이 실제로
   있는지 판정. 위반은 삭제 또는 I로 강등.
4. 재현성 측정: 표본 50건을 독립 에이전트가 재태깅 → 룰 태깅 일치율 보고. 일치율이
   낮은 룰은 §3 기준을 보강 후 해당 룰만 재태깅.
5. 집계: 6대패턴·주제·tier별 건수를 "E만 / E+I" 두 기준으로 병기.
