## 1. 프로젝트 정의

`Local AI Audit Assistant`는 전표 데이터를 업로드하면

- 수집
- 헤더 탐지
- 컬럼 매핑
- 데이터 검증
- 특징 생성
- 룰 기반 탐지
- ML 보조 탐지
- 결과 탐색
- HITL 예외 처리
- 이력 복원

까지 한 흐름으로 이어지는 **로컬 기반 감사 데이터 분석 워크벤치**다.

단, 이 워크벤치는 fraud 여부를 자동 확정하지 않는다. 결과는 감사인이 검토해야 할 후보와 그 근거를 정리한 review queue로 해석한다.

이 프로젝트의 핵심은 “이상탐지 모델 1개”가 아니라 다음 세 가지를 하나의 제품 흐름으로 엮었다는 점이다.

- 실무형 파일 수집과 매핑 UX
- 회계/감사 관점의 다층 탐지 체계
- 탐지 이후 검토, 예외 처리, 재탐지, 이력 복원까지 이어지는 UI/DB 구조

## 2. 이 프로젝트가 보여주는 포트폴리오 가치

이 프로젝트는 단순 데이터 분석 데모보다 아래 성격이 더 강하다.

- 멀티 회사/멀티 연도 구조를 전제로 한 감사 분석 제품 설계
- ERP 추출물의 지저분한 현실을 견디는 ingest 계층
- 룰 기반 탐지를 중심으로 ML을 보조 레이어로 배치한 설계
- 탐지 결과를 검토 가능한 review queue로 설명하고 기록하는 인터페이스
- 합성 데이터 생성기(DataSynth)와 탐지 로직을 함께 맞춰 가는 데이터 계약 설계

즉 “모델 성능”이나 “review queue 농축도”만 보여주는 프로젝트가 아니라, **감사 데이터 분석 제품을 설계하고 구현한 결과물**에 가깝다.

### 2.1 데이터와 성능 주장 범위

| 데이터 | 이 문서에서의 역할 |
|---|---|
| DataSynth V7 fixed3 | 메인 데모. 복식부기 전표 구조, 계정, 차대변, 승인, 적요, 관계사, 의도 주입 truth를 갖춘 end-to-end 기능 시연용 |
| OpenDataPhilly | 실제 공개 지급 데이터 ingest와 review workflow smoke test |
| Tritscher ERP-Fraud | 외부 synthetic ERP에서 VAE/document ranking shadow 검증 |

DataSynth truth 기반 recall/precision은 개발 검증 보조값이다. 실제 회사 fraud 탐지 성능으로 표현하지 않는다.

## 3. 전체 사용자 흐름

### 3.1 회사와 감사 단위 선택

- 회사 프로필을 선택하거나 생성한다.
- 회사별로 산업, ERP, 커스텀 설정, CoA, 키워드, 감사 규칙을 분리 관리한다.
- 회사 안에서 engagement 단위로 연도별 분석 컨텍스트를 선택한다.
- engagement별로 DB, 모델, 이력, export 경로가 분리된다.

### 3.2 파일 업로드 또는 경로 입력

- 파일 업로드
- 로컬 경로 직접 입력

두 가지 방식을 모두 지원한다.

대용량 파일은 브라우저 업로드 대신 경로 입력으로 바로 읽을 수 있어 실제 로컬 분석 흐름에 더 가깝다.

### 3.3 헤더 탐지와 원본 미리보기

- 시트 추천
- 헤더 행 탐지
- 컬럼 목록 확보
- 원본 샘플 미리보기

를 먼저 수행하고, 사용자는 매핑 전에 데이터 구조를 눈으로 확인할 수 있다.

### 3.4 컬럼 매핑 검토

- 자동 매핑 제안 확인
- 필수/권장 컬럼 충족 여부 확인
- 이전 업로드와 컬럼 변경 비교
- 매핑 확정

을 거친 뒤 준비 단계로 진입한다.

### 3.5 준비 단계

- 타입 캐스팅
- L1~L3 검증
- 특징 생성
- EDA 프로파일 생성
- DB 적재 준비

를 먼저 수행한다.

### 3.6 탐지 및 재탐지

- Phase 1 룰 기반 탐지
- 점수 집계
- 위험 등급화
- 임계값/가중치 변경 후 재탐지
- Phase 2 보조 탐지 실행
- Review Queue 우선순위 산출

으로 이어진다.

### 3.7 결과 탐색과 예외 처리

- KPI
- 분포/품질 탐색
- 이상 전표 그리드 탐색
- 상세 설명
- 화이트리스트
- 피드백 이벤트 기록

까지 UI 안에서 연결된다.
이 단계의 핵심 문구는 "부정 확정"이 아니라 "감사인이 확인해야 할 후보와 근거"다.

### 3.8 배치 복원

- 이전 분석 결과를 DuckDB에서 불러와
- session state에 다시 복원하고
- 이어서 탐색할 수 있다.

## 4. 아키텍처와 구조

### 4.1 Company-Centric 구조

이 프로젝트는 초기 범용 구조에서 벗어나 **Company + Engagement 2계층** 구조로 재설계되었다.

- `data/companies/{company_id}/company.yaml`
- `data/companies/{company_id}/chart_of_accounts.csv`
- `data/companies/{company_id}/keywords.yaml`
- `data/companies/{company_id}/audit_rules.yaml`
- `data/companies/{company_id}/risk_keywords.yaml`
- `data/companies/{company_id}/profiles/`
- `data/companies/{company_id}/engagements/{engagement_id}/engagement.yaml`
- `data/companies/{company_id}/engagements/{engagement_id}/audit.duckdb`
- `data/companies/{company_id}/engagements/{engagement_id}/models/`

이 구조 덕분에 다음이 가능하다.

- 회사별 CoA 분리
- 회사별 키워드 사전 분리
- 회사별 감사 규칙 오버라이드
- 회사별 매핑 프로필 재사용
- 연도별 DB 분리
- 연도별 모델 저장소 분리

### 4.2 Context 기반 실행 구조

`CompanyContext`가 다음을 묶는다.

- company_id
- engagement_id
- settings
- schema
- keywords
- audit_rules
- risk_keywords
- cleaning_config
- chart_of_accounts
- profile_dir
- db_path
- model_dir
- fiscal_year
- materiality_amount

즉 파이프라인은 더 이상 전역 설정 하나에 기대지 않고, **회사/연도별 해소가 완료된 컨텍스트**를 기준으로 실행된다.

### 4.3 3계층 설정 합성

설정은 다음 순서로 해소된다.

- 글로벌 기본값
- 회사 오버라이드
- engagement 오버라이드
- 런타임 오버라이드

추가로 preset과 slider 변경은 최종 런타임 레이어에서 덮는다.

이 구조는 단순한 편의 기능이 아니라, 실제 감사 실무에서 회사별 승인 한도, 야간 기준, SOD 기준, 키워드 집합이 다르다는 점을 시스템 수준에서 반영한 것이다.

### 4.4 오버라이드 안전장치

회사/engagement 설정 오버라이드는 아무 필드나 허용하지 않는다.

- 글로벌 전용 필드
- 회사 허용 필드
- engagement 차단 필드

를 나눠 관리한다.

즉, 운영 범위를 넘어서는 값이 하위 레이어에서 덮이지 않도록 안전장치를 둔 구조다.

### 4.5 파이프라인 중심 구조

핵심 실행 흐름은 `AuditPipeline`이 담당한다.

- ingest
- validate
- feature
- detection
- score aggregation
- db load

를 하나의 실행 단위로 묶는다.

UI는 이 파이프라인의 결과를 탐색하고, 재탐지/복원/후속 분석을 조작하는 역할에 집중한다.

## 5. 지원 파일 포맷과 수집 계층

### 5.1 지원 포맷

- CSV
- Excel `.xlsx`
- Excel `.xls`
- Excel `.xlsb`
- TSV
- TXT
- DAT
- Parquet

### 5.2 대용량 대응

- 브라우저 업로드뿐 아니라 파일 경로 직접 입력 지원
- 10MB 이상 텍스트 파일은 청크 단위 로딩
- 진행률 콜백 기반 progress 표시
- 대용량 CSV 라인 카운트로 진행률 분모 계산

즉 단순히 “파일을 올릴 수 있다”가 아니라, 실제 수십~수백 MB급 파일을 다루는 로컬 분석 흐름까지 고려했다.

### 5.3 Excel 처리 디테일

- 멀티시트 Excel 지원
- 헤더 탐지와 별도로 시트 scoring 수행
- 추천 시트 자동 선택
- 병합 셀 처리

병합 셀의 경우 unmerge 전에 좌상단 값을 저장하고, unmerge 후 범위 전체에 값을 채워 넣는다.  
즉 ERP 원본의 병합 셀 때문에 헤더 탐지가 깨지는 문제를 흡수한다.

### 5.4 인코딩/구분자 처리

- `charset_normalizer` 기반 인코딩 추정
- `ascii` 감지 시 `latin-1` 폴백
- `csv.Sniffer` 기반 구분자 추정
- 확장자 기반 기본 구분자 fallback
- 추정 구분자보다 fallback이 더 많은 컬럼을 살리면 fallback 채택

이 부분은 현업 CSV가 항상 “정상적인 UTF-8 콤마 분리 파일”이 아니라는 현실을 반영한 것이다.

## 6. 파일 이상 상태 감지와 자동 복구

수집 단계는 단순 파일 읽기가 아니라, **깨진 추출물을 진단하고 가능한 범위에서 복구하는 계층**에 가깝다.

### 6.1 자동 진단 항목

- 빈 행 감지
- 빈 컬럼 감지
- 행별 컬럼 수 불일치 감지
- 혼합 구분자 감지
- 미닫힌 따옴표로 인한 파싱 누락 감지
- 지수 표기법으로 변환된 숫자 감지

### 6.2 자동 복구 항목

- 빈 행 제거
- 빈 컬럼 제거
- 미닫힌 따옴표 재파싱
- 혼합 구분자 행 복구
- 줄바꿈으로 깨진 행 병합

### 6.3 복구 UX

- 복구 전/후 미리보기 제공
- 바로 강행하지 않고 사용자가 확인 후 적용
- 복구된 행 수와 열 수를 표시
- 복구 로그를 데이터 경고로 남김

즉 파일 품질 문제가 있을 때도 사용자는 “왜 이상한지”와 “무엇을 복구했는지”를 볼 수 있다.

## 7. 헤더 탐지와 컬럼 매핑

### 7.1 헤더 탐지

- 헤더가 항상 첫 줄에 있다고 가정하지 않는다.
- 시트별로 헤더 후보를 탐지한다.
- 키워드, 구조, 샘플 분포를 활용해 적절한 헤더 행을 찾는다.

### 7.2 컬럼 매핑 방식

컬럼 매핑은 단일 방식이 아니라 여러 단계를 조합한다.

- Exact match
- Fuzzy match
- Pattern match
- Data sample 기반 보정
- 키워드 사전 기반 힌트

### 7.3 매핑 설명 가능성

매핑 제안마다 다음을 함께 제공한다.

- confidence
- rationale
- source_type
- target_type

즉, “AI가 이렇게 정했음”이 아니라 왜 추천했는지까지 같이 보여주는 구조다.

### 7.4 필수/권장 컬럼 분리

표준 스키마는 다음처럼 나뉜다.

- 필수 컬럼
- 권장 컬럼
- 라벨 컬럼

매핑 화면도 이 구분을 그대로 반영한다.

- 필수 컬럼 누락은 진행 차단
- 권장 컬럼 누락은 품질 경고
- 라벨 컬럼은 누수 방지를 위해 일반 매핑에서 숨김 처리

### 7.5 누락 영향 설명

누락된 컬럼마다 “이 컬럼이 없으면 어떤 검사가 약해지는지”를 함께 보여준다.

예:

- `created_by` 누락 시 자기승인/직무분리 탐지 약화
- `header_text` 누락 시 위험 적요 탐지 약화
- `has_attachment` 누락 시 증빙 관련 검증 약화

즉 매핑 화면이 단순 매핑 UI가 아니라, **탐지 성능에 대한 사전 설명 UI** 역할도 한다.

### 7.6 매핑 프로필 재사용

매핑 결과는 fingerprint 기반 JSON 프로필로 저장된다.

- source column 집합을 정규화
- SHA-256 기반 fingerprint 생성
- 확정 매핑 저장
- suggestion/unmapped는 로그로 분리 저장

이 구조 덕분에 동일 ERP 양식을 다시 올리면 기존 매핑을 재사용할 수 있다.

### 7.7 회사별 매핑 프로필 격리

매핑 프로필은 글로벌이 아니라 회사별 디렉터리에도 저장할 수 있다.

즉 같은 fingerprint 구조여도 고객사마다 다른 매핑 습관을 가질 수 있다.

### 7.8 매핑 기반 키워드 학습

사용자가 확정한 매핑을 기반으로 회사별 `keywords.yaml`을 자동 학습한다.

즉 한 번 수동으로 맞춘 ERP 별칭이 다음 업로드부터는 자동 추천 정확도를 높인다.

### 7.9 연도 간 컬럼 변경 감지

이 프로젝트의 대표적인 디테일 중 하나다.

이전 업로드 프로필이 있으면 현재 컬럼과 비교해:

- added
- removed
- renamed

를 계산한다.

즉 `2022` 파일 다음에 `2023` 파일을 올리면:

- 새로 생긴 컬럼
- 사라진 컬럼
- 이름만 바뀐 컬럼

을 바로 확인할 수 있다.

renamed 추정도 단순 문자열 일치가 아니라 유사도 기반으로 잡는다.

이전과 완전히 같으면 “변경 없음”을 따로 명시한다.

## 11. Phase 1 룰 기반 탐지 상세

`DETECTION_RULES.md` 기준으로 현재 핵심 룰 체계는 다음과 같다.

### 11.1 L1: 데이터 무결성

- `L1-01` 차대변 균형
- `L1-02` 필수 필드 누락
- `L1-03` 무효 계정
- `L3-01` 계정 분류 불일치

### 11.2 L2: 부정 관련 검토 신호 식별

- `L4-01` 매출 이상 변동
- `L2-01` 승인한도 직하
- `L1-04` 승인한도 초과
- `L2-02` 중복 지급
- `L2-03` 중복 전표
- `L1-05` 자기 승인
- `L1-06` 직무분리 위반
- `L3-02` 수기 전표
- `L1-07` 승인 생략
- `L3-03` 관계사 거래 검토 신호
- `L2-04` 비용 자산화

`L3-02`는 anomaly label 하나를 맞히는 룰로 보지 않고, `source = manual/adjustment`인 수기전표 모집단을 넓게 태깅하는 review rule로 운용한다. DataSynth에서도 `ManualOverride` anomaly와 `manual entry population truth`를 분리해 관리한다.

### 11.3 L3/L4: 이상 징후

- `L3-04` 기말 대규모 전표
- `L3-05` 주말 전기
- `L3-06` 심야 전기
- `L3-07` 소급 전기
- `L1-08` 기간 불일치
- `L3-08` 위험 적요
- `L4-02` Benford 위반
- `L4-03` 이상 고액
- `L4-04` 희소 차대 계정쌍
- `L3-09` 가수금 장기체류
- `L2-05` 역분개 패턴
- `L4-06` 배치 전표 이상

DataSynth `v32_candidate` 기준으로 `L3-06`은 `AfterHoursPosting`만 정답 라벨로 사용하고, 라벨 없는 정상 심야 배경은 `normal_after_hours_context` sidecar로 분리한다. `L4-05`는 `AbnormalHoursConcentration` 라벨과 사용자별 비정상 시간 집중 sidecar를 별도로 가진다.

### 11.4 Variance: 기존 회사 전용 변동 분석

기존 engagement 데이터가 있을 때만 수행되는 비교 트랙이다.

- `D01` 계정과목 집계 급변
- `D02` 월별 분포 패턴 변화

즉 신규 회사와 기존 회사의 탐지 경로가 완전히 같지 않다.

## 13. 탐지 엔진의 실행/상태 관리 디테일

### 13.1 병렬 실행

즉 특정 detector 실패가 전체 탐지를 무너뜨리지 않도록 설계되어 있다.

### 13.2 detector 상태표

- activation_requirements

- reason
- flagged_docs
- rules_run
- elapsed_sec

예:

- disabled_by_settings

- anonymous_context
- missing_historical_data
- detector_exception

모델이나 데이터가 없으면 실패로 끝내지 않고 적절히 skip한다.

- ML 모델 없음 -> 룰 기반 경로 유지
- historical data 없음 -> Variance skip
- anonymous context -> 일부 detector skip
- optional dependency 없음 -> 관련 track skip

## 16. DataSynth 전략과 데이터 계층

이 프로젝트는 공개 데이터셋 한계를 넘기 위해 DataSynth를 핵심 데이터 생성 기반으로 채택했다.

### 16.1 DataSynth를 채택한 이유

`TROUBLESHOOT.md` 기준:

- 공개 GL 데이터는 레이블이 부족하거나 감사 시나리오가 빈약함
- SAP 구조와 충분히 맞지 않음
- 실제 부정 시나리오 다양성이 부족함

반면 DataSynth는:

- EY + ASU 계열 오픈소스 생성기
- PCAOB / ISA / COSO / SOX 논리를 코드 레벨로 반영
- 감사 시나리오 주입이 가능
- 구조 확장이 가능

### 16.2 DataSynth가 프로젝트에 준 가치

- 표준 컬럼 스키마 기준점 제공
- 탐지 룰 설계의 기준 데이터 제공
- 합성 이상치 레이블 제공
- 품질 게이트 실험 가능
- 공개 데이터와 교차 검증 가능

즉 DataSynth는 단순 샘플 데이터가 아니라, **프로젝트의 스키마/룰/품질 실험을 지탱하는 핵심 기반**이다.

### 16.3 DataSynth 운영 스케일

문서 기준으로 프로젝트는 다음 스케일의 합성 데이터를 운용한다.

- KR 3법인 시나리오
- 2022~2024 다기간
- 약 31만 건 전표
- 약 110만 행 라인아이템

즉 UI와 파이프라인도 이 규모를 견디는 방향으로 설계되어 있다.

### 16.4 DataSynth 품질 원칙

TS-5에서 정리된 핵심은 다음과 같다.

- 회계적 무결성은 정상 전표에서 보장
- 데이터 품질 결함(MCAR, typo, format)은 정상/비정상에 균일 적용
- “정상 = 완벽” 같은 합성 아티팩트는 피함
- 품질 게이트에 데이터를 맞추는 test fitting을 경계

즉 DataSynth를 “깨끗한 데모 데이터”로 쓰지 않고,
**ML 관점에서 왜곡이 덜한 합성 데이터**로 다듬는 방향을 택했다.

### 16.5 DataSynth와 설정 간 계약

TS-2에서 다룬 중요한 포인트다.

- 한국식 승인한도와 금액 범위가 실제 데이터에 반영되도록 재설정
- SOD 기준이 과탐되지 않도록 사용자 수와 역할 분포 조정
- posting_date에 시간 정보가 존재하도록 재생성
- Benford 위반 시나리오 주입
- IC 식별 규칙 보강

즉 룰을 데이터에 맞추는 것이 아니라,
**데이터가 룰을 검증할 수 있도록** DataSynth 쪽을 조정했다.

### 16.6 DataSynth 확장 컬럼

문서 기준 확장/설계된 컬럼군은 다음과 같다.

- 승인: `approved_by`, `approval_date`, `approval_level`
- 증빙: `has_attachment`, `supporting_doc_type`, `invoice_amount`
- 컷오프: `delivery_date`
- 변경 이력: `changed_by`, `change_date`, `changed_field`
- 접근: `ip_address`
- 문서 흐름: `document_number`

일부는 생성 완료, 일부는 DuckDB 파생, 일부는 향후 확장 단계로 남아 있다.

### 16.7 DataSynth 관련 보조 도구

코드베이스에는 품질/분석용 도구가 별도로 존재한다.

- `audit_fullcheck.py`
- `audit_labels.py`
- `dual_audit.py`
- `dual_audit_v2.py`
- `analyze_datasynth.py`
- `report_datasynth_yearly_drift.py`
- `split_datasynth_labels.py`

즉 합성 데이터를 단순 소비하지 않고, 별도의 검사/리포팅 툴로 함께 다루고 있다.

## 20. 화이트리스트와 HITL

### 20.1 화이트리스트 저장 방식

화이트리스트는 문서 하나만 저장하는 구조가 아니라:

- batch_id
- document_id
- rule_code
- reason
- actor

단위로 저장된다.

즉 같은 문서라도 어떤 룰은 유지하고 어떤 룰만 예외 처리하는 것이 가능하다.

### 20.2 즉시 반영

화이트리스트가 등록되면 메모리상의 결과도 바로 동기화한다.

- `flagged_rules`에서 해당 rule 제거
- 남은 룰이 없으면 `risk_level = Normal`
- `anomaly_score = 0.0`

즉 DB에만 기록하고 화면은 그대로 두는 구조가 아니다.

### 20.3 피드백 이벤트

정규화된 feedback event도 같이 기록한다.

- false_positive
- confirmed_issue
- whitelist_revoked
- rule_feedback

즉 향후 라벨 전략과 성능 평가, 운영 보고서에 재활용할 수 있게 되어 있다.

### 20.4 문서 단위 피드백 이력

상세 화면에서 최근 피드백 이력을 문서 단위로 다시 읽어 보여준다.

즉 현재 판단과 과거 판단을 같은 화면에서 볼 수 있다.

## 25. 메트릭과 평가 체계

### 25.1 ground truth 평가

DataSynth 같이 정답 라벨이 있는 경우:

- precision
- recall
- f1

등을 계산한다.

### 25.2 operational proxy 평가

정답 라벨이 없을 때는 다음 운영 지표를 본다.

- flagged_docs
- high_risk_docs
- high_risk_ratio
- whitelist_removed_docs
- false_positive_docs
- confirmed_issue_docs

### 25.3 해석상의 주의

프로젝트 문서에도 명시돼 있듯:

- whitelist는 precision의 정답 대체물이 아님
- DataSynth 평가는 synthetic 룰에 유리할 수 있음
- historical batch는 metric confidence가 partial일 수 있음

즉 메트릭 체계도 “정답 지표”와 “운영 근사 지표”를 구분해서 본다.

# DataSynth 기준 데이터

현재 프로젝트의 기준 합성 데이터셋은 `data/journal/primary/datasynth/` 아래의 `v126` freeze 버전이다. 동결 시점은 `2026-05-02`이며, 프로젝트의 Phase 1~3 기본 검증과 대시보드 시연 데이터도 이 기준본을 따른다.

## 데이터 규모와 범위

- 기준 경로: `data/journal/primary/datasynth/`
- freeze 버전: `v126`
- 연도 파일: `journal_entries_2022.csv`, `journal_entries_2023.csv`, `journal_entries_2024.csv`
- 통합 파일: `journal_entries.csv`
- 전체 행 수: `1,109,435`
- 전체 전표 수: `319,193`
- 전체 컬럼 수: `52`
- 회사 코드: `C001`, `C002`, `C003`
- 주요 프로세스: `P2P`, `O2C`, `R2R`, `H2R`, `TRE`, `A2R`
- 주요 문서유형: `AA`, `DR`, `DZ`, `HR`, `IC`, `KR`, `KZ`, `SA`, `WE`, `WL`
- 2025 spillover: `13` rows / `6` documents

연도별 분할 파일 기준 규모는 다음과 같다.

- 2022: `373,425` rows / `106,675` documents
- 2023: `366,465` rows / `105,525` documents
- 2024: `369,545` rows / `106,993` documents

## 행 단위 의미

이 데이터는 `전표 헤더 1건 = 여러 개의 분개 라인` 구조다.

- CSV 한 행은 `전표 라인 1개`를 뜻한다.
- 같은 `document_id`를 공유하는 여러 행이 모이면 `전표 1건`이 된다.
- 헤더 수준 정보는 같은 `document_id`를 가진 모든 라인에 반복 저장된다.
- 라인 수준 정보는 `line_number`별로 달라진다.

즉 모델 학습이나 규칙 검사에서는 `row 단위`와 `document 단위`를 구분해서 봐야 하며, 현재 프로젝트는 이 데이터에 대해 `document_id` 기준 split과 평가를 사용한다.

## 열 구성과 의미

실제 컬럼은 아래 44개다.

## 어떻게 만들었는가

이 데이터는 프로젝트 안의 Rust 기반 DataSynth 생성기에서 만든 한국형 회계 전표 합성 데이터다.

- 한국 제조/유통 계열사의 ERP 전표 흐름을 가정했다.
- 회사 3개와 다년도 회계기간을 기준으로 `P2P`, `O2C`, `R2R`, `H2R`, `TRE`, `A2R` 프로세스를 섞어 생성했다.
- 전표 헤더, 라인, 마스터데이터, 관계 정보, anomaly label sidecar를 함께 생성했다.
- 현재 `v126`은 V20~V23 운영 정합성 보정 이후 L3-08, L3-09, L3-10 평가 계약, L4-02 Benford group truth/holdout, D01/D02 계정 단위 평가 계약과 정상 대조군, L1-09/L2-02 필수 truth 복구, L1-03/L3-01 CoA 경계 보정을 누적 반영한 운영 기준본이다.

`v126`에 반영된 핵심 개선은 다음과 같다.

- 희소 피처 coverage 개선: `cost_center`, `tax_*`, `trading_partner`, `auxiliary_*`
- `user_persona` 오탈자 폭발 억제 및 제한적 surface noise
- 연도/회사 조건부 drift 도입
- 정상 전표 불균형 제거
- fallback anomaly 전략 축소
- contextual missingness 적용
- 텍스트 반복 완화와 `R2R`/`TRE` 적요 다양화
- `document_number` 문자열화 및 `delivery_date` 문맥 정리
- V20 CoA 정합성 보정과 `B04`/`B10` realism 보강
- V20.1 MisclassifiedAccount invalid CoA 오염 제거
- v20.3 `created_by`/`approved_by`와 `employees.user_id` 직접 조인 가능하도록 JE user master 재구성
- v20.4 `ExceededApprovalLimit`를 `approved_by.approval_limit` 기준으로 정정
- v43/v44 L3-08 결손/파손 적요 truth와 정상 경계 대조군 분리
- v42 L3-09 가수금/가계정 장기체류 모집단, confirmed subset, 정상 정산 대조군 분리
- v45 L3-10 민감 계정 사용을 review population, confirmed anomaly, normal control로 분리
- v52 L4-02 Benford를 전표 단위가 아니라 계정 group 단위 truth로 분리
- v54 L4-02 Benford adversarial/holdout sidecar 추가

## 프로젝트에서 이 데이터를 사용하는 방식

- Phase 1: 룰 기반 탐지와 회귀 테스트용 기준 데이터
- Phase 2: `document_id` 기준 평가와 연도 holdout 또는 group holdout 기반 학습 데이터
- Phase 3: 대시보드, 설명, 내보내기, 사례 탐색용 시연 데이터

프로젝트는 이 데이터를 그대로 쓰지 않고, 라벨 분리, validated metadata, feature quality gate, split policy를 추가로 적용해서 운영한다. 즉 DataSynth는 원천 데이터 생성기이고, 실제 분석 파이프라인에서는 누수 방지와 평가 정책이 한 번 더 덧씌워진다.
