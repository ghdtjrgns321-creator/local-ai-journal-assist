# 전표 감사 기준서 vs 프로젝트 갭 분석

> **PHASE1 역할 원칙**: PHASE1은 `fraud`를 확정하거나 정답 라벨을 맞히는 단계가 아니다. PHASE1의 목적은 전수 모집단에서 규칙 위반, 정책 위반, 이상 징후, 분석적 검토 신호를 넓게 올려 **감사인이 봐야 할 항목과 우선순위**를 만드는 것이다. DataSynth의 `is_fraud`/`is_anomaly`와 precision/recall은 개발 검증 보조 지표이며, 운영 해석은 예외 처리 대상, 감사인 리뷰 대상, 고위험 후보를 구분하는 review queue 기준으로 한다.

전표 감사 시 기준서가 요구하는 검사항목을 조사하고,
프로젝트 [AUDIT_DOMAIN_FINAL.md](AUDIT_DOMAIN_FINAL.md) 24개 룰에서 빠진 것을 분석한 문서.

> **프로젝트 전제**
> - CAATs 도구: journal_entries.csv **전수 검사** (표본추출 아님)
> - 입력: DataSynth 39개 컬럼 (ERP에서 추출된 전표 데이터)
> - document_id = UUID (순번 아님), posting_date = timestamp (시간 포함)
> - xstov (역분개 플래그) = 실제 데이터에 없음
> - DataSynth는 합성 데이터 생성기이므로 필요 컬럼 추가 생성 가능

## 2. 구현 가능 — 18건

### 2-7. 역분개(Reversal) 패턴 탐지

- **검사 내용**: 마감 전후 대량 역분개, 반복적 역분개
- **기준서**: 감사기준서 240호
- **현재 상태**: 도메인 용어에 `is_reversal` 매핑만 있고 룰 없음
- **구현 로직** (Audit Sight/Arbutus 표준 접근):
  1. **1:1 매칭**: 동일 `gl_account` + 동일 금액 + 반대 방향(차↔대) 전표를 매칭 (±1일 이내)
  2. **1:N / N:M 분할 역분개 탐지 (Rolling Sum Zero-Out)**: 1:1 매칭만으로는 금액을 쪼개서 역분개하는 패턴을 놓침. 특정 `gl_account` + `created_by` 조합에서 일정 기간(7일/30일) 내 `SUM(debit) - SUM(credit) ≈ 0`에 수렴하는 부분합을 DuckDB 윈도우 함수로 탐지. 예: 100억 차변 전표 → 30억+30억+40억 대변 3건으로 분할 역분개
     ```sql
     -- DuckDB Rolling Sum Zero-Out 개념
     SELECT gl_account, created_by, posting_date,
            SUM(debit_amount - credit_amount)
              OVER (PARTITION BY gl_account, created_by
                    ORDER BY posting_date
                    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS rolling_net
     FROM journal_entries
     WHERE ABS(rolling_net) < threshold  -- 0에 수렴하는 구간 탐지
     ```
  3. **Reversing vs Correcting 구분** (SAP Reversal Reason Code 기반):
     - RRC 05(Accrual Reversal) + 기초(1월) + `source='auto'` = **정상 역분개** (발생주의 자동 역분개)
     - RRC 01(당기 역분개) + `source='manual'` + 임의 시점 = **수정 전표 → 검토 대상**
     - RRC 02(타 기간 역분개) = **결산 후 수정 → 고위험**
  4. `line_text` 키워드 탐지 (한국 관례): "수정", "정정", "오류", "역분개", "결산조정", "미지급비용 역분개", "adjust", "correct", "restate"
  5. 기말 30일 이내 역분개 집중도: 전체 역분개 중 기말 비율이 유의적이면 가중
- **DataSynth**: 현재 39컬럼으로 구현 가능

### 3-4. Top-side Journal Entries (경영진 조정 전표)

- **기준서**: 감사기준서 240호, PCAOB AS 2401
- **현재 상태**: L3-02/L3-04/L1-05이 개별 존재하나 조합 탐지 없음
- **구현 로직** (Arbutus 가중 점수 방식):
  - `source='manual'` (수기 전표)
  - 기말 시점 (L3-04 기간)
  - `created_by == approved_by` (자기승인) 또는 승인 없음
  - 비정상/미사용 계정 사용 (L1-03/L4-04 해당)
  - 고액 전표 (L4-03 해당)
  - `line_text` 공백 또는 부실 (L3-08 해당)
  - **각 특성에 이진 점수 부여 → 합산 → 임계값 초과 시 Top-side JE 의심**
- **DataSynth**: 현재 39컬럼으로 구현 가능 (기존 룰 플래그 조합)

### 4-3. 비정상 시간대 입력자 집중 분석

- **검사 내용**: 특정 사용자가 심야/주말에 전표를 집중적으로 입력
- **기준서**: KLCA IT 체크리스트
- **현재 상태**: L3-05/L3-06이 건수만 탐지. 사용자별 집중도 미분석
- **구현 로직** (Greenskies Analytics 접근):
  1. **비정상 시간대 정의 (한국 실무 반영)**: 한국은 야근이 일상적(평균 퇴근 18:19, 시프티 2024)이므로 18시 이후를 바로 비정상으로 잡으면 false positive 폭증
     - 정상: 08:30~18:30
     - 야근(저위험): 18:30~22:00 — 결산기(12~1월)에는 정상 취급
     - **심야(고위험): 22:00~06:00** — 감사 플래그 대상
  2. **비근무일**: 주말 + 한국 공휴일(대체공휴일 포함). 공휴일 캘린더 config 필요
  3. 사용자별 심야/비근무일 전표 비율 산출
  4. **전체 사용자 평균 대비 3σ 이상 → 이상치** (통계적 접근)
  5. 추가: 입력자-승인자 간 시간 차이가 극히 짧은 경우 (부실 검토 의심)
- **DataSynth 시간대 multiplier 설정**:
  ```
  심야(00~06)=0.02 / 이른출근(06~08:30)=0.15 / 오전피크(08:30~11:30)=1.8
  점심(11:30~13)=0.3 / 오후(13~16)=1.2 / 마감러시(16~18:30)=1.5
  야근(18:30~22)=0.3(평상시), 0.7(결산기) / 심야야근(22~24)=0.05
  ```

### 4-4/4-5. 승인 프로세스·승인자 계층

- **검사 내용**: 전결규정 준수 여부, 승인 지연, 레벨 건너뜀
- **기준서**: 감사기준서 315호/330호 (ITGC 통제)
- **현재 상태**: L1-05/L1-07가 결과만 추론. 승인 컬럼 없음
- **구현 로직**:
  - 승인 누락률: `approved_by` IS NULL + 한도 초과 전표 비율
  - 승인 지연: `approval_date - posting_date` > N일 (DataSynth v1.2.0: approval_date는 date 타입)
  - 레벨 건너뜀: 금액 대비 `required_approval_level` (DuckDB 파생 컬럼, CASE WHEN 전결규정 6단계)
  - 자기승인 정밀화: `created_by == approved_by` (현재 L1-05 개선)
- **DataSynth 수정 필요** (approval.rs 활성화):
  1. `je_generator.rs` L1693~1699: 하드코딩된 레벨 결정 → config thresholds 참조로 변경
  2. `user.rs` UserPersona: 미국식 역할명 → 한국식 추가 (`TeamLead`, `DeptHead`, `CFO`)
  3. `approval.rs` `standard()`: 달러 기준 → 원화 기준 변경
  4. `config/datasynth.yaml`: approval 섹션 활성화
- **한국 전결규정 참고** (기업별 편차 큼, 설정값으로 외부화 필수):
  ```
  ~3만원       적격증빙 면제 (간이영수증 인정)
  ~100만원     자동 승인
  ~5천만원     팀장 1단계
  ~1억원       +본부장/이사 2단계
  ~2억원       +부서장/상무 3단계
  ~5억원       +CFO 4단계
  5억원+       +CEO/이사회 5단계
  ```
  > 3만원 초과 거래는 적격증빙(세금계산서/카드/현금영수증) 수취 의무.
  > 미수취 시 **증빙불비가산세 2%** (세법). 3만원 직하 분할(29,000원×N건)도 탐지 대상.

### 2-1. 증빙 존재 확인

- **검사 내용**: 전표에 증빙이 첨부되어 있는지
- **기준서**: 감사기준서 240호, 500호
- **현재 상태**: 증빙 관련 컬럼 없음
- **구현 로직**:
  - `has_attachment=False` + 수기 + 고액 → 증빙 누락 의심
  - **한국 세법 적격증빙**: 3만원 초과 거래는 세금계산서/계산서/카드매출전표/현금영수증 중 하나 필요. `amount > 30,000` + `doc_type NOT IN ('세금계산서','카드','현금영수증')` → 적격증빙 미수취 의심
  - 3만원 직하 분할 탐지: 동일 거래처 + 동일일 + 29,000원 이하 × N건 → 증빙 회피 의심
- **DataSynth 확장**: `has_attachment`(bool), `supporting_doc_type`(str: 세금계산서/카드/현금영수증/간이영수증/없음) 생성

### 2-4. 컷오프 (납품일 vs 전기일)

- **검사 내용**: 출하일/서비스완료일과 매출 인식일의 일치
- **기준서**: 감사기준서 315호, 330호 (K-IFRS 15 수익인식)
- **현재 상태**: L3-07/L1-08가 posting_date vs document_date만 비교
- **구현 로직** (감사 교과서적 접근):
  - 매출: `|posting_date - delivery_date|` > N 영업일 → 조기/지연 인식 의심
  - 비용: `|posting_date - invoice_date|` > N 영업일 → 기간귀속 오류 의심
  - **한국 실무**: 97% 이상 12월 결산. 밀어내기(channel stuffing) 집중 시기는 **12월 마지막 2주**. 1월~3월은 결산 조정 기간(법인세 신고기한 3월 말)
  - 기말 전후 5~10 영업일 구간 집중 분석
- **DataSynth 확장**: `delivery_date`(date) 생성. 12월 말에 delivery_date > posting_date (미출하 조기인식) 패턴 주입

### 2-5. 증빙 금액 불일치

- **검사 내용**: 전표 금액과 증빙(세금계산서) 금액의 일치
- **기준서**: 감사기준서 500호
- **현재 상태**: L3-08이 적요 기반만 커버
- **구현 로직** (3-way matching 간소화):
  - `|debit_amount - invoice_amount|` > 허용오차(1% 또는 절대금액) → 불일치 플래그
  - **부가세 검증**: 한국 부가세율 10%. `tax_amount ≠ round(supply_amount × 0.1)` → 부가세 오류
- **DataSynth 확장**: `invoice_amount`(float), `invoice_date`(date), `tax_amount`(float), `supply_amount`(float) 생성. 부가세 10% 기본 적용

### 4-1. 전표 수정/삭제 이력

- **검사 내용**: 전표 변경 시점, 변경자, 변경 내용
- **기준서**: KLCA IT 체크리스트 (변경관리 4.3~4.5)
- **현재 상태**: CSV에 최종 상태만 있음
- **구현 로직** (SAP CDHDR/CDPOS 접근):
  - **한국 SAP 실무**: 전기된 전표는 직접 수정 불가. 반드시 **역분개(FL3-02) + 재전기** 방식. 따라서 "수정"은 텍스트/참조 필드 변경만 해당하고, 금액/계정 변경은 역분개 패턴(2-7)에서 탐지
  - 텍스트 변경: `change_type='UPDATE'` + `changed_field IN ('line_text','header_text')` + 기말 시점 → 기말 적요 수정 의심
  - `created_by ≠ changed_by` + 고액 → 무단 수정 의심
  - 기말 30일 이내 변경 집중도: 월평균 대비 유의적 증가 시 가중
  - **상법 제33조**: 장부 10년, 전표류 5년 보존 의무. 삭제(`change_type='DELETE'`)는 원칙적으로 불가
- **DataSynth 확장**: `changed_by`(str), `change_date`(datetime), `changed_field`(str), `old_value`(str), `new_value`(str) 생성

- **기준서**: 감사기준서 315호, 330호
- **현재 상태**: L1-03이 계정 존재만 확인
- **구현 로직**: 계정-거래유형 매핑 마스터 필요 → Phase 2 MisclassifiedAccount에서 ML로 접근
- **DataSynth**: `tax_code`, `tax_amount` 컬럼 추가. K-IFRS 표준계정과목 코드 체계 반영:
  ```
  자산: 1xx~2xx (101~250)   부채: 2xx~3xx (251~330)   자본: 3xx~4xx (331~400)
  매출/수익: 4xx (401~450)   매출원가: 4xx~5xx (451~500)
  판관비: 8xx (801~899)      영업외비용: 9xx (901~960)
  ```
  > SAP 한국 구축 시 6자리(예: 411000=매출, 811000=급여) 또는 8자리 사용이 일반적

### 2-6. 경제적 실질 (실질우선)

- **검사 내용**: 형식상 매출이지만 실질은 대여·담보 등인지
- **기준서**: 감사기준서 315호, 240호
- **현재 상태**: L3-03(순환거래)만 부분 커버
- **구현 로직**: 계정-거래유형 불일치 + NLP(적요 분석) → Phase 2~3
- **DataSynth**: 기존 컬럼으로 부분 가능, NLP는 Phase 3

### 2-8. 회계추정치 편의(bias)

- **검사 내용**: 충당금·감가상각 등에서 경영진 편의 개입
- **기준서**: 감사기준서 240호 §32(b), ISA 540
- **현재 상태**: 대응 룰 없음
- **구현 로직** (ISA 540 소급 검토 방식):
  1. 추정치 관련 계정(충당금, 감가상각, 보증) 식별
  2. **전기 추정치 vs 실제 결과 차이(estimation error)** 시계열 산출
  3. 부호가 일관되게 한 방향이면 bias 의심 (이익 방향 편향)
  4. 추정치가 합리적 범위의 상한/하한에만 위치하는 패턴
- **DataSynth**: 다기간 데이터 필요 → Phase 2 TrendBreak

### 3-1. 재무제표-장부 대사

- **검사 내용**: GL 잔액과 보조원장 잔액의 일치
- **기준서**: 감사기준서 330호
- **현재 상태**: 전표 개별 검사만. 대사 절차 없음
- **구현 로직** (DataSnipper/CAATs 표준):
  - GL 잔액 vs AR/AP Aging, 고정자산 대장, 재고 보조부 합계
  - 중요성 이하 차이 자동 통과, 초과분 원인 추적
- **DataSynth**: Trial Balance 테이블 별도 생성 필요 → Phase 2

### 4-6. 배치 전표 이상 패턴

- **검사 내용**: 자동 생성 전표(배치)의 이상 패턴
- **기준서**: 금융권 IT 감사 가이드라인
- **현재 상태**: `source` 컬럼으로 구분만 가능
- **구현 로직**: `source='batch'` 전표의 기말 집중, 대량 동시 생성, 금액 이상 탐지
- **DataSynth**: 기존 `source` 컬럼으로 부분 가능 → Phase 2

### 3-6. 유의적 거래 합리성 평가

- **검사 내용**: 비경상 거래의 사업상 합리성
- **기준서**: 감사기준서 240호 §32(c)
- **현재 상태**: L4-03/L4-01이 유의적 거래 탐지. 합리성 판단 없음
- **구현 로직**: 탐지된 이상 전표의 적요+계정+금액을 LLM이 분석 → 보조 의견 제공
- **DataSynth**: 기존 컬럼 활용 → Phase 3 LLM

## 부록 A. 기준서 상세

### A.1 감사기준서 240호 — 전표검사의 핵심

**§32**: 감사인은 경영진 내부통제 무력화 위험 평가 결과에 관계없이, 분개기입과 기타 수정사항의 적정성을 테스트하여야 한다.

- §32(a)(i): 관여 개인에게 부적절/비정상 활동 질문
- §32(a)(ii): 보고기간 말 전표 선정·검사
- §32(a)(iii): 보고기간 전체 전표 검사 필요성 고려
- §32(b): 회계추정치 편의(bias) 검토
- §32(c): 유의적 거래의 사업상 합리성 평가

**A49**: 부적절한 분개기입의 식별 특성 — (a) 미사용 계정, (b) 비인가자 입력, (c) 기말+설명 부재, (d) 계정번호 없음, (e) 단수 금액/끝자리 패턴

**A50**: CAATs를 사용하여 비정상 분개기입을 식별할 수 있다.

### A.2 감사기준서 315호

§26(a)(ii): 분개기입에 대한 통제, 특히 비표준 분개기입에 대한 통제를 이해하여야 한다.

### A.3 감사기준서 330호

재무제표-회계기록 대사·조정 + 중요 전표 검사 = 필수 실질절차. 통제 양호 → TOE, 미흡 → 실증절차 확대.

### A.4 감사기준서 500호

증빙 적정성: 전표-증빙 일치, 필수기재사항, 부정위험 문서 추가 절차.

### A.5 감사기준서 520호

§5: 기대값 개발 → 차이 분석. L4-02(Benford) 직접 근거.

### A.6 감사기준서 550호

특수관계자 거래의 사업상 합리성 평가.

### A.7 감사기준서 1100호

감사인의 내회관 검사 의무 (외감법 §8⑥). 통제 설계·운영 평가. TOE 실패 시 실증절차 확대.
