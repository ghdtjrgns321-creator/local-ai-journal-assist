# 프로젝트 개선사항 정리

> **PHASE1 역할 원칙**: PHASE1은 `fraud`를 확정하거나 정답 라벨을 맞히는 단계가 아니다. PHASE1의 목적은 전수 모집단에서 규칙 위반, 정책 위반, 이상 징후, 분석적 검토 신호를 넓게 올려 **감사인이 봐야 할 항목과 우선순위**를 만드는 것이다. DataSynth의 `is_fraud`/`is_anomaly`와 precision/recall은 개발 검증 보조 지표이며, 운영 해석은 예외 처리 대상, 감사인 리뷰 대상, 고위험 후보를 구분하는 review queue 기준으로 한다.

이 문서는 현재 프로젝트의 문제점을 기반으로, 실제로 무엇을 고치면 좋은지 정리한 문서다.

정리 기준은 아래 4가지다.

- 무엇을 수정할 것인가
- 왜 필요한가
- 난이도는 어느 정도인가
- 효과는 얼마나 큰가

핵심 원칙은 간단하다.

> 기능을 더 많이 넣는 것보다,  
> 지금 있는 기능의 중심축을 명확히 하고,  
> 성능과 신뢰도를 숫자로 보여줄 수 있게 만드는 것이 먼저다.

## 1. 현재 문제를 한 줄로 요약하면

이 프로젝트는 기능은 넓고 구조도 잘 잡혀 있지만, 아직은 `좋은 분석 워크벤치`에 더 가깝고 `운영형 제품`으로 가기 전 단계다.

구체적인 문제는 아래에 가깝다.

- 핵심 기능과 실험 기능의 경계가 흐릴 수 있다.
- Phase 2 supervised는 실전 라벨 부족 때문에 가치가 제한적이다.
- 기능은 많지만 성능을 정량적으로 보여주는 지표가 약하다.
- 룰/설정/회사별 override가 많아질수록 운영 복잡도가 커진다.
- 설명 가능성이 강점인데, 그 강점을 UI/리포트/로그 전체에 완전히 표준화하지는 못했다.

## 3.2 탐지기별 운영 상태 표준화 ✅

### 무엇을 고칠까

각 탐지기에 운영 상태 메타를 붙이고, 그 값을 UI, 문서, 로그에서 공통으로 사용한다.

다만 상태를 아래 4개 중 하나로만 두면 의미가 섞인다.

- `production`
- `beta`
- `experimental`
- `disabled by default`

`disabled by default`는 성숙도라기보다 기본 활성화 정책에 가깝다.
그래서 실제 구현은 아래처럼 2축으로 나누는 편이 더 안전하다.

- `maturity`: `production | beta | experimental`
- `default_enabled`: `true | false`
- `activation_requirements`: `settings`, `trained_model`, `historical_data`, `external_api`, `optional_dependency` 같은 조건 목록

즉, "실험적이지만 기본 활성화", "베타이지만 기본 비활성화" 같은 상태를 분리해서 표현할 수 있어야 한다.

예를 들어 `DetectionResult.metadata`나 탐지기 클래스 메타에 아래 같은 구조를 둔다.

```python
{
    "maturity": "beta",
    "default_enabled": False,
    "activation_requirements": ["settings", "historical_data"],
}
```

### 왜 필요한가

지금은 “코드에 있다”와 “실제로 믿고 쓰는 기능이다”가 섞여 보일 수 있다.

현재 코드만 봐도 이미 이 문제가 드러난다.

- `layer_a`, `layer_b`, `layer_c`, `benford`는 `src/pipeline.py`에서 기본 경로로 항상 실행된다.
- `duplicate`, `intercompany`는 기본 경로에서 같이 실행되지만 가중치 체계에는 아직 정식 편입되지 않았다.
- `relational`, `graph`, `nlp`, `access_audit`, `evidence`, `layer_d`, `trendbreak`, `ml_*`는 구현되어 있어도 `config/settings.py`에서 기본값이 모두 `False`다.
- `ml_supervised`, `ml_unsupervised`는 설정뿐 아니라 학습된 모델이 있어야만 실제 실행된다.
- `nlp`는 OpenAI 임베딩 API, `graph`는 `networkx`, `layer_d`와 `trendbreak`는 전기/다기간 데이터가 필요하다.

즉 "구현 존재", "기본 실행", "운영 신뢰도", "실행 조건"이 서로 다른데 지금은 한 덩어리로 보인다.

### 난이도

- 중

### 효과

- 큼

### 기대 효과

- 기능 신뢰도 구분이 명확해진다.
- 실험 기능이 메인 제품 인상을 해치지 않는다.
- QA 범위와 기본 테스트 범위를 상태별로 나눌 수 있다.
- 대시보드에서 "왜 이번 배치에서 안 돌았는가"를 설명하기 쉬워진다.

### 현재 코드 기준 분류 초안

| 탐지기 | 현재 코드 기준 상태 | 근거 |
|--------|---------------------|------|
| `layer_a` | `production`, 기본 활성화 | 기본 파이프라인의 base detector |
| `layer_b` | `production`, 기본 활성화 | 기본 파이프라인의 base detector |
| `layer_c` | `production`, 기본 활성화 | 기본 파이프라인의 base detector |
| `benford` | `production`, 기본 활성화 | 기본 파이프라인의 base detector + 가중치 체계 포함 |
| `duplicate` | `beta`, 기본 활성화 | 기본 실행은 되지만 가중치 체계 미편입 |
| `intercompany` | `beta`, 기본 활성화 | 기본 실행은 되지만 가중치 체계 미편입 |
| `relational` | `beta`, 기본 비활성화 | 구현 완료, `enable_relational_detection=False` |
| `evidence` | `beta`, 기본 비활성화 | 구현 완료, `enable_evidence_detection=False` |
| `access_audit` | `beta`, 기본 비활성화 | 구현 완료, `enable_access_audit_detection=False` |
| `layer_d` | `beta`, 기본 비활성화 | 전기 데이터 의존, `enable_variance_detection=False` |
| `trendbreak` | `experimental`, 기본 비활성화 | 다기간 추정치/설정 의존, `enable_trendbreak_detection=False` |
| `graph` | `experimental`, 기본 비활성화 | `networkx` 의존, `enable_graph_detection=False` |
| `nlp` | `experimental`, 기본 비활성화 | 외부 임베딩 API 의존, `enable_nlp_detection=False` |
| `ml_supervised` | `experimental`, 기본 비활성화 | 학습 모델 필요, `enable_ml_detection=False` |
| `ml_unsupervised` | `experimental`, 기본 비활성화 | 학습 모델 필요, `enable_ml_detection=False` |
| `stacking_meta` | `experimental`, 기본 비활성화 | 학습된 meta-learner 필요, 기본 폴백 경로 아님 |
| `ml_transformer` | `experimental`, 비노출 | 학습 인프라는 있으나 현재 메인 추론 경로에 없음 |
| `ml_sequence` | `experimental`, 비노출 | 학습 인프라는 있으나 현재 메인 추론 경로에 없음 |

이 표의 핵심은 "좋다/나쁘다"를 나누는 것이 아니라, 사용자에게 현재 제품의 진짜 중심축을 거짓 없이 보여주는 것이다.

### 표시 규칙

- UI 배지: `Production`, `Beta`, `Experimental`
- 기본 비활성화 탐지기는 배지 옆에 `Off by default`를 별도 표기
- 실행 실패가 아니라 조건 미충족으로 스킵된 경우 `Skipped: missing requirement` 형식으로 노출
- 로그에는 `track=nlp maturity=experimental default_enabled=false skipped=requirement:external_api`처럼 구조화해 남긴다
- 문서 표와 대시보드 표의 용어를 동일하게 맞춘다

### 구현 순서

1. `src/detection/constants.py` 또는 별도 registry 파일에 탐지기 메타 사전을 만든다.
2. `BaseDetector`에 공통 메타 접근자 또는 `detector_profile` 속성을 추가한다.
3. `src/pipeline.py`에서 실행 결과와 스킵 사유를 같은 구조로 수집한다.
4. `dashboard/tab_phase2.py`, `dashboard/components/rule_panel.py`에서 동일한 상태 배지를 사용한다.
5. `docs/EXPLAIN.md`, `docs/상세.MD`의 탐지기 설명 표를 같은 분류 체계로 맞춘다.

### 수정 후보 파일

- `src/detection/constants.py`
- `src/detection/base.py`
- `src/pipeline.py`
- `dashboard/components/rule_panel.py`

## 3.3 성능 평가 리포트 추가 ✅

> 최신 PHASE1 기준 보정: 이 섹션은 완료된 개선 아이디어 기록이다. PHASE1의 현재 역할은 최종 부정 분류가 아니라 1차 후보 생성과 2차 case 분류이므로, `룰별 precision / recall`은 단독 성패 지표가 아니다. PHASE1 리포트는 rule truth coverage, review queue 규모, 예외 처리율, high-risk candidate 비율, PHASE2 overlay 전후 case priority 변화를 함께 보여줘야 한다.

### 무엇을 고칠까

룰/탐지기별 평가 지표를 자동으로 뽑는 리포트를 만든다.

예:

- 룰별 precision / recall
- false positive 비율
- whitelist 반영 전후 변화
- batch별 high-risk 비율
- Phase 1 단독 vs Phase 2 포함 성능 비교

### 왜 필요한가

지금은 구조 설명은 잘 되지만, “그래서 얼마나 잘 잡는가”를 숫자로 보여주는 운영 지표가 약하다.

### 난이도

- 중

### 효과

- 매우 큼

### 기대 효과

- 면접에서 신뢰도가 올라간다.
- 어떤 탐지기를 유지/제거할지 근거가 생긴다.
- 룰 튜닝 방향이 명확해진다.

### 수정 후보 파일

- `tests/phase1_rulebase/*`

## 5. 한 줄 결론

> 더 많은 기능을 넣는 것보다,

> 성능과 신뢰도를 보여주는 체계를 먼저 만드는 것
