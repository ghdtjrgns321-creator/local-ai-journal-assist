# Detection Results

## Phase 1 L1 - DataSynth v126

대상 데이터는 `data/journal/primary/datasynth_v126_candidate`입니다.

이 문서는 L1 결과를 **세 가지 독립적인 검증 축**으로 나누어 봅니다. 세 축은 측정하는 대상이 다르므로 같은 표에 섞으면 의미를 잃습니다.

| :-- | :----------------------------------------------------------------- | :----------------------------------------------------------------------------------------- | :-------------------------------------------- |

세 축의 관계:

- A축은 detector 사양과 데이터 생성기가 같은 정의를 공유하는지 보는 **계약 검증**입니다. 0/0이 합격선입니다.
- B축은 **detector 로직을 모르는 입장**에서 "현장에서 실제로 나올 만한 형태"를 sidecar로 만든 뒤 detector가 어떻게 반응하는지 봅니다. L1은 기본 정합성/승인 통제/기간 통제 영역이므로 "위험점수 직접 반영 vs 검토/별도 표시 분리"가 핵심입니다.
- C축은 **실제 조작 전표 표본**(`manipulated_entry_truth.csv`)이 L1 책임 영역(차대변, 필수필드, 무효계정, 승인, 기간 정합성)에 어떻게 걸리는지 봅니다.

L1은 Phase1 전체에서 **전표 기본 정합성과 승인 통제의 1차 게이트**입니다. 다만 L1이 잡았다고 모두 동일한 감사 위험점수로 처리하면 안 됩니다. L1-02처럼 탐지 가능성을 제한하는 항목은 위험 합산보다 별도 표시가 맞고, 시스템 승인 흐름이나 비감사 기간 이슈는 검토/별도 설명으로 분리해야 합니다.

## 요약

- **A축 결과**: 3개 연도 × 9개 룰 모두 **정탐 100%, 과탐 0, 미탐 0**. 룰 계약 정합성 합격.
- **B축 결과**: sidecar 기준으로 확정 이상, 정상 통제군, 시스템 통제 갭, 비감사 기간 이슈가 의도대로 분리됨. v121에서 문제가 되었던 L1-02 두 문서는 v126에서 통합 원장과 연도별 원장이 모두 `fiscal_period` 누락으로 일치.
- **C축 결과**: `manipulated_entry_truth.csv`의 실제 조작 전표 420건 중 L1 책임분 36건을 전부 포착. 나머지 384건은 L1 실패가 아니라 L2~L4 또는 D01/D02가 담당할 패턴형 조작.

세 축을 분리하는 이유:

- A축이 깨끗해도 sidecar처럼 detector 코드를 보지 않은 모집단에서 분류가 어긋나면 현장에서 과탐이 터집니다.
- B축이 깨끗해도 fraud 합성 데이터에서 책임분 포착이 빠지면 실제 감사에서 누락이 발생합니다.
- 세 축이 모두 통과해야 L1이 1차 통제 신호로 의도대로 동작한다고 말할 수 있습니다.

또 "L1이 잡았다"와 "최종 위험점수에 강하게 반영한다"는 다릅니다. 예를 들어 `wrong_period_non_audit_issue_truth`는 L1-08이 형식상 잡지만, 감사 위험이 아닌 기간 정합성 확인 항목이므로 위험점수에 넣으면 안 됩니다. L1-02도 필수 정보 누락으로 검사 제한을 보여주는 성격이 강하므로 최종 보고에서는 별도 표시하는 편이 맞습니다. 이런 분리는 B축 영역입니다.

## 검증 축 B — Sidecar 현실 케이스 포착력 (DETECTOR 비참조)

이 축은 **detector 코드를 직접 참조하지 않고** "현장에서 실제로 나올 만한 케이스"를 sidecar로 만들어 둔 뒤 detector가 그것을 어떻게 분기하는지 봅니다. L1은 Phase1 전체 점수에서 전표 정합성·승인 통제·기간 정합성의 1차 신호이므로 B축의 핵심은 "잡힌 결과를 위험점수에 직접 넣을지 검토/별도 설명으로 뺄지"의 분기가 detector 동작과 정합한지입니다.

### Sidecar 평가 기준

| 항목      | 의미                                                                      |
| :-------- | :------------------------------------------------------------------------ |
| L1 책임   | 해당 sidecar가 L1이 직접 잡아야 하는 대상인지 여부 (예/아니오/별도)       |
| 실제 잡음 | L1 detector가 실제로 포착한 문서 수                                       |
| 위험점수  | 최종 위험점수에 반영해야 하는 문서 수                                     |
| 검토/별도 | 잡혔지만 위험점수에서 빼고 검토 또는 별도 설명으로 보여줄 문서 수         |
| 미포착    | 해당 L1 룰이 잡지 않은 문서 수                                            |

L1 책임 값의 의미:

| 값     | 의미                                                                |
| :----- | :------------------------------------------------------------------ |
| 예     | L1이 직접 잡아야 하는 대상. 잡으면 위험점수 또는 검사 제한 신호에 반영 |
| 아니오 | L1 책임 영역이 아님. 잡지 않는 것이 정상                            |
| 별도   | 형식상 잡히지만 확정 위험으로 보지 않고 검토/시스템/비감사 항목으로 분리 |

## 검증 축 C — 실제 조작 전표 포착력 (`manipulated_entry_truth.csv`)

`manipulated_entry_truth.csv`는 **실제 조작 전표 평가용 표본**입니다. 이 파일은 L1 특정 룰의 정답 파일이 아니라, 여러 fraud 시나리오로 만들어진 전체 조작 전표 420건 목록입니다.

따라서 C축은 A축/B축과 보는 기준이 다릅니다.

| 축 | 기준 파일 | 평가 질문 | 숫자의 의미 |
| :--- | :--- | :--- | :--- |
| A축 | `rule_truth_L1_*.csv` | 각 L1 룰 계약을 정확히 맞추는가 | 룰별 정답/탐지/과탐/미탐 |
| B축 | L1 sidecar population | 현실적인 후보를 점수/검토/별도로 잘 분리하는가 | 후보 모집단의 점수/검토 분기 |
| C축 | `manipulated_entry_truth.csv` | 실제 조작 전표가 L1 책임 영역에 얼마나 걸리는가 | 조작 전표 420건 중 L1 책임분 포착 범위 |

L1이 모든 조작 전표를 잡아야 하는 것은 아닙니다. L1은 전표 형식, 필수값, 승인 통제, 기간 정합성의 1차 통제 영역을 봅니다. 금액 이상, 반복 패턴, 관계사 구조, 계정 수준 변동 같은 조작 유형은 L2~L4 또는 D01/D02가 담당합니다.

### 전체 요약

| 항목 | 건수 | 의미 |
| :--- | ---: | :--- |
| 전체 조작 전표 | 420 | `manipulated_entry_truth.csv`의 실제 조작 전표 수 |
| L1이 잡아야 하는 전표 | 36 | 조작 전표 중 L1 공식 truth와 겹치는 책임분 |
| L1이 실제 잡은 전표 | 36 | L1 책임분 중 실제 포착한 전표 |
| L1이 잡지 않은 책임분 | 0 | L1이 잡아야 했지만 못 잡은 전표 |
| L1 책임 밖 조작 전표 | 384 | L2~L4 또는 D01/D02가 봐야 하는 전표 |

v126 기준으로는 전체 조작 전표 420건 중 L1이 책임져야 하는 36건이 모두 L1에 포착됩니다. 나머지 384건은 L1 실패가 아니라 L2~L4, D01/D02가 담당해야 하는 패턴형 조작입니다.
