# Detection Reference — 법규·기준서·도메인 지식

> **PHASE1 역할 원칙**: PHASE1은 `fraud`를 확정하거나 정답 라벨을 맞히는 단계가 아니다. PHASE1의 목적은 전수 모집단에서 규칙 위반, 정책 위반, 이상 징후, 분석적 검토 신호를 넓게 올려 **감사인이 봐야 할 항목과 우선순위**를 만드는 것이다. DataSynth의 `is_fraud`/`is_anomaly`와 precision/recall은 개발 검증 보조 지표이며, 운영 해석은 예외 처리 대상, 감사인 리뷰 대상, 고위험 후보를 구분하는 review queue 기준으로 한다.

전표 감사 검토 후보 선별 룰의 법적 근거, 실증 데이터, 도메인 지식을 정리한 참고 문서.
탐지 룰 목록과 구현 상세는 [DETECTION_RULES.md](DETECTION_RULES.md) 참조.

## 2. 감사기준서 → 룰 도출

> 기준서/법이 "이걸 검사하라"고 말한다 → 그 요구를 자동화하면 [DETECTION_RULES.md](DETECTION_RULES.md)의 룰이 된다.

### 2.1 감사기준서 240호 — 전표검사의 핵심

#### §32: 전표검사 의무

> 감사인은 경영진에 의한 내부통제 무력화 위험에 대한 평가 결과에 **관계없이**,
> 총계정원장에 기록된 분개기입과 재무제표 작성과정에서 이루어진 기타 수정사항의
> 적정성을 테스트하기 위한 감사절차를 설계하고 수행**하여야 한다**.
>
> (a) (i) 재무보고과정에 관여하는 개인에게 분개기입 및 기타 수정사항의 처리와 관련된
>         **부적절하거나 비정상적인 활동**에 대하여 질문한다.
>     (ii) **보고기간 말**에 작성된 분개기입 및 기타 수정사항을 선정하여 검사한다.
>     (iii) **보고기간 전체**에 걸쳐 분개기입 및 기타 수정사항을 검사할 필요성이 있는지 고려한다.
>
> (b) 회계추정치에 경영진의 **편의**(bias)가 개입되어 있는지를 검토한다.
>
> (c) 기업의 정상적인 영업과정을 벗어나거나 비정상적으로 보이는 **유의적 거래**에 대하여
>     그 **사업상 합리성**을 평가한다.

```
§32(a)(ii)  → L3-04 기말/기초 결산 검토 후보군
§32(a)(iii) → 현재 32개 L1~L4 룰 전체 (연중 대상)
§32(c)      → L4-01 매출 이상변동, L4-03 이상 고액
```

#### A44: 전표검사의 성격·시기·범위

> 부정에 의한 재무제표의 중요왜곡표시는 **부적절하거나 비인가된 분개기입**의 기록을 통해
> 이루어지는 경우가 많다. 이는 **보고기간 전체 또는 보고기간 말**에 발생할 수 있다.

→ L3-04 기말/기초 결산 검토 후보군 (기말 집중) + 나머지 21개 룰 (연중 전체 대상)

#### A45: 부정한 분개의 식별 특성

> 부적절한 분개기입 또는 기타 수정사항은 고유한 식별 특성을 가지는 경우가 많다.

| #   | 한국 감사기준서 원문                                                                  | 도출된 룰              |
|-----|--------------------------------------------------------------------------------------|------------------------|
| (a) | 관련 없는, 비경상적이거나 **거의 사용되지 않는 계정**에 대한 기입                        | **L1-03**, **L4-04**       |
| (b) | **통상적으로 분개기입을 하지 않는 개인**에 의한 기입                                     | **L3-02**                |
| (c) | **보고기간 말** 또는 마감후 수정분개로 기록되며 **설명이 거의 없거나 전혀 없는** 기입      | **L3-04~L3-07**, **L3-08**   |
| (d) | 재무제표 작성 전이나 작성 중에 기록되며 **계정번호가 없는** 기입                          | **L1-02**                |
| (e) | **단수(round number)** 또는 **일관된 끝자리 숫자**를 포함하는 기입                       | **L2-01**, **L4-02**       |

#### A46: CAATs 활용

> 감사인은 **컴퓨터 보조 감사 기법**을 사용하여 검사 대상 분개기입 또는 기타 수정사항을
> 식별할 수 있다 (예: 총계정원장에서 **비정상적인 분개기입**을 식별).

→ 이 프로젝트 자체가 A46의 구현체. 현재 32개 L1~L4 룰 = CAATs 쿼리를 Python으로 자동화한 것.

### 2.2 외감법 §8 — 내부회계관리제도 (K-SOX)

#### §8 제1항: 내부회계관리제도 구축 의무

> 회사는 신뢰할 수 있는 회계정보의 작성과 공시를 위하여 다음 각 호의 사항이 포함된
> 내부회계관리규정과 이를 관리·운영하는 조직(이하 "내부회계관리제도"라 한다)을 갖추어야 한다.

이 조항들이 전표에 대해 요구하는 통제를 자동 검증하면:

```
1호 "식별·측정·분류·기록 및 보고 방법"
  → L3-08 적요 결손/파손 — 적요가 공백이거나 깨졌으면 "기록 방법" 위반
  → L3-07 전기일-문서일 장기 괴리 — 전표상 문서일과 전기일 괴리가 과도하면 "적시 기록" 검토 신호

2호 "오류를 통제하고 수정하는 방법"
  → L1-01 차대변 균형 — 차대 불일치 = 오류 통제 미작동
  → L1-02 필수필드 누락 — 계정번호 등 누락 = 오류 통제 미작동

4호 "위조·변조·훼손 및 파기를 방지하기 위한 통제"
  → L2-03 중복 전표 — 동일 전표 반복 = 위조 징후
  → L2-02 중복 지급 — 동일 건 이중 지급 = 변조 또는 오류

5호 "업무 분장과 책임"
  → L1-05 자기 승인 — 입력자 = 승인자 = 업무 분장 위반
  → L1-06 직무분리 위반 — 동일인이 전체 프로세스 수행
  → L2-01/L1-04 승인한도 — 승인 권한 체계 우회/미작동
  → L1-07 승인 생략 — 승인 절차 자체를 건너뜀
```

#### §8 제2항: 우회 금지

> 누구든지 내부회계관리제도에 의하지 아니하고 회계정보를 작성하거나
> 내부회계관리제도에 따라 작성된 회계정보를 **위조·변조·훼손 및 파기**하여서는 아니 된다.

```
→ L3-02 수기 전표 — source='manual'인 고액 전표 = 자동 프로세스 우회
→ L1-07 승인 생략 — 승인 절차 없이 처리된 한도 초과 전표 = §8② 직접 위반
```

> **금감원 실증**: 오스템임플란트(2021) — 재무팀장 1인이 입력·승인·이체 전부 수행, 2,215억 횡령.
> 금감원 권고: "전표 입력 시 적절한 승인권자의 승인이 이루어질 수 있도록 통제절차를 마련하여야 한다."

#### §8 제6항: 감사인의 내회관 검토/감사 의무

> 감사인은 회계감사를 실시할 때 해당 회사가 이 조에서 정한 사항을 **준수했는지 여부**를 검토하여야 한다.

→ 현재 32개 L1~L4 룰 전체가 §8⑥의 "준수 여부 검토" 수단

### 2.3 감사기준서 315호 §26(a)(ii) — 전표 통제 이해

> 감사인은 재무보고와 관련된 기업의 정보시스템에 대한 이해를 획득하여야 하며,
> 여기에는 **분개기입에 대한 통제**가 포함된다. 특히 비경상적이거나 비정상적인 거래
> 또는 수정사항을 기록하는 데 사용되는 **비표준 분개기입에 대한 통제**를 포함한다.

→ L1-05 자기 승인, L1-06 직무분리, L3-02 수기 전표 — 비표준 전표 통제 위반 탐지

### 2.4 감사기준서 520호 §5 — 분석적절차

> 감사인은 실증적 분석적절차를 설계하고 수행할 때 다음을 하여야 한다.
> (a) 특정 주장에 대하여 해당 실증적 분석적절차의 적합성을 결정한다.
> (c) 기록된 금액이나 비율에 대한 **기대값을 개발**하고, 그 기대값이 충분히 정확한지를 평가한다.
> (d) 추가적인 조사 없이 수용할 수 있는, 기록된 금액과 **기대값 간의 차이 금액**을 결정한다.

```
→ L4-02 Benford 위반 — Benford 분포 = 기대값, 실제 분포와의 차이 = MAD/KS 검정
```

> 520호에 Benford's Law 명시는 없으나, "기대값 개발 → 차이 분석" 프레임워크에 정확히 부합.
> Mark Nigrini (2012), Journal of Accountancy 등 학술·실무에서 분석적절차 도구로 널리 인정.

### 2.5 감사기준서 550호 §23 — 특수관계자

> 기업의 정상적인 영업과정을 벗어나는 것으로 식별된 유의적 특수관계자 거래에 대하여,
> 감사인은 그 근거가 되는 약정이나 합의사항을 검사하고, 해당 거래의 **사업상 합리성**이
> 그 거래가 **부정한 재무보고에 관여**하거나 **자산의 유용을 은폐**하기 위하여
> 이루어졌을 수 있음을 시사하는지 평가하여야 한다.

→ L3-03 관계사 거래 검토 신호 — 관계사 거래 모집단 및 후속 구조 분석 후보

> **금감원 실증**: 2024년 중점심사 4대 이슈 중 하나로 "특수관계자거래 회계처리" 선정.

### 2.6 감사기준서 330호 — 실증절차·재무제표 대사

> 감사인은 재무제표 작성 과정에 대해 재무제표와 기초 회계기록을 **대사·조정**하고,
> 중요한 전표 및 기타 조정분개를 **검사**하여야 한다.

현재 32개 L1~L4 룰 전체가 330호가 요구하는 전표 실증절차를 구현한 것이다.
통제테스트(TOE) vs 실증절차: 내부통제 양호 → TOE 표본 검사, 미흡 → 실증절차 확대.

```
→ 현재 32개 L1~L4 룰 = 330호 실증절차의 자동화
→ 재무제표-장부 대사 = Phase 2에서 Trial Balance 교차검증으로 구현 예정
```

### 2.7 감사기준서 500호 — 감사증거

> 감사인은 감사증거의 충분성·적합성을 평가하여야 한다.
> 전표-증빙 대조(계약서, 세금계산서, 영수증)는 핵심 절차.

현재 32개 L1~L4 룰은 전표 데이터 기반 패턴 탐지에 집중하며, 증빙 대조는 미포함. L3-12는 L1-06과 분리된 업무범위 review signal이다.
DataSynth에 증빙 관련 컬럼 추가 시 구현 가능.

```
→ 현재: L3-08(위험 적요)이 적요 기반으로 부분 커버
→ 확장: 증빙 존재 확인, 증빙 금액 불일치, 적격증빙 검증 (DataSynth 확장 시)
```

### 2.8 감사기준서 1100호 — 내부회계관리제도 감사

> 감사인은 내부회계관리제도의 준수 여부를 검토/감사하여야 한다 (외감법 §8⑥).

외감법 §8(§2.2에서 다룸)은 피감사회사의 **구축 의무**.
감사기준서 1100호는 감사인이 그 통제를 **어떻게 검사하는지**의 기준.

```
→ L1-05(자기승인), L1-06(직무분리), L3-02(수기전표), L1-07(승인생략) = 내회관 통제 결과 추론
→ DataSynth approval 활성화 시 통제 운영 효과성(승인 누락률, 지연, 레벨 우회)도 검증 가능
```

### 2.9 IT 감사 기준 — KLCA·KICPA JET·금융권 가이드라인

전표 감사의 IT 관점 검사항목:

```
검사항목              세부                              프로젝트 대응        출처
─────────────────────────────────────────────────────────────────────────────────
입력/변경 로그         입력·수정·삭제 시점, 사용자ID      DataSynth 확장 시     KLCA 4.3~4.5
비정상 시간대          심야/주말 입력자 집중              L3-05, L3-06 + 확장       KLCA
계정·금액 무결성       Benford, 비정상 계정 조합          L4-02, L4-04             KICPA JET 3.2
승인 흐름 추적         생성→승인→포스팅 로그              L1-05~L1-07 + 확장        315호/330호 ITGC
배치 통제             자동 전표 IT 승인, 배치 실패        Phase 2              금융권 가이드라인
```

### 2.10 요약 — 기준서 조항 → 룰 도출 흐름

```
감사기준서 240호 §32     "전표의 적정성을 테스트하여야 한다"    → 현재 32개 L1~L4 룰 전체
  ├─ §32(a)(ii)          보고기간 말 전표 선정·검사             → L3-04
  ├─ §32(c)              비경상적 중요 거래 합리성 평가          → L4-01, L4-03
  ├─ A44                 기말 집중 + 전체 기간 필요성 고려       → L3-04 + 나머지
  ├─ A45(a)              unrelated, unusual, seldom-used accounts → L1-03, L4-04
  ├─ A45(b)              individuals who typically do not make JE  → L3-02
  ├─ A45(c)              end of period + little or no explanation  → L3-04~L3-07, L3-08
  ├─ A45(d)              do not have account numbers               → L1-02
```

## 4. 3축 평가 방법론

DataSynth가 정의한 52개 anomaly 유형을 아래 3개 축으로 평가하여 채택 범위를 확정한다.
평가 결과 전체 목록은 [DETECTION_RULES.md 부록 A](DETECTION_RULES.md#부록-a-52개-유형-3축-평가-전체-목록) 참조.

### 축 정의

```
축 1. 법규 근거 (§2 기반)
  3 = 기준서 직접 명시 (240-A45/A49 개별 항목, 520§5, 550§23)
  2 = 기준서 포괄 근거 (240§32 "적정성 테스트")
  1 = 내회관(외감법§8)만, 기준서 미명시
  0 = 한국 법규 매핑 불가

축 2. 실증 빈도 (§3 FSS 189건 기반)
  3 = FSS 주요 패턴 20건+ (가공전표 50, 결산수정 27, 횡령은폐 24)
  2 = FSS 5~19건 (순환거래 10, 승인/SoD위반 5)
  1 = FSS 1~4건 (비정상시점 4)
  0 = FSS 사례 없음

축 3. 데이터 가용성 (39개 컬럼 스키마 기반)
  3 = 39개 컬럼만으로 즉시 탐지 (룰 기반 if문)
  2 = 파생 피처·통계 모델 필요 (ML/통계)
  1 = 외부 마스터 데이터 또는 NLP/그래프 필요
  0 = 현재 스키마로 탐지 불가
```

### 채택 판정 기준

```
Must  (7~9점) → Phase 1 필수 구현 (룰 기반)

Drop  (0~1점 또는 특별 사유) → 제외
```

## 6. 한국 실무 보조 지식

### 6.1 전결규정 (기업별 편차 큼, 설정값으로 외부화 필수)

```
~3만원       적격증빙 면제 (간이영수증 인정)
~100만원     자동 승인
~5천만원     팀장 1단계
~1억원       +본부장/이사 2단계
~2억원       +부서장/상무 3단계
~5억원       +CFO 4단계
5억원+       +CEO/이사회 5단계
```

> 3만원 초과 거래는 적격증빙(세금계산서/카드/현금영수증) 수취 의무.
> 미수취 시 **증빙불비가산세 2%** (세법). 3만원 직하 분할(29,000원×N건)도 탐지 대상.

### 6.2 한국 세법 적격증빙

- 3만원 초과 거래 → 세금계산서/계산서/카드매출전표/현금영수증 중 하나 필수
- `amount > 30,000` + `doc_type NOT IN ('세금계산서','카드','현금영수증')` → 적격증빙 미수취 의심
- 3만원 직하 분할 탐지: 동일 거래처 + 동일일 + 29,000원 이하 × N건 → 증빙 회피 의심

### 6.3 K-IFRS 표준계정과목 코드 체계

```
자산: 1xx~2xx (101~250)   부채: 2xx~3xx (251~330)   자본: 3xx~4xx (331~400)
매출/수익: 4xx (401~450)   매출원가: 4xx~5xx (451~500)
판관비: 8xx (801~899)      영업외비용: 9xx (901~960)
```

> SAP 한국 구축 시 6자리(예: 411000=매출, 811000=급여) 또는 8자리 사용이 일반적

### 6.4 SAP 한국 실무

#### 역분개 (Reversal)

- 한국 SAP: 전기된 전표는 직접 수정 불가 → 역분개(FL3-02) + 재전기 방식
- SAP Reversal Reason Code (RRC):
  - 05: Accrual Reversal — 발생주의 자동 역분개 (정상)
  - 01: 당기 역분개 — 수정 전표 (검토 대상)
  - 02: 타 기간 역분개 — 결산 후 수정 (고위험)

#### 전표유형 (Document Type)

```
SA=일반분개  KR=매입전표  KZ=매입지급  DR=매출전표  DZ=매출수취  AA=자산전기
```

#### 전표번호 체계

- `company_code + fiscal_year + document_type`별 번호범위 부여
- UUID(document_id)와 별도로 순차 `document_number`(int) 운영

### 6.5 한국 근무 시간 패턴

비정상 시간대 판정 시 한국 야근 문화 반영 필요 (평균 퇴근 18:19, 시프티 2024):

- **정상**: 08:30~18:30
- **야근(저위험)**: 18:30~22:00 — 결산기(12~1월)에는 정상 취급
- **심야(고위험)**: 22:00~06:00 — 감사 플래그 대상

### 6.6 한국 대기업 네트워크 구조

IP 주소 이상 탐지 시 참고:

```
사내(본사): 10.1.x.x    사내(공장): 10.2.x.x
VPN/재택:   10.10.x.x   외부(공인): 203.x.x.x 등
```

- 코로나 이후 재택근무(VPN) 비율 ~34% 반영: VPN 접속 자체는 정상
- 심야+VPN+고액은 가중

### 6.7 한국 결산 일정

- 97% 이상 12월 결산
- 밀어내기(channel stuffing) 집중 시기: **12월 마지막 2주**
- 1월~3월: 결산 조정 기간 (법인세 신고기한 3월 말)
- 상법 제33조: 장부 10년, 전표류 5년 보존 의무

## 7. 프로젝트 범위와 한계

### 프로젝트 범위

이 프로젝트는 **ERP에서 추출된 전표 CSV 데이터**를 입력으로 받는 CAATs(컴퓨터 보조 감사 기법) 도구이다.
전표 데이터에 대한 **전수 검사**를 수행하며, 감사인이 후속 수작업을 수행할 때의 **우선순위 추천**을 제공한다.

### 범위 밖 (2건)

| 항목 | 이유 |
|------|------|
| 배치 실패 로그 재처리 | 운영 로그는 전표 단위가 아닌 배치 작업 단위. 전표 CSV 구조와 다름 |
| 관련자 질문 절차 | 감사인의 대면 활동. 데이터로 대체 불가. 단, "이상 전표 집중 사용자 Top-N" 리포트로 질문 대상 추천 |

### 불필요 (5건 — 전수검사 프로젝트 특성)

| 항목 | 이유 |
|------|------|
| 감사기준서 530호 (표본추출) | 전수 검사 프로젝트 |
| 위험기반 표본 선정 | 전수 검사. 단, 대시보드에서 증빙 대조 대상 우선순위 제공 |
| 감사조서 문서화 | 감사인 산출물. CAATs 역할은 탐지 결과 제공 |
| 530호 참조 출처 | 전수조사이므로 해당 없음 |
| 내회관 Q&A 실무자료 | 통제테스트 표본수 → 전수조사이므로 해당 없음 |

### DataSynth 확장으로 해소 가능한 영역

현재 39개 컬럼에는 없지만, DataSynth에 컬럼을 추가 생성하면 구현 가능한 탐지 항목:
- 증빙 관련: `has_attachment`, `supporting_doc_type`, `invoice_amount`, `delivery_date`
- 승인 관련: `approval_level` (approved_by/approval_date는 39개 컬럼에 포함됨)
- 변경 이력: `changed_by`, `change_date`, `changed_field`
- 전표번호: 순차 `document_number` (UUID와 별도)
- IP 추적: `ip_address`

상세 구현 로직: [DETECTION_RULES.md §3.3](DETECTION_RULES.md#33-신규-컬럼-기반-룰-datasynth-확장-후-구현) 참조.
