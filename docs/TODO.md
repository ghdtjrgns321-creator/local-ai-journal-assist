  1. 역방향 leakage — 정상이 너무 완벽함                                                                                                               
                                                                                                                                                         정상 전표가 전부 깨끗하면 ML은 "노이즈가 없으면 정상"으로 학습합니다. 실제 데이터에는:                                                                 - 적요 오타, 빈 적요 (~15%)                                                                                                                          
  - 승인 지연 (posting_date 후 3~5일 뒤 approval)                                                                                                      
  - 소액 차대변 불균형 (반올림 오차 1~2원)                                                                                                             
  - 동일 금액 반복 결제 (정상적인 월세/급여)

  → 정상 전표의 "불완전도"가 비정상과 비슷한 수준인지 확인 필요

  2. 복합 피처 leakage

  단일 컬럼은 통과해도 2~3개 컬럼 조합이면 완벽 분리될 수 있음:
  - (gl_account=1190) + (amount > 5000만) → 100% fraud
  - (source=Manual) + (hour >= 22) + (document_type=SA) → 90%+ fraud
  - (user_persona=manager) + (approved_by=self) → 특정 anomaly

  3. 시계열 구조

  전표는 독립 행이 아니라 시간 순서가 있는 시퀀스:
  - fraud가 시간적으로 균일하게 분산되어 있으면 비현실적 (실제로는 클러스터링)
  - 특정 user가 fraud를 연속으로 발생시키는 패턴이 있어야 함
  - 월말 직전 fraud 집중 (결산 마감 압박)

  4. 전표 내부 구조 (line-level)

  지금까지 document 단위만 봤는데 라인 조합도 중요:
  - P2P 전표: 차변=재고(1xxx), 대변=AP(2000) — 이게 아닌 조합이면 비현실적
  - 급여 전표: 차변=인건비(6100), 대변=미지급(2xxx) — 고정 패턴
  - fraud 전표만 "아무 GL + 아무 GL" 랜덤 조합이면 그 자체가 leakage

  5. 금액의 Benford per GL group

  글로벌 Benford는 통과해도 계정군별 Benford가 비현실적일 수 있음:
  - 급여(6100): 첫 자리 3~8 집중 (300만~800만)
  - 감가상각: 거의 동일 금액 반복
  - 매출(4xxx): Benford 잘 따름

  → 모든 GL이 동일 Benford를 따르면 합성 데이터 artifact

  6. 관계형 일관성 (journal_entries 외부)

  CSV 단독이 아니라 다른 생성 파일과의 교차:
  - employees.json의 department와 GL 계정 매칭
  - vendors.json의 업종과 거래 금액 범위
  - P2P chain: PO → GR → Invoice → Payment 순서와 금액 일치

  핵심 이슈 해석

  1. 금액 중앙값 2만원 문제
  - LogNormal(14.0, 2.5) 설정이면 중앙값이 ~120만원이어야 하는데, 라인 레벨 중앙값은 2만원
  - 원인: 전표당 라인이 10.2개로 많아서, 전표 총액을 라인별로 sample_summing_to()로 분할할 때 개별 라인 금액이 급격히 작아짐 
  - 전표 단위 총액 중앙값은 138만원으로, 이건 120만원 기대값에 가까움 → 총액 수준에서는 합리적

  2. 전표당 라인 10.2개
  - 한국 중견 제조업 실무에서는 일반적으로 2~5개 라인(차변 1~2, 대변 1~2 + 세금)
  - 10개 이상은 결산 종합 전표나 급여 전표에서나 볼 수 있는 수치
  - 이건 Rust의 debit_credit_distribution 설정에서 라인 수 분포를 조정해야 할 수 있음

  3. GL 분포 특이점
  - 100xxx 6자리 GL이 상위에 많이 보이는데, 이는 보조원장 계정으로 보임
  - 1500(재고)/1510(대여금) 등 자산 계정이 최상위 → 제조업 특성 반영
