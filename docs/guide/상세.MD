# Local AI Audit Assistant 상세 정리

> **Local-first boundary (2026-05-26)**: PHASE3 LLM Narrator, LLM reranking, AI review memo, Text-to-SQL, and LLM rule feedback are removed from active product capability. Active product positioning is local PHASE1/PHASE2 ledger analytics plus deterministic Local Evidence Brief. See [LOCAL_FIRST_EVIDENCE_POLICY.md](../spec/LOCAL_FIRST_EVIDENCE_POLICY.md), [DECISION.md §D068](../spec/DECISION.md), and deprecated [PHASE3_REVIEW_NARRATOR_SPEC.md](../archive/abandoned/PHASE3_REVIEW_NARRATOR_SPEC.md).


> **포트폴리오 주장 범위 (2026-05-19)**: 이 프로젝트는 `fraud`를 판정하거나 실제 운영 부정 탐지 성능을 보장하는 모델이 아니다. 전수 모집단에서 감사인이 먼저 볼 review queue를 만들고, 무작위 검토 대비 상위 구간에 review-worthy synthetic anomaly를 강하게 농축하는 로컬 감사 분석 보조 도구다. DataSynth 기반 precision/recall은 개발 검증 보조 지표이며, 실데이터 운영 성능으로 주장하지 않는다.
> **금지 표현**: "부정을 정확히 탐지", "실무 운영 성능 검증 완료", "TOP100 precision 충분", "fraud 확정/자동 적발"처럼 확정적이거나 운영 성능을 보장하는 표현은 사용하지 않는다.

이 문서는 프로젝트의 핵심 흐름과 실제로 동작하는 분석 축을 짧고 명확하게 설명하기 위한 상세 문서다.  
목적은 "기능이 많아 보이는 시스템"이 아니라, 지금 기준으로 무엇이 핵심 가치이고 무엇이 보조 기능인지 구분해서 보여주는 데 있다.

> **포트폴리오 포지셔닝 기준 (2026-05-19)**: 이 프로젝트는 실제 부정을 확정하는 fraud detector가 아니다. 전수 원장/지급 데이터에서 감사인이 검토해야 할 후보를 설명 가능한 review queue로 정렬하고, 근거와 다음 감사 절차를 제공하는 로컬 감사 분석 보조 도구다. DataSynth는 메인 데모, OpenDataPhilly/Tritscher는 보조 검증으로 둔다.

## 1. 이 프로젝트를 한 문장으로 설명하면

> 회계 전표 데이터를 업로드하면 회사별 기준으로 정리하고 검증한 뒤, **Phase 1 규칙 기반 탐지**와 **Phase 2 비지도 이상탐지**를 중심으로 감사인이 검토해야 할 후보를 찾고, 결과를 설명 가능한 review queue와 대시보드, DB, 리포트로 연결하는 로컬 감사 분석 시스템이다.

이 설명에서 핵심은 두 가지다.

- `Phase 1 규칙 기반 탐지`
- `Phase 2 비지도 이상탐지`

현재 제품의 메인 분석 축은 이 두 가지다.
다만 이 축의 목적은 부정 확정이 아니라 감사 검토 우선순위와 근거 제공이다.

---

## 2. 왜 분석 축을 다시 정리해야 하는가

프로젝트에는 확장 가능한 기능이 많다.

- 규칙 기반 탐지
- 비지도 모델
- 지도학습 모델
- 통계/리포트 기능
- Local Evidence Brief
- 향후 NLP/그래프 탐지

문제는 이 기능들을 같은 무게로 소개하면 사용자가 "지금 실제로 무엇이 주력으로 동작하는가"를 이해하기 어렵다는 점이다.

그래서 현재 기준으로는 다음처럼 보는 것이 맞다.

- `Core`: Phase 1 규칙 기반 + Phase 2 비지도
- `Optional`: 지도학습
- `Experimental`: 고급 보조 모델
- `Future`: local-only NLP, 그래프, 고급 local model inference

지도학습/딥러닝/스태킹 계열은 합성 데이터 성능이 높더라도 기본 경로로 승격하지 않는다. 실제 회사 데이터나 회계사가 검토한 golden set이 없으면 생성기 shortcut을 학습할 위험이 크기 때문이다.

## 3. 실제 분석 축

### 3.1 Core 1: Phase 1 규칙 기반 탐지

감사인이 바로 이해하고 설명할 수 있는 탐지 축이다.

- 무결성 이상
- 승인/권한 위반
- 이상 징후 패턴
- Benford 같은 통계 규칙

장점은 명확하다.

- 왜 걸렸는지 설명 가능하다.
- 감사 문맥과 바로 연결된다.
- 결과를 검토 우선순위로 넘기기 쉽다.

### 3.2 Core 2: Phase 2 비지도 이상탐지

규칙으로 미리 정의하기 어려운 패턴을 보완하는 축이다.

- 정상 거래 분포를 학습한다.
- 그 분포에서 벗어나는 거래를 찾는다.
- 현재 메인 방향은 `VAE + Isolation Forest` 계열이다.

이 축이 중요한 이유는 감사 데이터에서 항상 충분한 정답 라벨이 있는 것은 아니기 때문이다.

### 3.3 왜 지도학습이 메인이 아닌가

지도학습은 가치가 있지만, 현재 제품의 기본 설명 축으로 놓기에는 제약이 크다.

- 충분한 정답 라벨이 필요하다.
- pseudo-label만으로는 기존 룰이나 비지도 결과를 다시 학습하는 구조가 되기 쉽다.
- 고객사별 반복 운영과 HITL 축적 이후에야 진짜 가치가 커진다.
- DataSynth의 truth는 기능 검증용 합성 라벨이므로 supervised 모델이 실제 부정 패턴이 아니라 생성기 흔적을 배울 수 있다.

따라서 지금은 `optional`로 보는 것이 맞다.

### 3.4 코드 기준으로 보면 ML/DL은 어떻게 동작하는가

룰기반 탐지와 달리 ML/DL은 원본 파일 형식을 직접 먹지 않는다.  
먼저 ingest에서 회사별 컬럼명을 표준 스키마로 맞추고, 그 다음 feature generation에서 시간/금액/패턴/텍스트 계열 감사 피처를 공통 생성한 뒤, 그 위에서 모델별 전처리와 학습이 갈라진다.

- 공통 입력 레이어: `ingest -> validation -> feature generation`
- 공통 피처: 시간, 금액, 패턴, 텍스트 계열 파생변수
- 모델 분기: 비지도(VAE/IF), 지도학습(LR/RF/XGB/LGBM), 딥러닝(FT-Transformer, BiLSTM+Attention), 최종 앙상블(Stacking)

즉 회사마다 들어오는 자료가 달라도, 모델은 "원본 ERP 파일"이 아니라 "표준화된 감사 피처 테이블"을 기준으로 본다.

#### 3.4.1 회사마다 자료가 다른데 어떻게 흡수하는가

코드상 흡수 방식은 크게 세 단계다.

1. **표준 컬럼 매핑**
   - 회사별 원본 헤더를 `document_id`, `posting_date`, `debit_amount`, `credit_amount`, `gl_account` 같은 공통 컬럼으로 정규화한다.

2. **공통 피처 생성**
   - feature 엔진이 시간/금액/패턴/텍스트 피처를 일괄 생성한다.
   - 예: `is_after_hours`, `days_backdated`, `amount_zscore`, `is_round_number`, `is_manual_je`, `is_suspense_account`

3. **학습용 품질 보정**
   - `user_persona` 같은 도메인 컬럼은 표준값으로 정규화한다.
   - 결측률이 매우 높은 컬럼은 제외한다.
   - `cost_center`, `tax_code`, `trading_partner`처럼 회사별 편차가 큰 옵션성 컬럼은 coverage가 낮으면 학습에서 자동 제외한다.

따라서 실제 전략은 "회사별 파일 차이를 모델이 직접 견디게 한다"가 아니라, "모델에 들어가기 전에 공통 표현으로 압축한다"에 가깝다.

#### 3.4.2 모델별 전처리는 어떻게 다른가

같은 피처 테이블을 쓰더라도 모델별로 전처리 방식은 다르다.

- **지도학습 LR/RF/XGB/LGBM**
  - 숫자형: median imputation
  - 고카디널 범주형: TargetEncoder
  - 저카디널 범주형: OrdinalEncoder
  - 불리언: passthrough
  - 목적: 표 형태 피처에서 바로 분류 성능을 내는 것

- **비지도 VAE / Isolation Forest**
  - 숫자형: median imputation + power transform + standard scaling
  - 고카디널 범주형: 타깃 의존 인코딩을 쓰지 않기 위해 제외
  - 목적: 정상 분포에서 벗어난 패턴을 찾는 것

- **FT-Transformer**
  - 표 형태 피처를 그대로 받아 self-attention으로 관계를 학습
  - 입력은 탭уляр 데이터지만 학습기는 딥러닝 모델

- **BiLSTM + Attention**
  - `created_by`, `posting_date`, `posting_time` 기준으로 사용자별 시퀀스를 구성
  - 2D tabular feature를 3D sequence window로 바꿔 시계열/행동 패턴을 본다
  - 목적: 같은 사용자나 같은 흐름에서 반복되는 이상 입력 패턴을 찾는 것

정리하면, "전처리는 하나"가 아니라 "공통 피처 레이어 하나 + 모델별 최종 전처리 여러 개" 구조다.

#### 3.4.3 모델별 비교는 어떻게 하는가

이 프로젝트는 모델 비교를 단순 random split으로 하지 않는다.  
핵심은 **문서 누수(document leakage)** 와 **사용자 누수(user leakage)** 를 막는 것이다.

- 지도학습 후보 비교
  - LR / RF / XGB / LightGBM을 `GroupKFold(document_id)`로 비교
  - 지표는 기본적으로 `F1-macro`
  - 가장 높은 평균 F1 모델을 자동 선택

- train / validation 분리
  - 가능하면 연도 기준 temporal holdout
  - 그게 안 되면 document group holdout
  - 같은 전표(document_id)가 train/test 양쪽에 동시에 들어가지 않도록 방지

- threshold 최적화
  - 최고 모델을 고른 뒤 끝내지 않고, validation set에서 anomaly threshold를 다시 탐색
  - 따라서 "모델 선택"과 "실제 flag 기준"이 분리돼 있다

- 시퀀스 모델
  - 사용자 단위로 split해서 같은 사용자의 패턴이 train/validation에 동시에 섞이지 않게 함

- 앙상블
  - rule/ML/DL 결과를 그냥 평균내지 않는다
  - leakage 위험이 있는 supervised/transformer/sequence는 OOF(out-of-fold) 점수를 다시 만들고
  - 그 점수들 위에서 stacking meta-learner를 학습한다
  - 라벨이 부족하면 percentile ranking fallback으로 내려간다

즉 이 코드베이스의 모델 비교는 "무슨 알고리즘이 더 좋냐"만 보는 게 아니라,
"누수 없이 비교했을 때도 유지되는 성능인가"를 함께 본다.

#### 3.4.4 라벨은 어떻게 다루는가

지도학습이 optional인 이유도 여기에 있다.

- DataSynth GT가 있으면 ground-truth 라벨 사용
- HITL feedback이 쌓이면 feedback 라벨 사용 가능
- pseudo-label은 circular risk로 간주
- 신뢰할 수 없는 라벨 소스면 supervised를 막고 unsupervised fallback으로 보냄

즉 "라벨이 있으니까 무조건 supervised"가 아니라,
"그 라벨이 학습에 써도 되는 출처인가"를 먼저 게이트로 검증한다.

#### 3.4.5 실무적으로 어떻게 이해하면 되는가

실무적으로는 아래처럼 이해하는 것이 가장 정확하다.

- **룰기반**: 설명 가능한 이상을 먼저 찾는 1차 필터
- **비지도 ML**: 규칙 밖 패턴을 보완하는 기본 Phase 2 축
- **지도학습/딥러닝**: 라벨 품질과 누수 방어가 확보될 때 성능을 더 끌어올리는 옵션
- **앙상블**: 여러 탐지기의 신호를 누수 없이 재조합하는 최종 계층
- **Local Evidence Brief**: 후보를 확정하지 않고 이미 계산된 로컬 근거와 다음 감사 절차를 deterministic하게 요약하는 계층

그래서 이 프로젝트의 ML/DL은 "회사별 raw 데이터에 모델을 바로 붙이는 구조"가 아니라,
"감사용 공통 피처 계층 위에 여러 학습기를 얹고, 누수 방지 split과 게이트로 품질을 통제하는 구조"라고 보는 것이 맞다.

## 4. 사용자 흐름

사용자 관점의 기본 흐름은 아래와 같다.

```text
회사 선택
-> 수행연도 선택
-> 파일 업로드
-> ingest / validation / feature generation
-> Phase 1 규칙 기반 탐지
-> Phase 2 비지도 보완
-> PHASE1/PHASE2 queue 검토
-> Local Evidence Brief / 리포트 / export
```

여기서 Local Evidence Brief와 Export는 메인 탐지 축 그 자체라기보다 탐지 이후 감사인이 근거를 이해하고 다음 절차를 정하는 활용 단계에 가깝다. Chat, 외부 LLM 제안, Text-to-SQL은 active product path가 아니다.

## 6. Local Evidence Brief의 위치

이 프로젝트에서 외부 LLM/API는 active product path가 아니다.

- PHASE1 rule evidence
- PHASE2 family lane signal
- case metadata
- recommended audit actions

위 근거를 로컬 deterministic template으로 요약하는 Local Evidence Brief만 active 설명 경로로 둔다.

즉, "LLM이 들어간 시스템"이 아니라 "외부 API 없이 원장을 분석하고, 로컬 근거를 구조화해 보여주는 감사 분석 시스템"으로 설명하는 편이 맞다.

## 7. 발표용 설명

### 30초 버전

이 프로젝트는 회계 전표 데이터를 분석하는 로컬 감사 분석 시스템이다.  
핵심은 Phase 1 규칙 기반 탐지로 설명 가능한 이상을 먼저 찾고, Phase 2 비지도 이상탐지로 규칙 밖 패턴을 보완하는 구조다.  
Chat이나 Export 같은 기능도 있지만, 메인 가치는 분석 파이프라인 자체에 있다.

### 1분 버전

실무 회계 데이터는 회사마다 형식이 다르고, 결과는 설명 가능해야 한다.  
그래서 파일을 업로드하면 ingest, validation, feature generation을 거친 뒤 먼저 감사 규칙 기반 탐지로 설명 가능한 이상을 찾고, 이어서 비지도 이상탐지로 정상 분포에서 벗어난 패턴을 보완하도록 설계했다.  
지도학습은 라벨 품질이 확보될 때 확장 가능하지만, 현재 메인 분석 축은 Phase 1 규칙 기반과 Phase 2 비지도 탐지다. 외부 LLM/API 기능은 active product path가 아니다.

## 8. 정리

이 프로젝트의 메시지는 단순하다.

- 기능이 많아 보이는 시스템이 아니다.
- 지금 실제로 동작하고 가치가 있는 분석 축이 분명한 시스템이다.

그 중심 축은 `Phase 1 규칙 기반 + Phase 2 비지도 이상탐지`다.

## 9. DataSynth 라벨 누수 방지 설계

```text
                  ┌─────────────────────────┐
                  │   DataSynth 원본 CSV    │
                  │   (본문 + 라벨 포함)    │
                  └────────────┬────────────┘
                               │
              ┌────────────────┴────────────────┐
              ▼                                 ▼
      ┌───────────────┐                 ┌───────────────┐
      │   본문 추출   │                 │   라벨 분리   │
      │  (라벨 제거)  │                 │   (Sidecar)   │
      └───────┬───────┘                 └───────┬───────┘
              │                                 │
       ┌──────┴──────┐                          │
       ▼             ▼                          ▼
  ┌─────────┐  ┌─────────┐              ┌──────────────┐
  │ Phase 1 │  │ Phase 2 │   허용 라벨   │  Supervised  │
  │  규칙   │  │  모델   │ ◀──────────── │     Gate     │
  │  탐지   │  │  학습   │               │ (label_source)│
  └─────────┘  └────┬────┘               └──────┬───────┘
                    │                           │
                    └────────────┬──────────────┘
                                 ▼
                        ┌─────────────────┐
                        │   Evaluation    │
                        │   성능 평가     │
                        │ (Ground Truth)  │
                        └─────────────────┘
```

### 9.1 박스별 역할

| 박스                     | 역할                                                                           | 왜 필요한가                                                           |
| ------------------------ | ------------------------------------------------------------------------------ | --------------------------------------------------------------------- |
| **DataSynth 원본 CSV**   | 본문(전표 데이터)과 정답 라벨이 한 파일에 섞여 있는 합성 데이터                | 모델 평가용 GT가 필요해서 원본에는 라벨이 들어 있다                   |
| **본문 추출**            | 원본에서 라벨 컬럼을 제거하고 순수 전표 데이터만 남김                          | 모델이 라벨을 "훔쳐보지" 못하게 하기 위함 (라벨 누수 방지의 1차 차단) |
| **라벨 분리 (Sidecar)**  | 라벨만 따로 저장하고, 원본 위치(`source_path`)로 연결고리 유지                 | 평가 때만 꺼내 쓸 수 있도록 본문과 물리적으로 분리                    |
| **Phase 1 규칙 탐지**    | 라벨 없이 감사 규칙으로만 이상 탐지                                            | 설명 가능한 결과를 우선 확보                                          |
| **Phase 2 모델 학습**    | 라벨 없는 본문으로 학습 (VAE/Sequence) 또는 Gate 통과 라벨로 학습 (Supervised) | 비지도/지도 모두 누수 없이 동작                                       |
| **Supervised Gate**      | 허용된 `label_source`만 통과시키는 검문소                                      | 학습용 라벨인지, 평가용 라벨인지 구분해서 학습에 들어갈 라벨을 통제   |
| **Evaluation 성능 평가** | Sidecar 라벨을 GT로 꺼내와 Phase 1/2 결과의 정밀도·재현율 측정                 | 평가 단계에서만 라벨 사용 → 학습-평가 분리 보장                       |

### 9.2 흐름 해설 (왜 이렇게 갈라지는가)

1. **원본을 받자마자 두 갈래로 쪼갠다**
   - 본문 트랙(왼쪽)에는 라벨이 한 글자도 들어가지 못한다
   - 라벨 트랙(오른쪽)은 Sidecar 파일로 격리되어 별도 관리

2. **왼쪽 본문 트랙에서 두 개의 탐지가 평행하게 동작**
   - Phase 1: 라벨 없이도 동작하는 규칙 기반 탐지
   - Phase 2: 비지도 모델은 라벨 없이 학습, 지도 모델은 Gate를 거친 라벨만 받아 학습

3. **Supervised Gate가 라벨 누수의 마지막 방어선**
   - "허용된 출처(label_source)의 라벨"만 모델로 들어갈 수 있다
   - 평가용 GT 라벨이 학습에 섞여 들어가는 사고를 차단

4. **Evaluation 단계에서 두 트랙이 만난다**
   - 모델 결과(예측)와 Sidecar 라벨(정답)을 비교해 성능 측정
   - 이 시점에 라벨이 처음으로 "정답지"로 사용됨

> 핵심은 **라벨이 모델로 들어가는 경로를 단 하나(Gate)로 좁히는 것**이다.
> 본문과 라벨을 처음부터 물리적으로 분리하고, 학습 입력 직전에 검문소를 두면 합성 데이터로도 누수 없는 평가가 가능하다.
