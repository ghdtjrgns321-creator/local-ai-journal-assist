# 개발 방법론 — 정상 데이터 기반 검증 중심 탐지 개발

> 사용자/포트폴리오용 기록물. 이 프로젝트가 부정 탐지기를 만드는 "순서와 원칙"을 설명한다.
> 상태: living 문서(진행 중). 탐지 성능 주장 범위는 프로젝트 공통 caveat(전수 review queue 보조 도구,
> DataSynth 기반 지표는 개발 검증용)를 따른다.

## 한 줄

부정 탐지기를 "부정부터" 만들지 않는다. **정상 데이터를 먼저 진짜 회사 장부 수준으로 세우고**, 그 위에서
룰과 부정을 검증하며, **PHASE1 룰로 못 잡거나 우선순위에서 밀리는 것만 PHASE2로** 넘긴다. 데이터-우선 ·
검증중심 · anti-fitting 방법론이다.

## 왜 이 순서인가

부정은 항상 정상 위에 얹힌다. 정상 데이터가 회계적으로 깨져 있거나(차대변 불일치, 재무제표 불일치),
의미가 어긋나 있으면(거래처×계정×적요 부정합), 탐지기와 ML은 부정의 구조가 아니라 **합성 데이터의
아티팩트**를 학습한다. 그래서 정상을 먼저 완벽히 만든다. 정상이 단단해야 그 위의 부정이 깨끗이 드러난다.

## 파이프라인 (6단계)

```text
단계  내용                                          확인 기준
----  --------------------------------------------  --------------------------------------------
1     NORMAL 데이터 생성                            회계 정합(차대변·재무제표)·구조(내부거래·순환)·
                                                    자연 노이즈까지 실제 회사 장부처럼
2     PHASE1 룰로 검증                              정상 데이터가 룰을 잘못(높은 비율로) 울리지 않나
3     NORMAL 데이터 정합성 확보                     2를 통과할 때까지 반복. 통과는 "비-자명한 수치로
                                                    실제 측정"이어야 함(빈 통과 금지)
4     PHASE1용 룰 위반 데이터 생성                  룰마다 위반 케이스를 심어 PHASE1이 제대로 잡는지 검증
5     PHASE1 데이터·룰 정합성 + PHASE2 타깃 확정    실제 감리 사례 기반 부정을 정상에 심고 PHASE1을
                                                    돌려서, 못 잡거나 등수 밀린 것 = PHASE2 타깃으로 동결
6     PHASE2 구현                                   동결된 타깃 위에서 ML/DL로 끌어올린다
```

핵심은 5단계다. PHASE2의 타깃을 머릿속이나 family 구조에 맞춰 정하지 않고, **실제 부정을 정상에 심고
PHASE1을 돌렸을 때 "빠지거나 밀리는 것"을 관찰해서 정한다.** 이때 데이터·탐지기를 그 결과에 맞춰 고치지
않는다(그건 fitting). 한 번 관찰해 타깃을 동결하고, PHASE2는 새로 생성한 데이터로 검증한다.

## 관통하는 원칙

### 단위 (무엇을 정답으로 세나)
- 정답·측정은 자연 단위에서만: **전표(document) 또는 흐름(flow)**.
- 줄(row)은 단위가 아니라 "위반이 어느 줄인지" 가리키는 증거 포인터.
- 화면의 case(사용자×월 등)는 표시용 집계 뷰이며 자체 정답·점수가 없다.
- 부정마다 단위를 하나만 배정해 중복 카운트를 구조적으로 막는다.

### 검증 (완료를 어떻게 판정하나)
- **PASS는 증거가 아니다.** 검사가 빈 집합·fallback·누락 필드를 통과로 둔갑시킨 게 아닌지 확인한다.
- hollow-pass를 찾으면 인스턴스만 고치지 말고 **검사 자체를 강화**해 재발을 막는다.
- **회귀 가드**: 알려진 실패 수를 baseline으로 두고, 새 실패는 전부 원인을 규명한다.

### 근본 수정 / anti-fitting
- 증상 패치가 아니라 뿌리(생성기·정의)를 고친다.
- truth 라벨이나 "숫자가 좋아 보이게"에 맞춰 튜닝하지 않는다. 임계값은 회계·ERP·감사 도메인에서 가져온다.

### 위반 주입 (4단계 overlay) 원칙
정상 위에 룰 위반을 심을 때, **detector가 실제로 보는 조건**을 만들어야 한다. "위반이라는 표시"만
박으면 안 잡힌다. P3-2 전수 규명(2026-06-07, `dev/active/p3-2-overlay-injection-audit.md`)에서 확인된
3대 실패 메커니즘:

1. **flag override 금지** — `is_period_end`·`sod_violation`·`exceeds_threshold` 같은 파생 플래그를
   직접 박아도, feature 파이프라인이 raw 필드(posting_date, created_by/approved_by, 금액 vs 한도)에서
   그 값을 **재계산해 덮어쓴다.** 주입은 raw 조건 자체를 만들어 feature 재계산 후에도 살아남게 해야 한다.
   (예: 결산기 전표는 플래그가 아니라 posting_date를 결산 ±margin 안으로 둔다.)
2. **임계를 실제로 넘긴다** — 위반의 "정도"가 detector 임계 미달이면 못 잡는다. 주말 위반은 진짜 주말
   날짜, 한도 초과는 의미 있는 초과폭, 금액·날짜 불일치는 tolerance 초과로. 정상 라벨
   (`normal_accrual_reversal`, `NORMAL-BATCH`) 재사용은 위반이 아니다.
3. **단위를 맞춘다** — 집계·모집단·생애주기 룰(Benford, 변동, 분포, 시간 클러스터, 사용자 집중,
   suspense lifecycle)은 2줄짜리 전표 1건으로 만들 수 없다. 분포/모집단/prior 베이스라인을 함께
   흔들어야 한다. 이런 룰은 PHASE2(집계·ML)의 자연 타깃이기도 하다.

검증은 **주입했다는 사실이 아니라 주입한 행이 detector가 보는 조건을 담았는지**를 전수로 확인한다.
"잡았다"도 발화율 높은 룰에서는 우연일 수 있으니, 의도한 트리거로 잡혔는지까지 본다.

## PHASE1 / PHASE2 역할 분담

- **PHASE1**: 룰 기반 전수 필터. 감사인이 먼저 볼 우선순위(triage)를 만든다. "룰 많이 어길수록 부정"이
  아니라, 독립적인 여러 각도가 같은 전표/흐름을 가리킬수록 먼저 보게 한다. 부정 확정이 목적이 아니다.
- **PHASE2**: PHASE1이 룰로 못 잡거나 등수에서 밀린 부정을 ML/DL로 끌어올린다. PHASE1을 덮어쓰지 않고
  나란히 간다.

## 현재 위치 (living)

- 1~3단계(정상 데이터): 완료 — 회계 정합, 재무제표(시산표·회계등식·roll-forward·보조원장), 내부거래·
  순환 구조, semantic·참조 고유성, 자연 노이즈까지 갖춘 정상 baseline 확보.
- PHASE1 단위 통일(전표/흐름/집계뷰): 진행 중.
- 4~6단계(룰 위반 데이터 → PHASE2 타깃 확정 → PHASE2 구현): 이후.
