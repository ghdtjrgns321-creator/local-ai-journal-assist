# 06. DuckDB 데이터베이스 레이어 [Phase 1b — 의존: 05-detection]

## 목적

이상탐지 결과를 DuckDB에 적재하여 OLAP 쿼리를 지원한다.
대시보드(07-dashboard)와 Text-to-SQL(08-llm)의 데이터 소스 역할.

---

## 데이터 흐름

```
[pipeline.py가 수집한 detection 출력]      (from detection/)
  ├── DataFrame: score_aggregator.aggregate_scores() 반환값
  │     (anomaly_score, risk_level, flagged_rules 컬럼 포함)
  ├── list[DetectionResult]: integrity/fraud/anomaly layer.detect() 직접 반환
  └── BenfordResult: AnomalyLayer DetectionResult.metadata["benford"]에 포함
       ↓
① get_connection(settings.duckdb_path)     → DuckDB 커넥션 획득
       ↓
② initialize_schema(conn)                  → 4개 테이블 + 1 VIEW DDL 실행
       ↓
③ loader.load_general_ledger(conn, df, batch_id)
   loader.load_anomaly_flags(conn, results, df, batch_id)
   loader.load_benford(conn, results, batch_id)
       ↓
[DuckDB 4개 테이블 + 1 VIEW]
       ↓
④ queries.execute_preset(conn, "batch_ledger", batch_id=batch_id)
       ↓
[Raw DataFrame] → dashboard/ (pandas 필터 + groupby로 KPI·차트 집계)
```

## 구현 상태 & 모듈별 가이드

### ① 커넥션 관리 (connection.py) — ✅ 구현 완료

```
src/db/
└── connection.py    # 싱글톤 DuckDB 커넥션
```

#### 이 모듈이 하는 일

DB 레이어의 **진입점**으로, DuckDB 커넥션을 싱글톤으로 관리한다.
최초 호출 시 파일 기반 DB에 연결하고 스키마를 초기화한다.

```
파이프라인 위치:
  pipeline.py  →  connection.get_connection()  →  loader / queries
                                                       ↓
                                        settings.duckdb_path ("data/audit.duckdb")

역할:
  ① 싱글톤 커넥션 반환 — 여러 모듈이 동일 커넥션 공유
  ② 최초 연결 시 schema.initialize_schema() 자동 호출
  ③ 명시적 종료 — Streamlit 세션 종료 시 리소스 해제
  ④ threading.Lock — 프로세스 내 스레드 간 이중 생성 방지
  ⑤ health check — closed 커넥션 감지 시 자동 재생성
  ⑥ _override_connection — 테스트에서 :memory: 커넥션 주입
```

#### 구현 내용

```python
_connection: duckdb.DuckDBPyConnection | None = None
_lock = threading.Lock()

def get_connection(path: str | None = None) -> duckdb.DuckDBPyConnection:
    """싱글톤 DuckDB 커넥션 반환.

    1. Lock 획득
    2. 기존 커넥션 있으면 health check (SELECT 1) → 유효하면 반환
    3. 없거나 무효 → 새 커넥션 생성 + 스키마 초기화
    4. IOException → 프로세스 간 잠금 안내 메시지 로깅
    """

def close_connection() -> None:
    """커넥션 명시적 종료. Lock 보호 하에 close + None 리셋."""

def _override_connection(conn: duckdb.DuckDBPyConnection) -> None:
    """테스트 전용 — 전역 커넥션을 외부 주입으로 교체."""

def _is_alive(conn: duckdb.DuckDBPyConnection) -> bool:
    """SELECT 1 실행으로 커넥션 유효성 확인."""
```

#### 잠금 범위 (Gotcha)

```
┌─ threading.Lock 보호 범위 ──────────────────────────┐
│  Streamlit 프로세스 1개 안에서                        │
│  여러 스레드(사용자 요청)가 동시에 get_connection()   │
│  호출 시 커넥션 이중 생성 방지                        │
└──────────────────────────────────────────────────────┘

┌─ DuckDB 파일 잠금 (threading.Lock과 무관) ──────────┐
│  동일 .duckdb 파일에 2개 프로세스가 접근하면          │
│  두 번째 프로세스는 무조건 IOException 발생            │
│  → except 블록에서 프로세스 간 충돌 안내 메시지 출력  │
└──────────────────────────────────────────────────────┘
```

#### 구현 시 주의사항

**경로 자동 생성:**
`data/` 디렉토리가 없으면 `Path(path).parent.mkdir(parents=True, exist_ok=True)`로 생성.
DB 파일이 없으면 DuckDB가 자동 생성하지만, 디렉토리는 수동 생성 필요.

**settings 참조:**
`config/settings.py` 라인 93의 `duckdb_path: str = "data/audit.duckdb"` 사용.
path 파라미터가 None이면 `get_settings().duckdb_path`로 기본값 획득.

**순환 import 방지:**
`schema.initialize_schema`는 `get_connection()` 함수 내부에서 지연 import.

#### 설계 결정

| 이슈                      | 결정                                              | 사유                                                  |
|:--------------------------|:--------------------------------------------------|:------------------------------------------------------|
| 커넥션 관리 방식          | 모듈 레벨 전역 변수 싱글톤                         | 경량 구현. Phase 1b 단일 사용자 전제                   |
| path 기본값               | `settings.duckdb_path` fallback                   | 테스트 시 in-memory(`:memory:`) 주입 가능              |
| 스키마 초기화 시점        | 최초 `get_connection()` 호출 시 자동               | 명시적 초기화 호출 누락 방지                            |
| close 후 재연결           | `get_connection()` 재호출 시 새 커넥션 생성        | 테스트 격리 용이                                       |
| threading.Lock            | 프로세스 내 스레드 간 보호                         | Streamlit 멀티스레드 커넥션 이중 생성 방지              |
| health check              | SELECT 1로 유효성 확인                             | closed 커넥션 자동 감지 → 재생성                       |
| 테스트 주입               | `_override_connection()` 함수                     | loader·queries가 내부 get_connection() 호출 시 격리     |
| context manager           | 미적용                                             | 싱글톤과 양립 불가 (블록 종료 시 다른 모듈 커넥션 끊김) |
| read_only 커넥션          | 미적용                                             | 파일 잠금 복잡성 증가. Phase 1c에서 필요 시 검토        |

- 싱글톤 동작: 두 번 호출 시 동일 객체 반환
- close 후 재호출 시 새 커넥션 생성 확인
- health check 실패 시 자동 재생성 확인
- `_override_connection`으로 주입 커넥션 반환 확인

### ② DDL 정의 (schema.py) — ✅ 구현 완료

```
src/db/
└── schema.py    # SCHEMA_DDL dict + initialize_schema()
```

#### 이 모듈이 하는 일

DuckDB에 생성할 **4개 테이블 + 1 VIEW의 DDL**을 정의하고, 일괄 실행하는 초기화 함수를 제공한다.

```
테이블 구조:
  general_ledger          원본 전표 39개 + approval_level 파생 + 피처 18종 + 탐지 결과 3종
  anomaly_flags           룰별 플래그 상세 (행×룰 단위, score > 0만)
  benford_summary         Benford 배치 요약 (배치당 1행)
  benford_digits          Benford 자릿수별 분포 (배치당 9행)
  anomaly_flag_summary    VIEW — anomaly_flags 룰별 집계
```

detection 파이프라인 출력물이 이 4개 테이블로 분산 저장된다:

```
┌──────────────────────┬──────────────────────────┬─────────────────────────┐
│ detection 출력       │ 저장 테이블              │ 매핑 방식               │
├──────────────────────┼──────────────────────────┼─────────────────────────┤
│ DataFrame            │ general_ledger           │ 컬럼 1:1 대응           │
│  (anomaly_score,     │  (anomaly_score,         │                         │
│   risk_level,        │   risk_level,            │                         │
│   flagged_rules)     │   flagged_rules)         │                         │
├──────────────────────┼──────────────────────────┼─────────────────────────┤
│ DetectionResult      │ anomaly_flags            │ details melt → 행 전개  │
│  .details DataFrame  │  (rule_code, score, ...) │ score > 0 필터          │
├──────────────────────┼──────────────────────────┼─────────────────────────┤
│ BenfordResult        │ benford_summary          │ 배치당 1행 (통계 요약)  │
│  (metadata 내)       │ benford_digits           │ 자릿수별 9행 (분포)     │
└──────────────────────┴──────────────────────────┴─────────────────────────┘
```

#### score_aggregator 출력 → general_ledger 컬럼 매핑

score_aggregator.aggregate_scores()가 반환하는 DataFrame 3개 컬럼이
general_ledger 테이블의 이상탐지 결과 영역에 1:1 대응한다:

```
score_aggregator 출력         general_ledger 컬럼     타입      비고
─────────────────────────────────────────────────────────────────────
anomaly_score (0.0~1.0)   →  anomaly_score DOUBLE    가중합 점수
risk_level (High/Med/...)  →  risk_level VARCHAR      4등급 분류
flagged_rules ("A01,B03") →  flagged_rules VARCHAR   comma-separated
```

#### approval_level 파생 컬럼

DataSynth CSV에는 `approval_level` 컬럼이 없다.
전결규정 6단계 한도(`generation_principles.md` §11)가 명확하므로,
`loader.py`에서 적재 전 전표 단위 금액 기반으로 파생한다.

```
파생 로직 (전표 단위):
  amount = document_id별 MAX(debit_amount, credit_amount)의 합산

  CASE
    WHEN amount <=     10,000,000 THEN 1  -- 자동 승인
    WHEN amount <=    100,000,000 THEN 2  -- 담당자
    WHEN amount <=  1,000,000,000 THEN 3  -- 팀장
    WHEN amount <=  5,000,000,000 THEN 4  -- 본부장
    WHEN amount <= 10,000,000,000 THEN 5  -- CFO
    ELSE                               6  -- 이사회
  END AS approval_level

구현 위치:
  loader.py → load_general_ledger() 적재 전 pandas groupby + map
  DuckDB GENERATED ALWAYS AS는 미지원 → pandas 파생이 현실적

탐지 활용:
  B06(승인 한도 초과) — approval_level vs user_persona 비교로 권한 위반 탐지
```

#### 구현 시 주의사항

**파생변수 컬럼 정합성:**
general_ledger DDL의 파생변수 18종은 `src/feature/engine.py` `EXPECTED_COLUMNS`와 1:1 대응.
feature 모듈에서 컬럼이 추가/변경되면 DDL도 함께 수정 필요.
Phase 2에서 ML 피처가 추가되면 `ALTER TABLE` 마이그레이션 고려.

**DuckDB 타입 매핑:**
pandas dtype → DuckDB 타입 자동 매핑에 주의.
`bool → BOOLEAN`, `int64 → BIGINT`, `float64 → DOUBLE`은 자동이지만,
`object → VARCHAR`는 혼합 타입 컬럼에서 예상치 못한 변환 발생 가능.

**CREATE IF NOT EXISTS:**
멱등성 보장. 동일 스키마로 여러 번 호출해도 기존 데이터 유지.
단, 컬럼 추가/변경은 별도 마이그레이션 필요.

#### 설계 결정

| 이슈                          | 결정                                           | 사유                                                      |
|:------------------------------|:-----------------------------------------------|:----------------------------------------------------------|
| 테이블 수                     | 4개 + 1 VIEW                                    | 정규화: 원본+점수 / 룰별 상세 / Benford 요약·분포 분리      |
| general_ledger에 탐지 결과 포함 | anomaly_score, risk_level, flagged_rules 3컬럼 | 대시보드에서 JOIN 없이 단일 테이블 조회 가능                |
| anomaly_flags 별도 테이블     | details melt 기반, score > 0만 저장             | 룰별 통계는 anomaly_flag_summary VIEW로 제공                |

### ③ 데이터 적재 (loader.py) — ✅ 구현 완료

```
src/db/
└── loader.py    # LoadResult + load_all() + load_general_ledger() + load_anomaly_flags() + load_benford()
```

#### 이 모듈이 하는 일

detection 파이프라인 출력물 3종을 DuckDB 4개 테이블에 적재하는 함수를 제공한다.
DuckDB의 **pandas 직결합** 기능을 활용하여 DataFrame을 SQL 테이블처럼 직접 참조한다.
`load_all()`이 4개 테이블을 **트랜잭션으로 원자적 적재**하는 진입점.

```
파이프라인 위치:
  score_aggregator 출력  →  loader.load_all()  →  DuckDB 4개 테이블

적재 대상:
  ① DataFrame (원본 + 파생변수 + 탐지 결과)  →  general_ledger
  ② list[DetectionResult] (룰별 플래그)       →  anomaly_flags
  ③ BenfordResult (Benford 분석)              →  benford_summary + benford_digits
```

#### 결과 dataclass

```python
@dataclass
class LoadResult:
    """4개 테이블 적재 결과 통합."""
    batch_id: str
    general_ledger_rows: int
    anomaly_flags_rows: int
    benford_summary_rows: int
    benford_digits_rows: int
    elapsed_seconds: float
    warnings: list[str] = field(default_factory=list)

    @property
    def total_rows(self) -> int:
        return (self.general_ledger_rows + self.anomaly_flags_rows
                + self.benford_summary_rows + self.benford_digits_rows)

    @property
    def is_success(self) -> bool:
        return self.general_ledger_rows > 0
```

#### 구현 내용

```python
def load_all(
    conn: duckdb.DuckDBPyConnection,
    df: pd.DataFrame,
    results: list[DetectionResult],
    batch_id: str,
) -> LoadResult:
    """4개 테이블 원자적 적재 (트랜잭션).

    Why: general_ledger만 적재되고 anomaly_flags/benford 실패 시 불일치 방지.
    """
    start = time.monotonic()
    warnings: list[str] = []
    conn.execute("BEGIN TRANSACTION")
    try:
        gl_rows = load_general_ledger(conn, df, batch_id)
        af_rows = load_anomaly_flags(conn, results, df, batch_id)
        bs_rows, bd_rows, bf_warnings = load_benford(conn, results, batch_id)
        warnings.extend(bf_warnings)
        conn.execute("COMMIT")
    except Exception:
        conn.execute("ROLLBACK")
        raise
    elapsed = time.monotonic() - start
    return LoadResult(
        batch_id=batch_id,
        general_ledger_rows=gl_rows,
        anomaly_flags_rows=af_rows,
        benford_summary_rows=bs_rows,
        benford_digits_rows=bd_rows,
        elapsed_seconds=elapsed,
        warnings=warnings,
    )
```

#### DetectionResult.details → anomaly_flags 변환

`details` DataFrame(columns=룰ID, values=float 0.0~1.0)을 melt하여 행 단위로 전개한다.
RuleFlag는 **룰별 요약** 전용이며 개별 행 인덱스를 보유하지 않으므로 변환에 사용하지 않는다.
details의 값은 이미 `severity/5 × flagged(0|1)`로 정규화되어 있다 (`anomaly_layer.py:111`).

```
변환 흐름:
  details (index=행, columns=룰ID, values=score)
    → melt → (row_idx, rule_code, score) where score > 0
    → df.loc[row_idx, "document_id"] 조인
    → anomaly_flags 테이블 행
```

```
입력 예시 (DetectionResult):                    출력 (anomaly_flags 행):
─────────────────────────────                   ─────────────────────────────
.track_name: "layer_b"                          track_name: "layer_b"
.details:                                       document_id | rule_code | score
  index | B01  | B03  | B05                     ────────────┼───────────┼──────
  5     | 0.0  | 0.8  | 0.0                     JE-005      | B03       | 0.8
  12    | 0.6  | 0.8  | 0.4                     JE-012      | B01       | 0.6
                                                 JE-012      | B03       | 0.8
                                                 JE-012      | B05       | 0.4
```

벡터화 구현 (`_build_anomaly_flags_df`):

```python
def _build_anomaly_flags_df(
    results: list[DetectionResult],
    df: pd.DataFrame,
    batch_id: str,
) -> pd.DataFrame:
    """DetectionResult.details → anomaly_flags 테이블 DataFrame 변환.

    Why: melt 벡터화로 100만행 × 24룰도 수초 내 처리. apply(axis=1) 회피.
    """
    empty = pd.DataFrame(columns=ANOMALY_FLAGS_COLUMNS)
    if not results:
        return empty

    chunks: list[pd.DataFrame] = []
    for result in results:
        if result.details.empty:
            continue
        melted = result.details.melt(
            ignore_index=False, var_name="rule_code", value_name="score",
        )
        melted = melted[melted["score"] > 0].copy()
        if melted.empty:
            continue
        # Why: .values 사용 시 인덱스 정렬 우회 → 불일치 위험. 인덱스 기반 할당 사용.
        melted["document_id"] = df.loc[melted.index, "document_id"]
        melted["line_number"] = (
            df.loc[melted.index, "line_number"]
            if "line_number" in df.columns
            else None
        )
        melted["track_name"] = result.track_name
        melted["upload_batch_id"] = batch_id
        chunks.append(melted)

    if not chunks:
        return empty
    combined = pd.concat(chunks, ignore_index=True)
    return combined.reindex(columns=ANOMALY_FLAGS_COLUMNS)
```

`load_benford()`는 `_extract_benford()` 헬퍼를 통해 BenfordResult를 추출한다:

```python
def _extract_benford(results: list[DetectionResult]) -> BenfordResult | None:
    """results에서 layer_c의 BenfordResult를 추출한다.

    추출 경로: track_name="layer_c" → metadata["benford_result"]
    """
    for r in results:
        if r.track_name == "layer_c" and "benford_result" in r.metadata:
            return r.metadata["benford_result"]
    return None
```

```python
def _build_benford_summary_df(
    br: BenfordResult,
    batch_id: str,
) -> pd.DataFrame:
    """BenfordResult → benford_summary 테이블 DataFrame 변환 (배치당 1행).

    Source of truth: src/validation/models.py:BenfordResult (L70-88)
    DDL 컬럼명과 BenfordResult 필드명이 1:1 대응 (v1의 언더바 불일치 해소).
    """
    return pd.DataFrame([{
        "upload_batch_id": batch_id,
        "sample_size": br.sample_size,
        "mad": br.mad,
        "mad_conformity": br.mad_conformity,
        "chi2_statistic": br.chi2_statistic,
        "chi2_p_value": br.chi2_p_value,
        "ks_statistic": br.ks_statistic,
        "ks_p_value": br.ks_p_value,
        "is_conforming": br.is_conforming,
        "confidence": br.confidence,
    }]).reindex(columns=BENFORD_SUMMARY_COLUMNS)

def _build_benford_digits_df(
    br: BenfordResult,
    batch_id: str,
) -> pd.DataFrame:
    """BenfordResult → benford_digits 테이블 DataFrame 변환 (자릿수별 9행)."""
    rows = []
    for digit in range(1, 10):
        obs = br.observed.get(digit, 0.0)
        exp = br.expected.get(digit, 0.0)
        rows.append({
            "upload_batch_id": batch_id,
            "digit": digit,
            "observed_freq": obs,
            "expected_freq": exp,
            "deviation": obs - exp,
        })
    return pd.DataFrame(rows).reindex(columns=BENFORD_DIGITS_COLUMNS)
```

#### 구현 시 주의사항

**🚨 DuckDB pandas 직결합 변수 스코프 (Gotcha):**
DuckDB는 `conn.execute("INSERT INTO t SELECT * FROM df")`에서 SQL 내 `df`를
**파이썬 로컬 변수명으로 직접 Introspection**한다.
헬퍼 함수가 반환한 DataFrame을 `melted_df`에 담아두고
SQL에서 `SELECT * FROM df`라고 쓰면 → `Catalog Error: df not found`.
**규칙: SQL 쿼리 문자열 안의 변수명 = 파이썬 로컬 변수명 100% 일치 필수.**

```python
# 올바른 예시 — load_general_ledger (가드절/copy() 생략 — 변수 스코프 예시에 집중)
def load_general_ledger(conn, df, batch_id):
    df["upload_batch_id"] = batch_id
    # Why: approval_level은 DataSynth에 없는 파생 컬럼.
    #      전표 단위 최대 금액으로 전결규정 6단계 승인 레벨 산출.
    df["approval_level"] = _derive_approval_level(df)
    gl_df = df.reindex(columns=GENERAL_LEDGER_COLUMNS)
    # Why: created_at은 DEFAULT → INSERT 대상에서 제외. 명시적 컬럼 지정 필수.
    col_list = ", ".join(GENERAL_LEDGER_COLUMNS)
    # SQL 내 'gl_df' == 파이썬 로컬 변수 'gl_df' — 일치
    conn.execute(f"INSERT INTO general_ledger ({col_list}) SELECT * FROM gl_df")
    return len(gl_df)

# approval_level 파생 헬퍼 (generation_principles.md §11 기준)
# Why: 전결규정 한도는 settings.yaml이 아닌 generation_principles에 정의.
#      향후 설정 외부화 가능하나 MVP에서는 상수로 충분.
_APPROVAL_THRESHOLDS = [
    (10_000_000, 1),       # 자동 승인
    (100_000_000, 2),      # 담당자
    (1_000_000_000, 3),    # 팀장
    (5_000_000_000, 4),    # 본부장
    (10_000_000_000, 5),   # CFO
]

def _derive_approval_level(df: pd.DataFrame) -> pd.Series:
    """전표 단위 최대 금액 → 전결규정 승인 레벨 (1~6)."""
    amount = df[["debit_amount", "credit_amount"]].max(axis=1)
    doc_max = amount.groupby(df["document_id"]).transform("max")
    level = pd.Series(6, index=df.index)  # 기본값: 이사회
    for threshold, lv in reversed(_APPROVAL_THRESHOLDS):
        level = level.where(doc_max > threshold, lv)
    return level

# 올바른 예시 — load_anomaly_flags
def load_anomaly_flags(conn, results, df, batch_id):
    flags_df = _build_anomaly_flags_df(results, df, batch_id)
    if flags_df.empty:
        return 0
    af_cols = ", ".join(ANOMALY_FLAGS_COLUMNS)
    # SQL 내 'flags_df' == 파이썬 로컬 변수 'flags_df' — 일치
    conn.execute(f"INSERT INTO anomaly_flags ({af_cols}) SELECT * FROM flags_df")
    return len(flags_df)
```

**batch_id 주입:**
DataFrame에 `upload_batch_id` 컬럼이 없으면 적재 전 `df["upload_batch_id"] = batch_id`로 추가.
여러 번 업로드 시 batch_id로 데이터 구분.

**컬럼 순서 정합성 (reindex 직접 사용):**
`schema.py`에 `GENERAL_LEDGER_COLUMNS: list[str]` 등 4개 컬럼 상수를 DDL과 동기화하여 정의.
적재 전 `df.reindex(columns=GENERAL_LEDGER_COLUMNS)`로 순서 보장 + 누락 컬럼 NaN 패딩.
DDL에 없는 초과 컬럼은 자동 드롭, DDL에만 있는 누락 컬럼은 NaN 채움.
`_build_anomaly_flags_df`, `_build_benford_summary_df`, `_build_benford_digits_df` 모두 동일 패턴 적용.

**risk_level StrEnum → VARCHAR 변환:**
`score_aggregator`의 `classify_risk_level()`은 `RiskLevel` StrEnum을 반환.
DuckDB pandas 직결합 시 StrEnum → VARCHAR 자동 변환이 보장되지 않으므로,
적재 전 `df["risk_level"] = df["risk_level"].astype(str)` 명시적 변환 필수.

**대용량 DataFrame 적재:**
500MB+ DataFrame은 Parquet 경유 적재로 피크 메모리 절감 가능.
하단 "대용량 파일 직접 적재" 섹션 참조.

#### 아키텍처 결정: SQL 집계 vs pandas 후처리

v1 설계는 SQL 단에서 GROUP BY/SUM으로 사전 집계된 요약본을 반환했다.
그러나 대시보드 `filters.py`에서 날짜/risk_level/계정 필터를 걸면,
이미 집계된 요약본에서는 **필터 적용 후 재계산이 불가능**하다.

```
예시: SQL이 전체 배치를 1행(총 전표수/이상비율)으로 집계하여 반환
  → 사용자가 "2월 데이터만 보기" 필터 적용
  → 1년 치로 뭉뚱그린 1행에서 2월만 분리 불가
```

MVP에서는 **Raw Data를 DB에서 퍼온 뒤 대시보드에서 pandas로 직접 집계**한다.

```
쿼리 유형       역할                       필터 대응
────────────── ────────────────────────── ──────────────────────
Raw 데이터     DB → 행 단위 DataFrame     pandas 필터 + groupby
               대시보드에서 집계·시각화    → 완전한 필터 유연성

Benford 통계   배치 단위 고정 분석        필터 불필요 (배치 전체)
               (최대 9행, 변경 불가)

드릴다운       사용자 클릭 시 on-demand   document_id 지정
```

v1 사전 집계 쿼리(`high_risk_summary`, `monthly_anomaly_trend`, `top_anomalies`,
`rule_violation_stats`)는 파기하고, Raw 데이터 쿼리로 대체한다.

#### 구현 내용

```python

    # Why: 대시보드 진입 시 1회 호출 → session_state 캐싱
    #      pandas 필터/groupby로 KPI·차트·히트맵 직접 계산
    "batch_ledger": """
        SELECT document_id, company_code, fiscal_year,
               posting_date, document_date, document_type,
               line_number, gl_account, debit_amount, credit_amount,
               line_text, header_text, created_by, source,
               business_process, user_persona, approved_by,
               approval_level, reference,
               is_fraud, fraud_type, is_anomaly, anomaly_type,
               sod_violation, sod_conflict_type,
               anomaly_score, risk_level, flagged_rules
        FROM general_ledger
        WHERE upload_batch_id = ?
        ORDER BY anomaly_score DESC
    """,

    # Why: 룰별 위반 통계를 대시보드에서 pandas groupby로 집계
    #      Explorer 탭 드릴다운 시에도 document_id 필터링 소스
    "batch_flags": """
        SELECT document_id, line_number, track_name, rule_code, score
        FROM anomaly_flags
        WHERE upload_batch_id = ?
        ORDER BY document_id, score DESC
    """,

    # Why: 배치 전체 대상 고정 분석 (요약 통계)
    "benford_summary": """
        SELECT sample_size, mad, mad_conformity,
               chi2_statistic, chi2_p_value,
               ks_statistic, ks_p_value,
               is_conforming, confidence
        FROM benford_summary
        WHERE upload_batch_id = ?
    """,

    # Why: 배치 전체 대상 고정 분석 (자릿수별 분포)
    "benford_digits": """
        SELECT digit, observed_freq, expected_freq, deviation
        FROM benford_digits
        WHERE upload_batch_id = ?
        ORDER BY digit
    """,

    # Why: 룰별 위반 통계를 VIEW로 간결하게 조회
    "rule_violation_stats": """
        SELECT track_name, rule_code, flagged_count, avg_score, max_score
        FROM anomaly_flag_summary
        WHERE upload_batch_id = ?
        ORDER BY flagged_count DESC
    """,

    # Why: Explorer 탭에서 행 클릭 시 on-demand 호출
    #      파라미터 2개 (batch_id, document_id)
    "document_rule_detail": """
        SELECT track_name, rule_code, score
        FROM anomaly_flags
        WHERE upload_batch_id = ? AND document_id = ?
        ORDER BY score DESC
    """,
}


class QueryNotFoundError(KeyError):
    """존재하지 않는 프리셋 쿼리명."""


class QueryExecutionError(RuntimeError):
    """SQL 실행 중 오류."""


def execute_preset(
    conn: duckdb.DuckDBPyConnection,
    query_name: str,
    params: tuple | None = None,
    *,
    batch_id: str | None = None,
) -> pd.DataFrame:
    """프리셋 쿼리 실행 후 DataFrame 반환.

    Args:
        query_name: PRESET_QUERIES 키.
        params: SQL 파라미터 바인딩 튜플. None이면 (batch_id,) 자동 구성.
        batch_id: params가 None일 때 사용하는 편의 파라미터.

    Raises:
        QueryNotFoundError: query_name이 PRESET_QUERIES에 없을 때.
        QueryExecutionError: SQL 실행 중 오류 발생 시.

    Returns:
        결과 DataFrame. 0행이면 빈 DataFrame (컬럼 스키마 유지).
    """
```

#### 구현 시 주의사항

**파라미터 바인딩:**
SQL injection 방지를 위해 `?` 플레이스홀더 + 파라미터 바인딩 사용.
`f-string`이나 `.format()`으로 값을 직접 삽입하지 않는다.

**다중 파라미터 지원:**
`execute_preset`은 `params` 튜플과 `batch_id` 편의 파라미터를 모두 지원한다.
`params`와 `batch_id` 동시 전달 시 `params` 우선.
`document_rule_detail`처럼 파라미터가 2개인 쿼리는
`params=(batch_id, document_id)` 형태로 호출.

**에러 처리:**
`QueryNotFoundError`(쿼리명 미존재)와 `QueryExecutionError`(SQL 실행 오류)를 분리.
`duckdb.Error` → `QueryExecutionError`로 변환 시 원본 예외를 `__cause__`로 체이닝.

**빈 결과:**
쿼리 결과가 0행이면 빈 DataFrame 반환 (에러 아님).
대시보드에서 "데이터 없음" 표시 처리.

## 대용량 파일 직접 적재 (Phase 1b 확장 고려)

> Phase 1a 분석 결과: ingest 파이프라인이 전부 pandas 기반이라 DuckDB를 리더로 쓰면
> `.df()` 변환이 필수 → 메모리 이점 소실. 따라서 **ingest 완료 후 적재 시점**에서 DuckDB를 활용.

### 시나리오: 대용량 CSV → DuckDB 직접 적재

ingest 파이프라인(validate → read → header → map → cast) 완료 후,
표준 DataFrame이 너무 크면 **중간 Parquet 저장 → DuckDB에서 Parquet 읽기**로 메모리 절감 가능.

```python
# loader.py 확장안 (Phase 1b 구현 시 구체화)
# Why: table_name을 f-string에 직접 삽입하므로 허용 목록으로 SQL injection 방어
ALLOWED_TABLES = frozenset(SCHEMA_DDL.keys())

def load_large_file(
    conn: duckdb.DuckDBPyConnection,
    df: DataFrame,
    table_name: str,
    batch_id: str,
    threshold_mb: int = 500,
) -> int:
    """대용량 DataFrame은 Parquet 경유, 소용량은 직접 적재.

    Why: 500MB+ DataFrame을 DuckDB에 직접 INSERT하면 피크 메모리가 2배.
    Parquet 중간 저장 → DuckDB COPY로 읽으면 피크를 DataFrame 크기로 제한.
    """
    if table_name not in ALLOWED_TABLES:
        raise ValueError(f"허용되지 않은 테이블: {table_name}")

    estimated_mb = df.memory_usage(deep=True).sum() / (1024 * 1024)

    if estimated_mb > threshold_mb:
        # 대용량: Parquet 경유
        tmp_path = Path("data/tmp") / f"{batch_id}_{table_name}.parquet"
        tmp_path.parent.mkdir(parents=True, exist_ok=True)
        df.to_parquet(tmp_path)
        del df  # 메모리 해제
        conn.execute(f"INSERT INTO {table_name} SELECT * FROM read_parquet('{tmp_path}')")
        tmp_path.unlink()  # 임시 파일 삭제
    else:
        # 소용량: pandas 직결합
        conn.execute(f"INSERT INTO {table_name} SELECT * FROM df")
```
