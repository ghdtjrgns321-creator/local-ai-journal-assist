PRESET_QUERIES = {
    "high_risk_summary": """
        SELECT risk_level, COUNT(*) as cnt, SUM(debit_amount + credit_amount) as total_amount
        FROM general_ledger
        WHERE upload_batch_id = ?
        GROUP BY risk_level
        ORDER BY CASE risk_level WHEN 'High' THEN 1 WHEN 'Medium' THEN 2 ELSE 3 END
    """,

    "top_anomalies": """
        SELECT document_id, posting_date, gl_account, debit_amount, credit_amount,
               line_text, anomaly_score, risk_level
        FROM general_ledger
        WHERE upload_batch_id = ? AND anomaly_score > 0.5
        ORDER BY anomaly_score DESC
        LIMIT 50
    """,

    "rule_violation_stats": """
        SELECT rule_code, COUNT(*) as violation_count,
               AVG(score) as avg_score
        FROM anomaly_flags
        WHERE upload_batch_id = ?
        GROUP BY rule_code
        ORDER BY violation_count DESC
    """,

    "monthly_anomaly_trend": """
        SELECT DATE_TRUNC('month', posting_date) as month,
               COUNT(*) as total_entries,
               COUNT(CASE WHEN risk_level = 'High' THEN 1 END) as high_risk_count,
               AVG(anomaly_score) as avg_score
        FROM general_ledger
        WHERE upload_batch_id = ?
        GROUP BY month
        ORDER BY month
    """,

    "benford_distribution": """
        SELECT digit, observed_freq, expected_freq, deviation
        FROM benford_results
        WHERE upload_batch_id = ?
        ORDER BY digit
    """,

- **DuckDB 파일 경로:** `settings.duckdb_path` 참조, `data/` 폴더에 저장
- **pandas 직결합:** `conn.execute("SELECT * FROM df")` → DataFrame을 SQL 테이블처럼 직접 참조 가능
- **batch_id:** 업로드마다 UUID 생성 → 여러 번 업로드 시 데이터 구분
- **동시성:** DuckDB는 single-writer → Streamlit에서 동시 쓰기 주의 (read는 다중 가능)
- **스키마 변경:** Phase 2에서 컬럼 추가 시 `ALTER TABLE` 마이그레이션 고려
- **인덱스:** `document_id`, `upload_batch_id` 기준 인덱스 생성으로 쿼리 성능 확보

## 대용량 파일 직접 적재 (Phase 1b 확장 고려)

> Phase 1a 분석 결과: ingest 파이프라인이 전부 pandas 기반이라 DuckDB를 리더로 쓰면
> `.df()` 변환이 필수 → 메모리 이점 소실. 따라서 **ingest 완료 후 적재 시점**에서 DuckDB를 활용.

### 시나리오: 대용량 CSV → DuckDB 직접 적재

ingest 파이프라인(validate → read → header → map → cast) 완료 후,
표준 DataFrame이 너무 크면 **중간 Parquet 저장 → DuckDB에서 Parquet 읽기**로 메모리 절감 가능.

```python
# loader.py 확장안 (Phase 1b 구현 시 구체화)
def load_large_file(
    conn: duckdb.DuckDBPyConnection,
    df: DataFrame,
    table_name: str,
    batch_id: str,
    threshold_mb: int = 500,
) -> int:
    """대용량 DataFrame은 Parquet 경유, 소용량은 직접 적재.

    Why: 500MB+ DataFrame을 DuckDB에 직접 INSERT하면 피크 메모리가 2배.
    Parquet 중간 저장 → DuckDB COPY로 읽으면 피크를 DataFrame 크기로 제한.
    """
    estimated_mb = df.memory_usage(deep=True).sum() / (1024 * 1024)

    if estimated_mb > threshold_mb:
        # 대용량: Parquet 경유
        tmp_path = f"data/tmp/{batch_id}_{table_name}.parquet"
        df.to_parquet(tmp_path)
        del df  # 메모리 해제
        conn.execute(f"INSERT INTO {table_name} SELECT * FROM read_parquet('{tmp_path}')")
        Path(tmp_path).unlink()  # 임시 파일 삭제
    else:
        # 소용량: pandas 직결합
        conn.execute(f"INSERT INTO {table_name} SELECT * FROM df")
```
