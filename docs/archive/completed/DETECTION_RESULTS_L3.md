# Detection Results

> **PHASE1 역할 원칙**: PHASE1은 `fraud`를 확정하거나 정답 라벨을 맞히는 단계가 아니다. PHASE1의 목적은 전수 모집단에서 규칙 위반, 정책 위반, 이상 징후, 분석적 검토 신호를 넓게 올려 **감사인이 봐야 할 항목과 우선순위**를 만드는 것이다. DataSynth의 `is_fraud`/`is_anomaly`와 precision/recall은 개발 검증 보조 지표이며, 운영 해석은 예외 처리 대상, 감사인 리뷰 대상, 고위험 후보를 구분하는 review queue 기준으로 한다.

## Phase 1 L3 - DataSynth v126

대상 데이터는 `data/journal/primary/datasynth_v126_candidate`입니다.

이 문서는 L3 결과를 **세 가지 독립적인 검증 축**으로 나누어 봅니다. 세 축은 측정하는 대상이 다르므로 같은 표에 섞으면 의미를 잃습니다.

| 축  | 무엇을 검증하는가                                                  | 검증 데이터                                                                                | 합격 기준                                       |
| :-- | :----------------------------------------------------------------- | :----------------------------------------------------------------------------------------- | :---------------------------------------------- |

세 축의 관계:

- A축은 detector 사양과 데이터 생성기가 같은 정의를 공유하는지 보는 **계약 검증**입니다. 0/0이 합격선입니다.
- B축은 **detector 로직을 모르는 입장**에서 "현장에서 실제로 나올 만한 형태"를 sidecar로 만든 뒤 detector가 어떻게 반응하는지 봅니다. L3는 보조/검토 신호이므로 "위험점수 직접 반영 vs review-only 분리"가 핵심입니다.
- C축은 **실제 조작 전표 표본**(`manipulated_entry_truth.csv`)이 L3 책임 영역(수기/기말/관계사/고위험계정/비정상시점/업무범위)에 어떻게 걸리는지 봅니다.

L3는 Phase1 전체 점수에서 **단독 결론 신호가 아니라 보조/검토 신호**입니다. Phase1 집계는 L3 룰 점수를 단순 합산하지 않고 L3 family 안의 최대 normalized score에 L3 weight `0.20`을 곱합니다. L3가 많이 잡혀도 L3 단독으로 Low/Medium/High를 과하게 만들지 않고, L1/L2/L4 또는 다른 보강 증거와 결합될 때 우선순위를 올리는 구조입니다.

## 요약

- **A축 결과**: L3-01 ~ L3-12와 L3-12-CAND 모두 공식 정답 기준 과탐·미탐 0건.
- **B축 결과**: L3-12 raw candidate population sidecar(`work_scope_raw_candidate_population.csv`) 기준 "scored 점수 64건 / review 검토 63건"이 detector 분류와 일치. 과탐·미탐 없이 점수/검토 분기 정합.
- **C축 결과**: `manipulated_entry_truth.csv`의 실제 조작 전표 420건을 기준으로 보면 L3가 418건을 포착. 이 중 372건은 하나 이상의 L3 직접점수를 받고, 380건은 하나 이상의 L3 review 신호를 받음. 직접점수와 review는 서로 배타적이지 않아 같은 전표가 양쪽에 동시에 포함될 수 있음. L3 미포착 2건은 L1 승인/직무분리 책임 영역.

세 축을 분리하는 이유:

- A축이 깨끗해도 sidecar처럼 detector 코드를 보지 않은 모집단에서 분류가 어긋나면 현장에서 과탐이 터집니다.
- B축이 깨끗해도 fraud 합성 데이터에서 책임분 포착이 빠지면 실제 감사에서 누락이 발생합니다.
- 세 축이 모두 통과해야 L3가 보조/검토 신호로 의도대로 동작한다고 말할 수 있습니다.

또 "L3가 잡았다"와 "최종 위험점수에 강하게 반영한다"는 다릅니다. 예를 들어 L3-12는 Phase1 원칙상 업무범위가 넓으면 후보로 넓게 잡지만, 실제 점수는 `scored` 사용자-year에만 부여합니다. v126에서도 이 기준을 `L3-12 scored`와 `L3-12 candidate`로 분리한 상태가 유지되며, 둘 다 detector와 DataSynth 정답이 완전히 일치합니다. 이 분기 자체는 B축 영역입니다.

## 검증 축 B — Sidecar 현실 케이스 포착력 (DETECTOR 비참조)

이 축은 **detector 코드를 직접 참조하지 않고** "현장에서 실제로 나올 만한 케이스"를 sidecar로 만들어 둔 뒤 detector가 그것을 어떻게 분기하는지 봅니다. L3는 Phase1 전체 점수에서 보조/검토 신호이므로 B축의 핵심은 "잡힌 결과를 위험점수에 직접 넣을지 review-only로 뺄지"의 분기가 detector 동작과 정합한지입니다.

### Sidecar 평가 기준

| 항목      | 의미                                                                      |
| :-------- | :------------------------------------------------------------------------ |
| L3 책임   | 해당 sidecar가 L3가 직접 잡아야 하는 대상인지 여부 (예/아니오/별도)       |
| 실제 잡음 | L3 detector가 실제로 포착한 단위 수                                       |
| 위험점수  | 최종 위험점수에 반영해야 하는 단위 수                                     |
| 검토/별도 | 잡혔지만 위험점수에서 약하게 반영하거나 후보/별도 설명으로 보여줄 단위 수 |
| 미포착    | 해당 L3 룰이 잡지 않은 단위 수                                            |

L3 책임 값의 의미:

| 값     | 의미                                                                |
| :----- | :------------------------------------------------------------------ |
| 예     | L3가 직접 잡아야 하는 대상. 잡으면 L3 점수 또는 검토 점수에 반영    |
| 아니오 | L3 책임 영역이 아님. 잡지 않는 것이 정상                            |
| 별도   | 형식상 잡히지만 확정 위험으로 보지 않고 검토/후보/드릴다운으로 분리 |

### Phase1 전체 점수 유입 관점

L3는 Phase1 전체 점수에서 `RULE_LEVEL_WEIGHTS["L3"] = 0.20`으로 들어갑니다. 또한 L3 룰끼리는 단순 합산하지 않고 L3 family 안에서 가장 큰 normalized score 하나가 대표값이 됩니다.

v126 기준 L3 종합 점수의 동작은 다음과 같습니다.

| 항목                        |        값 |
| :-------------------------- | --------: |
| 전체 rows                   | 1,109,435 |
| L3 종합점수 > 0 rows        |   954,272 |
| L3 후보지만 종합점수 0 rows |   155,159 |
| L3 단독 Low 이상 rows       |         0 |
| L3 단독 Medium 이상 rows    |         0 |
| L3 단독 High 이상 rows      |         0 |
| L3 최대 Phase1 기여점수     |      0.12 |

상위 점수 driver는 L3-09입니다. 가수금 장기체류 raw score `0.80`이 L3 normalized score `0.60`으로 들어가고, Phase1 L3 weight `0.20`을 곱해 row-level 기여점수 `0.12`가 됩니다. 즉 위험한 L3는 위로 올라오지만, L3 단독으로 전체 위험등급을 과하게 만들지는 않습니다.

## 검증 축 C — 실제 조작 전표 포착력 (`manipulated_entry_truth.csv`)

`manipulated_entry_truth.csv`는 **실제 조작 전표 평가용 표본**입니다. 이 파일은 L3 특정 룰의 정답 파일이 아니라, 여러 fraud 시나리오로 만들어진 전체 조작 전표 420건 목록입니다.

따라서 C축은 A축/B축과 보는 기준이 다릅니다.

| 축 | 기준 파일 | 평가 질문 | 숫자의 의미 |
| :--- | :--- | :--- | :--- |
| A축 | `rule_truth_L3_*.csv` | 각 L3 룰 계약을 정확히 맞추는가 | 룰별 정답/탐지/과탐/미탐 |
| B축 | L3 sidecar population | 현실적인 후보를 점수/검토로 잘 분리하는가 | 후보 모집단의 점수/검토 분기 |
| C축 | `manipulated_entry_truth.csv` | 실제 조작 전표가 L3 신호에 얼마나 걸리는가 | 조작 전표 420건 중 L3 포착 범위 |

C축의 `L3 후보`, `직접점수`, `review`는 서로 더해서 420이 되는 값이 아닙니다.
한 조작 전표가 `L3-02 직접점수`와 `L3-12 review`를 동시에 받을 수 있으므로, 직접점수 372건과 review 380건은 서로 겹칠 수 있습니다.

L3가 모든 조작 전표를 확정 위험점수로 잡아야 하는 것은 아닙니다.
다만 L3는 수기/기말/관계사/고위험 계정/비정상 시점/업무범위 같은 조작 패턴을 넓게 포착해야 하므로, C축에서는 "실제 조작 전표가 L3 후보·점수·review 신호에 얼마나 걸리는지"를 봅니다.

### 전체 요약

| 항목 | 건수 | 의미 |
| :--- | ---: | :--- |
| 전체 조작 전표 | 420 | `manipulated_entry_truth.csv`의 실제 조작 전표 수 |
| L3가 실제 잡은 전표 | 418 | 하나 이상의 L3 raw candidate에 걸린 전표 |
| 직접 위험점수 전표 | 372 | 하나 이상의 L3 `score_series > 0`을 받은 전표 |
| review/검토 점수 전표 | 380 | 하나 이상의 L3 `review_score_series > 0`을 받은 전표 |
| L3가 잡지 않은 조작 전표 | 2 | 어떤 L3 후보에도 걸리지 않은 전표 |

v126 기준으로는 전체 조작 전표 420건 중 418건이 L3 후보로 포착됩니다.
그중 372건은 하나 이상의 L3 직접 위험점수를 받고, 380건은 하나 이상의 L3 review 신호를 받습니다.
직접점수와 review는 같은 전표에 동시에 붙을 수 있으므로 `372 + 380`으로 합산하지 않습니다.
