# 단위 측정 정책

이 문서는 Local AI Audit Assistant의 PHASE1·PHASE2 탐지 단위와 측정 기준의 단일 출처다.
사용자와 에이전트는 탐지 단위와 측정 기준을 해석할 때 이 문서를 우선 적용한다.

## 0. 목적

프로젝트는 PHASE1 룰 hit와 PHASE2 모델 신호를 먼저 `document` 또는 `flow` 탐지 단위에 귀속한다.
row는 해당 단위 안에서 위반 위치를 가리키는 증거 포인터로만 사용한다.
사용자 화면의 case, 사용자×월, 계정×월 같은 묶음은 집계 뷰이며, 자체 정답·분모·독립 점수를 갖지 않는다.

단위가 정의되지 않아 PHASE1·PHASE2의 분모가 룰·case·버전마다 뒤섞였고, 측정이 서로 비교 불가가 됐다.
이 문서는 탐지 단위와 측정 기준의 단일 출처다.

## 1. 단위 3층 모델

```text
Layer   이름                         정책
------  ---------------------------  ------------------------------------------------------------
1       탐지 단위                    정답이 붙는 곳이다. document, flow만 단위다.
2       룰 분류                      각 룰은 document-rule 또는 flow-rule이다. 구현용이며 측정 분모에 쓰지 않는다.
3       집계 뷰                      탐지 결과를 속성으로 GROUP BY 한 표시 레이어다. 자기 점수·자기 분모가 없다.
```

- Layer 1의 탐지 단위는 `document`, `flow` 두 가지뿐이다.
- `row`(분개 한 줄)는 단위가 아니다. `document` 안에서 위반이 어느 줄인지 가리키는 증거 포인터다.
- Layer 2의 룰 분류는 각 룰이 `document-rule`인지 `flow-rule`인지 나타낸다. 이 분류는 전표 링크 필요 여부를 결정하는 구현용 태그이며, 측정 분모에 쓰지 않는다.
- Layer 3의 집계 뷰는 탐지 결과를 속성으로 `GROUP BY` 한 표시 레이어다. 집계 뷰는 자기 점수·자기 분모를 갖지 않는다.

## 2. 탐지 단위 정의

### 2.1 document

`document`(전표)는 차변 합 = 대변 합으로 균형 잡힌 row 묶음이다.
식별자는 `document_id`다.
`document`는 회계 사건의 원자 단위다.

### 2.2 flow

`flow`(흐름)는 단일 전표만으로는 이상 판단이 불가능하고, 여러 전표를 명시적 링크로 묶어야 의미가 생기는 거래 흐름이다.
식별자는 `flow_id`와 링크 키다.

```text
링크 키 종류   링크 키
-------------  ------------------------------------------------
중복           거래처 + 금액 + 근접기간
내부거래       회사쌍 양방향 대사
순환           그래프 경로(A→B→C→A)
```

### 2.3 row

`row`는 증거 포인터다.
표현은 `document_id + row_index`다.
`row`는 탐지 단위가 아니다.

## 3. 정답(truth) 단위 배정 규칙

- 각 부정 시나리오는 자연 단위를 정확히 하나만 갖는다. 단위는 `document XOR flow`다.
- 각 부정 시나리오의 자연 단위는 카탈로그에서 고정한다.
- 판정 기준은 단일 전표만 보고 이상 판단이 가능하면 `document`, 여러 전표를 링크해야 보이면 `flow`다.
- 결과적으로 모든 truth 집합은 서로 겹치지 않는다. 즉, truth 집합은 disjoint다.

## 4. 측정 규칙

### 4.1 분모

분모는 `(document-natural 부정 수) + (flow-natural 부정 수)`다.
두 집합이 disjoint이므로 단순 합산해도 안전하다.

### 4.2 분자

분자는 truth item 기준으로 dedup한다.
truth 1건이 1개 이상 룰 hit으로 cover 되면 1로 세고, 아니면 0으로 센다.

### 4.3 중복 방지 원리

`flow`는 `document`를 포함하므로 `document 수 + flow 수`를 그냥 더하면 같은 부정을 여러 번 세게 된다.
부정마다 단위를 하나로 배정하는 것(§3)이 중복 방지 장치다.

어느 단위로 잡았는지, 즉 `flow` 또는 `document`는 진단 메타데이터일 뿐이며 세는 축이 아니다.

예를 들어 순환거래 1건이 전표 3건으로 구성되면 단위는 `flow`이므로 truth는 1건이다.
그 안의 전표가 `document-rule`에도 걸려도 같은 truth의 추가 증거이지 새 truth가 아니다.
따라서 이 경우에도 1건으로 센다.

## 5. 집계 뷰 정책

집계 뷰는 기존 `case`의 위치를 정의한다.
집계 뷰는 `document` 또는 `flow` 탐지 결과를 속성으로 `GROUP BY` 한 화면이다.
예시는 사용자×월, 계정×월이다.

집계 뷰의 목적은 감사인이 어느 사용자의 몇 월 등 특정 상황에 위반이 몰렸는지 보고 집중 검토하게 하는 것이다.

- 집계 뷰는 자기만의 정답·분모·독립 점수를 갖지 않는다.
- 집계 뷰에 보이는 숫자는 1차 탐지 결과(`document` 또는 `flow`)의 카운트 또는 합이다.
- 집계 뷰를 recall 또는 precision의 분모로 사용하지 않는다.
- 집계 뷰의 유용성은 coverage로 측정한다. coverage는 그 버킷 안에 truth가 몇 % 포함되는가를 보는 지표이며, recall과 다른 지표다.
- 중복세트, 내부거래, 순환 같은 흐름은 집계 뷰가 아니라 Layer 1의 `flow` 탐지 단위다.

## 6. 룰 분류 정책

각 룰에는 `document` 또는 `flow` 태그를 붙인다.
`flow-rule`은 탐지 전에 전표를 링크하는 단계가 필요하다.

이 태그는 코드 구조용이며 측정 분모로 쓰지 않는다.
측정 분모는 부정의 단위(§3)를 기준으로 정한다.

## 7. PHASE1 / PHASE2 공통 적용

PHASE1은 룰 hit(row 발화)를 `document` 또는 `flow` 탐지 단위로 귀속한 뒤 집계 뷰로 표시한다.
정답은 `document` 또는 `flow`에서만 센다.

PHASE2의 타깃은 PHASE1이 못 잡거나 순위가 밀린 `document` 또는 `flow`다.
PHASE2는 PHASE1과 동일한 단위 체계를 쓴다.

PHASE1의 집계 뷰와 PHASE2의 집계 뷰가 다르게 묶여도 무방하다.
둘 다 동일한 `document` 또는 `flow` 식별자를 가리키므로, 같은 부정을 봤는지는 집계 뷰가 아니라 `document` 또는 `flow` 식별자 교집합으로 비교한다.

## 8. 금지·가드

- 단위를 룰·case·버전마다 다르게 두지 않는다. 단일 체계를 사용한다.
- 같은 부정을 여러 단위로 중복 카운트하지 않는다.
- 집계 뷰를 정답 분모로 쓰지 않는다.
