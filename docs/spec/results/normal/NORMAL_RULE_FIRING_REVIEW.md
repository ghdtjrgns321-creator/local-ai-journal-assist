## 2. L2-05 역분개 — datasynth 설계 vs detector 발화 (핵심)

`scratchpad/design_vs_detector_v51.py`로 실제 `c11_reversal_entry(match_window_days=45)`를
호출해 발화 문서와 설계 역분개 라벨을 교차했다.

| 구분                                                           | 문서 수 | 비율         |
| -------------------------------------------------------------- | ------- | ------------ |
| datasynth 설계 역분개(`reversal_type=normal_accrual_reversal`) | 2,600   | 2.33%        |
| L2-05 detector 발화                                            | 20,034  | 17.97%       |
| ├ 설계 역분개와 겹침(정상 탐지)                                | 2,600   | 발화의 13.0% |
| └ 거울쌍만으로 걸림(설계 밖)                                   | 17,434  | 발화의 87.0% |
| 설계 역분개 중 놓친 것(미탐)                                   | 0       | recall 100%  |

- **정상 데이터의 "진짜 역분개"는 2.33%**다(datasynth가 원전표↔익월취소 pair로 의도 생성).
  detector는 이 2,600건을 100% 잡는다(미탐 0).
- 그러나 detector 발화의 **87%(17,434건)는 설계 밖 거울쌍**이다. 같은 계정·유사금액(±2%)·반대
  방향·45일 내 조건이, 정상 영업의 상계성 거래를 역분개로 넓게 집계한다. 걸린 계정은
  `standard_account` 13,628 + `COGS_MATERIAL` 2,144 등 일반 영업 계정이고 업무는 O2C·P2P·TRE
  전반에 퍼져 있다.
- **해석**: 이건 datasynth 결함이 아니다(설계 2.33% 정확, gate PASS). 룰 카드가 L2-05를
  "정상에 많은 검토 트리거(단독 LOW)"로 의도한 대로의 넓은 집계다. 다만 정상 데이터의 18%를
  역분개 후보로 올리는 것은 리뷰 부담이 크므로, 45일 창 추가 축소나 "같은 계정" 외 상계 배제
  조건 강화가 선택적 개선 과제다.
- 웹 문헌: "정상 회사 원장의 역분개 비율" 공개 정량 수치는 **5회 검색**(AccountingTools 직접
  fetch, Oracle/Sage ERP 문서, MindBridge JE memo, PLANERGY, Indeed 등)으로도 없음 —
  회사 결산관행마다 달라 업계 공통 벤치마크가 존재하지 않음이 확정적이다. 따라서 임계 판정
  근거는 문헌이 아니라 **datasynth 설계 계약(2.33%)**이 정답 기준이다.

## 3. D01 계정활동 급변 — 신규 vs 기존급변 분해

VarianceDetector를 CSV 내 연도쌍으로 직접 구동(`prior_data_loader`와 동일 집계). 파이프라인은
anonymous context라 `src/pipeline.py:1543`에서 layer_d를 스킵하므로 별도 구동.

| 연도쌍       | review 계정     | 신규계정 | 기존계정 급변 | 가중변동률(median/p90/max) |
| ------------ | --------------- | -------- | ------------- | -------------------------- |
| 2023 vs 2022 | 142/426 = 33.3% | 0        | 142 (33.3%)   | 0.75 / 3.48 / 95.54        |
| 2024 vs 2023 | 179/428 = 41.8% | 2        | 177 (41.4%)   | 0.86 / 3.10 / 1291.31      |

(임계 `variance_threshold`=0.5. 가중변동률 = total_var×0.5 + count_var×0.3 + avg_var×0.2)

- D01이 잡은 계정은 **거의 전부 기존계정 급변**이다(신규계정은 0~2개). 즉 "계정이 새로
  생겨서"가 아니라 "기존 계정의 전기 대비 활동량이 50%+ 변동"해서 걸린다.
- 계정의 **절반(median 0.75~0.86)이 전년비 75~86% 변동**한다. 안정적 제조업의 정형 계정
  (임차·감가상각·정기비용)은 보통 이렇게 크게 안 변한다 — datasynth가 연도별 규모를 크게
  흔들었거나 3사×3년 소표본 영향으로 보인다.
- **검증 갭**: datasynth 테스트 카탈로그 C07은 "안정 계정 활동량이 매년 무작위로 급변하면
  실패"로 정의하지만, **C07/C08은 realism gate 스크립트에 아직 구현되지 않았다**(gate test_id에
  C07/C08 없음). 즉 "이 급변이 정상 사업 이벤트인지 안정계정 무작위 급변(C07 위반)인지"를
  자동 검증하는 장치가 없다. 현재 유일한 증거는 위 실측(median 변동률 0.75+)이며, 이는 안정계정
  기준으로는 높은 편이다.
- D01 detector 자체는 설계대로 작동한다(row score 0, macro 큐로 분리, recall 우선). detector
  버그가 아니라 **datasynth의 연도별 계정 변동폭이 실제 안정 제조업 대비 큰지 확인이 필요한
  realism 문제**다.
- 웹 실무 기준(flux/fluctuation analysis): 실무 표준 조사 임계는 **"10% 또는 절대금액(예:
  $25,000) 중 먼저 걸리는 것"** 조합이다(Ramp·FloQast·Forbes Finance Council). materiality
  자체는 0.5~10% 범위로 단일 합의값 없음. 즉 실무는 **10%부터** 계정을 조사하는데, 우리 D01은
  **50% 가중변동** 임계라 실무보다 오히려 보수적(넓지 않음)이다. 그 보수적 임계로도 33~42%
  계정이 걸린다는 것은 datasynth 데이터의 연도별 계정 변동폭이 실제 안정 기업보다 크다는 방향을
  가리킨다. "정상 회사에서 계정 몇 %가 유의변동하는지"의 직접 정량 통계는 공개된 것이 없다
  (회사·업종마다 다름).

## 3-A. L2-05/D01 원인 귀속 — datasynth냐 detector냐 (2026-07-02 확정)

`scratchpad/l205_d01_attribution.py`, `scratchpad/d01_reclassify.py`로 실측.

### L2-05 → **detector 수정 대상** (datasynth·45일창 아님)

거울쌍-only 매칭 24,466라인(설계 역분개 제외)의 정체:

| 지표                  | 값                 | 의미                             |
| --------------------- | ------------------ | -------------------------------- |
| 금액 정확일치(±0)     | 294 = **1.2%**     | 진짜 되돌림이면 정확 상쇄돼야 함 |
| 금액 ±2% 근사(불일치) | 24,172 = **98.8%** | 우연히 금액만 비슷한 별개 거래   |
| reference 공유        | 138 = **0.6%**     | 같은 거래면 증빙 공유돼야 함     |
| trading_partner 공유  | 32 = **0.1%**      | 같은 거래면 거래처 동일해야 함   |

- 판정: reference 공유 0.6%(<30%) AND 정확일치 1.2%(<50%) → **무관 거래 우연매칭이 압도적**.
  detector의 "같은 계정 + 반대방향 + 금액 ±2% 근사" 조건이, 정상 영업 순환(예: 매출채권 계정에
  이번 달 매출 발생 / 몇 주 뒤 무관한 다른 건 회수)을 역분개로 오인한다.
- **45일 창은 합리적이고, datasynth 역분개(2.33%)도 적정하다. 고칠 대상은 detector다.** 창 축소나
  datasynth 역분개 감소가 아니라, 매칭 조건 강화가 근본 해결: (a) 금액 정확일치 요구, 또는
  (b) reference/original_document_id 연결 없으면 제외, 또는 (c) AR/AP 등 정상 순환계정 제외.
  datasynth를 줄이면 실제 회사에 흔한 정상 순환 거래를 비현실적으로 없애는 부작용이 난다.

#### L2-05 개선안 시뮬레이션 (`scratchpad/l205_simulate.py`, 실제 _s0/_s1 함수 호출)

detector는 두 경로 OR: s0(ERP 연결 필드 `original_document_id`/`reversal_document_id`/
`reversal_reason` 등) | s1(같은계정 거울쌍, 금액 tolerance ±2%).

**결정적 사실: 설계 역분개 2,600건은 전부 s0(ERP 연결)로 잡힌다(recall 100%). s1은 recall에
0 기여, 순수 FP만 추가한다.**

s1 tolerance 민감도(45일 창 고정):

| tolerance      | s1 발화 | 설계 recall | s1-only(s0밖 FP) |
| -------------- | ------- | ----------- | ---------------- |
| ±2% (현행)     | 20,034  | 2600/2600   | 17,434           |
| ±0.5%          | 8,237   | 2600/2600   | 5,637            |
| ±0% (정확일치) | 2,807   | 2600/2600   | 207              |

개선안 대조 (recall = 설계역분개 M/2600):

| 개선안                     | 총 발화 | 설계 recall | 설계밖(FP추정) |
| -------------------------- | ------- | ----------- | -------------- |
| 현행 s0\|s1@2%             | 20,034  | 2600/2600   | **17,434**     |
| s0만 (거울쌍 제거)         | 2,600   | 2600/2600   | **0**          |
| s0\|s1@0% (정확일치)       | 2,807   | 2600/2600   | **207**        |
| s0\|s1@0% + reference 공유 | 2,634   | 2600/2600   | **34**         |

- **근본 원인은 45일 창이 아니라 ±2% 금액 tolerance다.** ±2%가 "같은 계정·반대방향인데 금액만
  우연히 2% 이내로 비슷한 무관 거래"를 17,434건 오탐한다. tolerance를 ±0%로만 바꿔도 FP가
  17,434→207로 **98.8% 감소**하며 설계 recall은 100% 유지된다.
- **권장(균형): s1 tolerance ±2% → ±0%(정확일치)로 축소.** 근거: 진짜 역분개는 원전표를 정확히
  되돌리므로 금액이 정확 일치한다. 정확일치가 아니면 역분개가 아니라 별개 거래다. config 값
  (`amount_tolerance`)만 바꾸면 되고 코드 구조 변경 불필요.
- **더 강하게(FP 34): + reference 공유 조건.** 단 실데이터에 reference 없이 정확일치하는 진짜
  역분개가 있으면 놓칠 수 있어, ERP 필드 coverage가 낮은 실데이터에서는 신중히.
- **s0만 쓰기(FP 0)는 권장 안 함**: datasynth는 모든 역분개에 ERP 필드를 채웠지만, 실데이터엔
  ERP 연결 필드 없는 역분개가 있을 수 있어 s1은 fallback 안전망으로 남겨야 한다(단 tolerance 0).
- 검증 케이스: 정상 데이터 v51 1종 + s1 3개 tolerance(2%/0.5%/0%) = ripple. fraud 데이터에서
  tolerance ±0%가 진짜 역분개 기반 부정을 놓치는지는 별도 확인(사용자 fraud 트랙).

### D01 → **datasynth realism 수정 대상** + C07 gate 신설 (detector는 정상)

급변(activity_variance) 계정 중 "안정성 계정"(법인세·이자·감가상각·임차·대손) 개수:

| 연도쌍       | 급변 계정 | 안정성 계정 급변 | 대표 극단값                       |
| ------------ | --------- | ---------------- | --------------------------------- |
| 2023 vs 2022 | 142       | **12**           | 법인세 9.37배, 이자 5.38배        |
| 2024 vs 2023 | 177       | **21**           | 법인세 **133.90배**, 이자 15.36배 |

- 판정기준(안정성 계정 ≥5개면 datasynth realism 문제)을 크게 초과(12·21개). 법인세·이자비용은
  이익/차입 연동이라 어느 정도 변동은 정상이나, **133배·15배는 비현실적**이다. 감가상각누계·임차료도
  급변 목록에 있다. 안정적이어야 할 계정이 연도별로 크게 흔들린다 = datasynth의 연도별 계정 규모
  변동이 실제 안정 제조업 대비 과도하다.
- **D01 detector 자체는 정상**이다(row score 0, macro 큐 분리, recall 우선 넓은 포착). 고칠 대상은
  datasynth(안정계정 연도별 변동 축소)다.
- caveat: 3사×3년 소표본이라 전기 금액이 작은 계정은 total_var 분모가 작아 변동률이 폭발할 수
  있다(법인세 133배는 전기 법인세가 작았을 가능성). 그래도 안정계정 12~21개 급변은 패턴이므로
  realism 개선 대상이다.
- **C07/C08 gate 신설 필요**: 이 "안정계정 급변"을 자동으로 잡을 realism 검증(C07)이 gate에 없어서
  지금까지 안 걸렸다. datasynth 수정과 함께 gate 항목을 추가해야 회귀 방지가 된다.

### D02 → **realism 문제 아님** (D01과 대조적, detector·datasynth 모두 손댈 필요 없음)

`scratchpad/d02_attribution.py`로 flagged 계정그룹을 계정명 분류(2연도쌍):

| 연도쌍       | 진단계정 | flagged | 안정성 계정 flagged                 |
| ------------ | -------- | ------- | ----------------------------------- |
| 2023 vs 2022 | 426      | 46      | **1** (Interest expense, jsd 0.442) |
| 2024 vs 2023 | 426      | 43      | **0**                               |

- 판정기준(안정계정 flagged ≥3이면 datasynth realism 문제)을 **미달**(1개·0개). D01(안정계정
  12·21개 급변)과 정반대다.
- flagged 계정은 거의 전부 고정자산·매입/매출채무·현금·단기/장기차입금·미지급비용·재고·선수금
  등 **월별 잔액·거래가 자연스럽게 변동하는 자산·부채 계정**이다. 이자·감가상각·임차 같은 안정계정의
  월분포 요동은 사실상 없다.
- 결론: D02가 올리는 flagged 10%는 정상 영업의 월별 분포 변화를 넓게 포착한 것(설계 의도, macro
  큐 분리). **detector도 datasynth도 이 룰 때문에 손댈 필요 없다.**
- caveat: 자산·부채 계정의 월분포 변화가 "자연스러운 계절 패턴"인지 "연도간 무작위 요동(C08 위반)"
  인지까지는 JSD 값만으로 구분 못 했다. 완전 검증은 C08 gate(월별 비율 연도간 안정성)가 필요하나,
  안정계정 관점에서는 문제 없음이 확인됐다.

## 4. 웹 조사 결론 (2026-07-02, 완료)
