# 6. PHASE1-1 룰 엔진

## 6.1 파이프라인 내 위치

```
생성 → 품질검증 → 수집·파생·검증 → ▶▶▶ [탐지 + 점수 정규화] ◀◀◀ → 표면 → 적재 → 화면
```

전표 단위 규칙 위반을 명명해 올리는 층이다. 여기서 나온 발화가 [7장](7_PHASE1-1_COMBO-BUILDER.md)의 조합 빌더 어휘가 된다.

## 6.2 내부 흐름

```
검증 통과 DataFrame
        │
        ├─▶ IntegrityDetector (layer_a) ── L1-01~03 ─────┐
        ├─▶ FraudLayer      (layer_b) ── 14개 룰 ───────┤
        ├─▶ AnomalyDetector (layer_c) ── 11개 룰 ───────┤  ThreadPool 병렬
        ├─▶ BenfordDetector (benford) ── L4-02 ─────────┤  결과는 입력 순서로 재정렬
        └─▶ EvidenceDetector(evidence) ── L3-11 ────────┘
                        │
                        ▼
             DetectionResult × 5
             (flagged_indices, scores, rule_flags, details, metadata)
                        │
                        ▼
             rule_scoring — normalized_score
             = 신호강도 × 심각도 × 증거강도 × 역할 × 기여가중치
                        │
                        ├──▶ score_aggregator (행 단위 anomaly_score)
                        └──▶ phase1_case_builder (전표·흐름 단위 표면)
```

탐지기는 **실패해도 결과 객체를 반환한다.** `run_status`가 `executed`/`skipped`/`failed`이고 `skip_reason`이 계약 필드로 존재해, "이 룰은 실행되지 않았다"가 침묵이 아니라 데이터로 남는다. 컬럼이 없어 돌지 못한 룰과 돌았는데 0건인 룰은 전혀 다른 의미이므로 구분이 필요하다.

## 6.3 룰이 몇 개인가 — 세는 기준이 넷이다

이 프로젝트에서 "룰 개수"는 단일 숫자로 말할 수 없다. 실측한 네 가지 기준이 서로 다르다.

| 기준                     |   개수 | 무엇을 세나                                                                              |
| ------------------------ | -----: | ---------------------------------------------------------------------------------------- |
| 탐지 callable 레지스트리 | **35** | 실제 탐지 함수. 접근감사 4종·증빙 2종 포함                                               |
| 점수 정규화 레지스트리   | **37** | `RULE_SCORING_REGISTRY` 키. L2-03 하위변형 4종과 Benford 별칭 포함, 접근감사·증빙 미포함 |
| S2 활성 표면             | **29** | 35 − 비활성 6 (설정 게이트 기본 off 5 + **미구현 스켈레톤 1**)                           |
| S1 판정 분모             | **28** | 대역표 33행 − PHASE1-2 이관 1 − 미분리 계측 1 − 탐지기 부재 구 ID 3                      |
| 문서상 canonical         |     29 | `CANONICAL_TRANSACTION_RULE_IDS` (import 시점에 개수를 assert)                           |

비활성 6종의 내역이 중요하다. 접근감사 AA01·AA03·AA04와 증빙 EV01·EV03은 설정 게이트가 기본 off인 옵션 확장이고, **AA02는 미구현 스켈레톤**이다 — 필요한 `ip_address` 컬럼이 원장에 없어 어떤 데이터로도 발화할 수 없다. 이것을 "구현됨"으로 세면 분모가 오염되므로 S2 판정에서 제외했다.

이 구분을 흐리면 "31개 룰 전부 검증 완료" 같은 문장이 만들어진다. 실제로는 29개를 검증했고 6개는 애초에 대상이 아니었으며 3개는 탐지기가 존재하지 않는 유령 ID였다.

## 6.5 점수 정규화

```
normalized_score = 신호강도 × 심각도계수 × 증거강도계수 × 역할계수 × 기여가중치  → [0,1]
```

| 계수     | 값                                                                |
| -------- | ----------------------------------------------------------------- |
| 신호강도 | 상 1.0 / 중 0.6 / 하 0.3 / 정보 0.2                               |
| 증거강도 | strong 1.0 / medium 0.75 / weak 0.45 / info 0.25                  |
| 역할     | primary 1.0 / booster 0.65 / combo_only 0.35 / **macro_only 0.0** |

역할 분포는 primary 26 / macro_only 6 / booster 5다. macro_only 6종(L3-12, L4-02, Benford, L4-06, D01, D02)은 계정·모집단 단위 신호라 **행 점수 기여가 0**이다.

여기에 설계 함정이 코드 주석으로 명시돼 있다. macro_only 룰을 레지스트리에서 **지우면** 폴백이 이를 primary로 잡아 점수가 오히려 붙는다. 따라서 제거가 아니라 역할계수 0으로 "중화"해야 한다. 등록을 지우는 것이 안전해 보이지만 반대로 작동하는 경우다.

행 단위 점수 산출은 두 단계다. **탐지기 내부에서는 최댓값**을 취해 같은 트랙의 룰 여러 개가 걸려도 가장 강한 신호 하나로 대표시키고, **종합 점수는 L1~L4 계층별 최댓값을 가중 합산**한다(L1 0.40 / L2 0.25 / L3 0.20 / L4 0.15). 같은 계층 안에서는 누적되지 않고 서로 다른 계층에 걸칠 때만 누적된다. 승인 통제 위반과 시간 이상이 함께 있는 전표는 둘 중 하나만 있는 전표보다 높은 점수를 받되, 같은 계층 룰이 여러 개 걸렸다는 이유만으로는 올라가지 않는다.

## 6.6 정상에서 조용한가 — 발화율 대역 판정

검증의 첫 관문은 부정 탐지가 아니라 **정상 데이터에서의 침묵**이었다. 룰이 정상 원장의 30%에 발화한다면 그것은 검토 목록이 될 수 없다.

핵심 장치는 **측정 전 대역 선언**이다. 실측한 뒤 그 숫자에 맞춰 기준을 정하면 아무것도 증명하지 못하므로, 룰별 허용 대역을 문서에 먼저 박고 나서 측정했다. 대역 밖은 자동 불합격이 아니라 조사 대상이며, 조사 결과를 ①생성기 결함 ②룰 과탐 ③대역 오선언으로 3분류한다. ③으로 판정하려면 도메인 근거가 필수다(숫자 맞추기 금지).

### 유일한 실패 — L4-03의 복합 결함

절대 고액 룰이 3.6137%에 발화했다(대역 상한 0.6%). 조사 결과 두 원인의 복합이었다.

**룰 쪽 결함(주원인).** 임계를 세전이익 기준으로 산출하는데, 마감분개를 식별하는 키가 현행 데이터와 맞지 않았다. 그 결과 마감 라인이 매출 합산에 섞여 순이익이 잘못 계산되고(3년간 2.1B), 임계가 연 2,018만~3,007만원으로 붕괴했다. 전표 금액 90분위가 1,944만원이므로 사실상 상위 10%가 전부 걸린 것이다.

**생성기 쪽 결함(병행).** 마감을 제외한 실측 손익이 매출 연 119~124B 대비 비용 연 398~420B로, **비용이 매출의 3.4배**다. 존속 불가능한 구조이며 이 상태로는 올바른 임계도 계산할 수 없다.

당시 판정은 **FAIL로 기록**했다. 대역표에 "27/28 통과"로 남기고 실패를 감추지 않는 것이 이 검증의 규칙이다.

**이후 수정 (2026-07-20) — 룰 쪽 주원인 해결.** 마감분개를 subtype이 아니라 header_text의 "손익 마감" 패턴으로 식별해 매출·비용 벤치마크 집계에서 제외하도록 고쳤다(설정 `l403_materiality.closing_header_patterns`). 그 결과 FY2024 매출이 gross 1,175억으로 복원돼 수행중요성이 **4.41억으로 정상 산출**된다(붕괴 전 0.2억=2천만원). 다만 이 임계로 L4-03 발화율을 재측정하지는 않아 대역 판정은 열려 있고, 생성기 쪽 손익 비정합(비용>매출, 적자)은 별도 과제로 그대로다([12.3 ⑧](12_DIFFERENTIATION.md)).

**수행중요성(PM) 자동 산출 방식 — ISA 320.** 감사팀이 값을 넣지 않으면 회사×연도 단위로 원장에서 자동 결정된다:

| 조건               | 임계(수행중요성)                  | 근거                              |
| ------------------ | --------------------------------- | --------------------------------- |
| 순이익(PBT) > 0    | `순이익 × 5% × 0.75`              | PBT 벤치마크                      |
| 순이익 ≤ 0·저마진  | `매출 × 0.5% × 0.75` (매출 floor) | ISA 320 — 적자·저마진은 매출 기준 |
| 감사팀 확정값 입력 | 그 값이 override로 우선           | 감사인 판단                       |

- 두 벤치마크 모두 **마감분개를 합산에서 제외**한다(위 수정). 제외하지 않으면 마감 차변이 매출을 상쇄해 임계가 붕괴한다.
- 대시보드에서 중요성 칸을 **비우면 이 자동값**이, 값을 넣으면 그 값이 쓰인다(리터럴 기본값 10억 제거).
- 현행 데이터는 적자라 매출 floor가 적용돼 **4.41억**이 자동으로 들어간다.

### 죽은 룰 3종을 되살린 라운드

s9 세대에서 12개 룰이 0건이었다. 0건은 "정상이 깨끗하다"일 수도 있고 "룰이 죽었다"일 수도 있어 전수 재점검했고, 결과는 수정 3 / 정당 6 / 탐지기 부재 3이었다.

| 룰                  | 0건이었던 이유                                                 | 수정                                                            |
| ------------------- | -------------------------------------------------------------- | --------------------------------------------------------------- |
| L3-11 수익 컷오프   | 통제이전일이 입고 문서 전용이라 매출 전표에 없음 → 구조적 침묵 | 출하·입고 기준 통제이전일 원설계 복원, 최종 전기일 확정 후 앵커 |
| L3-07 증빙 괴리     | 생성 분포가 ±29일로 잘려 있어 30일 초과가 원천 불가            | 매입·수기 계열 0.2%에 31~90일 지연 꼬리 주입                    |
| L3-09 가계정 미정리 | 정리 신호 컬럼 자체가 원장에 없음                              | `is_cleared` 신설(가계정 행 전용, 최근 30일 미정리 50%·이후 3%) |

셋 다 **실제 회사에는 있어야 할 현상이 생성기 구멍으로 0이었던 경우**다. 처음에는 "정상이니까 0이 맞다"고 대역을 재선언하려 했으나, "정상 데이터에 컷오프가 한 건도 없는 게 이상하다"는 지적으로 재개봉해 생성기 갭으로 재분류했다. 셋 다 수정 후 사전 선언 대역 안에 들어왔고 게이트 회귀도 없었다.

0건이 정당한 6종도 사유가 분명하다: L1-01~03은 ERP가 구조적으로 강제하는 항목(차대 불일치·필수필드 누락·미등재 계정은 애초에 저장되지 않음), L1-08은 특별회계기간을 쓰지 않는 ERP에서 기간이 곧 전기월, L1-07-02는 승인자가 전원 명부에 실재, L3-12는 사용자 전담 설계와 정합.

## 6.7 쏴야 할 때 쏘는가 — 룰 단위시험

정상에서 조용한 것만으로는 부족하다. 룰이 아예 고장 나 있어도 조용하기 때문이다. 그래서 룰별로 **교과서적 위반 전표를 심고 자기 표적에서 발화하는지** 판정했다.

| 항목 | 내용                                                                          |
| ---- | ----------------------------------------------------------------------------- |
| 배경 | 정본 base의 FY2024 슬라이스 119,205행                                         |
| 주입 | 룰별 최소 위반 사례 2~5건 — 총 74행 / 36문서                                  |
| 분모 | 탐지기 레지스트리에서 스크립트로 추출해 파일로 고정 (문서의 숫자를 믿지 않음) |
| 판정 | **표적 적중 기준** — "발화했지만 다른 전표에서"는 실패로 센다                 |
| 결과 | **29 / 29 PASS, 룰 결함 0건**                                                 |

1차 판정은 26/29였다. 미발화 3건을 추적한 결과 **전부 시험 장치의 결함**이었다.

| 룰              | 1차 미발화 원인                                                                             | 교정                                                           |
| --------------- | ------------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| L1-03 무효 계정 | "미등재 계정"으로 심은 `999999`가 **실제 계정과목표에 등재된 계정**이었다                   | 미등재 코드를 상수로 박지 말고 런타임에 계정과목표를 읽어 선택 |
| L2-03 중복 전표 | 중복 키가 전기 **타임스탬프**(시각 포함)까지 비교하고, 거래처가 빈 행은 모집단에서 제외된다 | 동일 시각·동일 거래처로 교정                                   |
| L3-12 업무범위  | 점수 비병합 검토 신호 전용 룰이라 점수 채널이 항상 0 — 판정기가 한쪽 채널만 읽었다          | 판정기가 두 채널 모두 읽도록 보강                              |

세 번째가 특히 교훈적이다. **점수 0이 침묵을 의미하지 않는다.** 이 룰은 설계상 점수를 싣지 않고 검토 채널에만 신호를 보내므로, 점수만 보는 측정기에는 영원히 0으로 보인다. 측정 도구가 제품 구조를 모르면 멀쩡한 룰을 고장으로 판정한다.

L2-03에서는 부수적 발견도 있었다. 룰이 배경 데이터의 자연 중복 4문서를 이미 잡고 있었다 — 주입 전표를 못 잡은 것이지 룰이 죽은 게 아니라는 검출력 증거였다.

## 6.8 실증 예시 — 승인 생략 룰의 22.73%가 정상인 이유

2.3절에서 추적한 전표(`ef8bb035-…`)는 `approved_by`가 공란이라 L1-07에 발화한다(같은 전표에 L3-04·L4-03도 걸린다). 이 룰의 정상 발화율 22.7319%는 얼핏 과탐으로 보인다.

스펙은 이 동작을 알면서 고정했다. **승인자가 공란이면 시스템·반복 전표를 포함해 무조건 표기**하되, 단독으로는 검토 목록을 이루지 못하게 한다(조합 빌더의 "특징" 축이라 몸통과 결합돼야 목록이 된다 — 등급 자동 판정은 2026-07-17 폐지). 근거는 데이터에 있다. 이 발화의 대부분은 자동화 비중의 산물이고, 수기·조정 전표 중 승인 공란은 212행(0.06%)에 불과하며, **1천만원 초과 수기 전표의 승인률은 100%**다. 즉 사람이 손으로 넣은 큰 금액은 전부 승인을 받았고, 공란은 자동 처리에서 나온 것이다.

여기서 두 가지 선택지가 있었다. 자동 전표를 룰 대상에서 빼서 발화율을 0.06%로 낮추거나, 전부 표기하되 조합 빌더에서 몸통과 결합될 때만 검토 목록이 되게 하거나. 후자를 택한 이유는 **"승인 기록이 없다"는 사실 자체가 감사인이 알아야 할 정보**이기 때문이다. 자동 전표라서 승인이 없는 것이 정상이라는 판단은 도구가 아니라 감사인이 해야 한다. 다만 그 판단을 8만 건의 목록을 다 보면서 하게 만들면 안 되므로, 조합 빌더에서 이 룰은 "특징" 축에 놓이고 **몸통(조작 대상 계정·거래) 없이는 유효한 검토 목록이 되지 않게** 설계했다([7장](7_PHASE1-1_COMBO-BUILDER.md)).

이 사례가 발화율 대역 판정의 존재 이유다. 22.7319%는 사전 선언 대역 [15%, 30%] 안이라 통과였다. 만약 60%가 나왔다면 룰을 고치기 전에 **데이터의 자동화 비중이 비현실적인지부터** 조사했을 것이다.
