감사인이 전표 100만 건에서 "먼저 볼 것"을 고르는 일을, 등급을 대신 선언하지 않고 **근거 있는 어휘와 정확한 집계**로 돕는 로컬 전용 도구다. 이 보고서는 그 도구가 무엇을 검증했고 **무엇에 실패했는지**를 같은 비중으로 기록한다.

## 이 보고서를 신뢰할 수 있는 이유

집필 근거를 사람의 기억이 아니라 기계가 고정했다. 분모 N은 `census.py inventory`가 프로젝트 폴더 전체를 순회해 확정했고(11,030), 131개 묶음으로 나눠 전수 정독한 뒤, 마지막에 `census.py verify`가 커버리지 표와 인벤토리의 차집합이 0임을 확인했다. 수치는 팩트시트 한 곳에서만 인용했고, "무엇이 무엇을 호출한다"는 동작 서술은 문서의 자기소개가 아니라 배선 grep으로 확인한 것만 현재형으로 적었다.

그리고 집필이 끝난 뒤, **집필에 관여하지 않은 감사관이 16개 장 전부의 주장을 코드와 대조**했다. 두 차례에 걸쳐 **836개 주장**을 검증해 치명적 불일치 **15건**을 찾았고 전부 교정했다. 대표적인 것들이다.

- 발화율 표 28행 중 19행이 **이전 세대(s9) 수치**인데 최신 세대로 표기돼 있었다 → 결과 파일에서 다시 뽑아 분모까지 교체
- 회계 검증을 "게이트가 아니다"라고 썼으나 실제로는 **금액비율 1%·전표비중 10%를 넘으면 탐지를 차단**하고 있었다
- 통계 검증이 **파이프라인에 연결돼 있지 않은데** 점수에 반영된다고 썼다
- 예시 전표가 "룰 하나만 발화한다"고 썼으나 실측은 **세 룰 발화**였고, 그중 하나가 6장에서 실패로 판정한 결함의 실물이었다
- 학습 입력 차단 목록의 개수 네 항목이 전부 실제 상수 크기와 달랐다(17→18, 14→15, 24→30, 6→7)
- 시나리오 수 19종 → 실측 **20종**
- 관계사 게이트의 원인을 프로젝트가 이미 **기각한 가설**로 적어 두었다

교정 내역은 각 장에 반영돼 있다. 감사가 잡아낸 오류의 다수가 "코드는 있는데 실행 경로에 없는 것을 동작한다고 쓴" 유형이었다 — 이 보고서가 스스로 강조하는 **존재 ≠ 배선 ≠ 동작** 원칙을 집필 과정에서 어긴 대목들이고, 그래서 감사 단계를 별도로 둔 값어치가 여기서 나왔다.
