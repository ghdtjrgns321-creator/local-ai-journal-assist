# 5. 수집·파생변수·검증

## 5.1 파이프라인 내 위치

```
생성 → 품질검증 → ▶▶▶ [수집 → 파생 → 검증] ◀◀◀ → 탐지 → 표면 → 적재 → 화면
```

감사인이 받는 원장은 ERP마다 컬럼명도 형식도 다르다. 이 구간은 그 잡다한 입력을 탐지기가 기대하는 표준 형태로 바꾸고, 바꾼 결과가 쓸 만한지 판정한다.

## 5.2 내부 흐름

```
파일 ──▶ file_validator ──▶ reader ──▶ header_detector ──▶ column_mapper ──▶ type_caster
         크기·형식 검사     확장자별    데이터 시작 행     유사도 매핑      금액·날짜 정제
              │                                                │
              │ PDF·HWP 거부                                   │ 3구간 분류
              ▼                                                ▼
         (사유 명시)                            매핑 ≥80 / 제안 40~80 / 미매핑 <40
                                                               │
                                                    감사인이 제안 구간을 확정
                                                               ▼
                                          FeatureEngine ──▶ 금액·시간·패턴·텍스트
                                                               ▼
                                       ┌─── L1 구조 (Pandera) ────┐
                                       │   실패 → 탐지 진입 차단   │
                                       └───────────┬──────────────┘
                                                   ▼
                                       L2 회계 (2차 게이트)
                                    금액비율>1% 또는 전표비중>10% → 차단
                                       그 외는 경고로 통과
```

## 5.3 수집

| 단계        | 모듈                 | 핵심 규칙                                                                                        |
| ----------- | -------------------- | ------------------------------------------------------------------------------------------------ |
| 파일 검증   | `file_validator.py`  | Excel 100MB / 텍스트 800MB / Parquet 1000MB 상한. PDF·HWP는 **사유를 명시하고 거부**             |
| 읽기        | `reader_api.py`      | 텍스트는 10MB 초과 시 5만 행 청크 처리, 인코딩은 64KB 샘플로 추정                                |
| 헤더 탐지   | `header_detector.py` | 신뢰도 = 타입다양성 0.35 + 고유성 0.25 + 결측밀도 0.15 + 키워드 0.15 + 문자열비율 0.10           |
| 컬럼 매핑   | `column_mapper.py`   | 정확 일치(100점) → 유사도(rapidfuzz + 타입 호환 매트릭스) → 데이터 패턴(70~85) → 탐욕적 1:1 배정 |
| 타입 캐스팅 | `type_caster.py`     | 금액 7단계 정제, 날짜 5단계 폴백, 차대변 통합 3가지 경우                                         |

헤더 탐지에서 **키워드가 하나도 없으면 신뢰도 임계를 0.7로 올린다.** 낯선 ERP라 아는 단어가 없을 때는 구조적 근거만으로 판단해야 하므로 기준을 높이는 것이다. 기준을 낮춰 억지로 찾으면 데이터 행을 헤더로 오인한다.

컬럼 매핑 결과를 **매핑·제안·미매핑 3구간**으로 나누는 것도 같은 사고방식이다. 확실한 것만 자동 확정하고, 애매한 구간(40~80점)은 감사인에게 판단 근거와 함께 넘긴다. 매핑을 100% 자동화하면 조용히 틀린 컬럼으로 탐지가 돌아간다.

차대변 통합의 경우 C(부호로 추정)는 **경고를 필수로 동반한다.** 금액 부호로 차변·대변을 추정하는 것은 관행이지만 회사마다 다르므로, 추정했다는 사실이 결과에 남아야 한다.

## 5.4 파생변수

| 모듈                  | 개수 | 대상 룰                                  | 핵심 임계                                                                    |
| --------------------- | ---: | ---------------------------------------- | ---------------------------------------------------------------------------- |
| `amount_features.py`  |    6 | L2-01, L1-04, L4-03, L4-01, L2-02        | Z-score 3단 폴백 — 그룹 n≥30이면 그룹, 아니면 상위 계정군, 전체 n<10이면 NaN |
| `time_features.py`    |    7 | L3-05, L3-06, L3-04, L3-07, L1-08, L4-05 | 정상 근무 08:30–18:30, 심야 22:00–06:00, 결산집중 12/20~1/15                 |
| `pattern_features.py` |    5 | L3-02, L3-03, L4-01, L4-02, L2-04        | 가계정 판별은 계정과목표 플래그 우선, 없을 때만 키워드                       |
| `text_features.py`    |   3+ | NLP01~05(미배선) · L3-02 보조 주석       | 형태소 태그 명사·동사·형용사                                                 |

**Z-score 3단 폴백**이 이 구간의 설계 태도를 보여준다. 계정별로 금액 이상치를 보려면 그 계정의 표본이 충분해야 하는데, 거래가 드문 계정은 표본이 없다. 이때 전체 평균으로 억지 계산하면 소액 계정의 정상 거래가 이상치로 뜬다. 그래서 그룹(30건 이상) → 상위 계정군 → 전체 순으로 내려가고, 전체마저 10건 미만이면 **계산하지 않고 NaN을 남긴다.** 모르는 것을 0으로 채우지 않는다.

시간 축에는 보정 규칙이 있다. 결산 집중기(12월 20일~1월 15일)의 야근은 `overtime`이 아니라 `normal`로 낮추고, 주말·공휴일 근무는 `normal`이라도 `overtime`으로 올린다. 결산기 야근은 한국 회계 실무에서 정상이고, 주말 전기는 평일 야근보다 이례적이라는 도메인 판단이다.

### 라운드넘버 정의의 재설계

`is_round_number`의 구 정의는 "금액이 100만원의 배수"였다. 그런데 원장 금액이 1,526원부터 2,057억원까지 8자릿수에 걸쳐 있어 단일 절대 단위로는 잴 수 없었다. 실측 결과 **거래의 64.9%가 100만원 미만**이라 구조적으로 판정 대상에서 빠졌고, 정상·부정 양쪽 38만 행 중 round로 잡힌 것이 141행(0.037%)뿐이라 밀집도 검사기가 **어느 데이터에서도 발화하지 못했다.**

현행 정의는 금액 규모에 상대적이다: 소수부가 없고, 끝자리 0을 제거한 유효숫자가 2자리 이하이며, 총 자릿수가 3 이상. 이 기준의 실측 기저율은 5.16%다(유효숫자 1자리는 2.46%로 너무 희박, 3자리는 14.49%로 과다).

## 5.5 3계층 검증

| 계층        | 판정                                                                        | 게이트인가                        |
| ----------- | --------------------------------------------------------------------------- | --------------------------------- |
| **L1 구조** | Pandera 스키마 — 필수 10컬럼, 회계연도 2000~2099, 회계기간 1~12, 차·대변 ≥0 | **예 — 실패 시 탐지 진입 차단**   |
| L2 회계     | 전표별 차대 차액 (허용오차 0.01), 영업일 연속성                             | **부분 게이트** — 아래 설명       |
| L3 통계     | Benford MAD 4단계, 변동성, 시계열 통계                                      | **해당 없음 — 파이프라인 미배선** |

L1이 1차 게이트인 이유는 명확하다. 필수 컬럼이 없거나 타입이 깨진 데이터로 탐지를 돌리면 결과가 나오긴 하지만 전부 무의미하다.

L2는 **비율 기반 2차 게이트**다. 차대 불일치가 나와도 단일 행 문제라면 멈추지 않지만, 전체 차변 대비 차액 비율이 1%를 넘거나 불일치 전표 비중이 10%를 넘으면 예외를 던져 탐지 진입을 차단한다. 복식부기가 근본적으로 깨진 원장은 어떤 룰을 돌려도 의미가 없되, 사소한 불일치로 감사를 멈추게 하지는 않는다는 절충이다.

L3 통계 검증은 **구현돼 있으나 현행 파이프라인에 연결돼 있지 않다.** 통계 검증 함수는 패키지 export 외에 호출부가 없고, 검증 리포트의 통계 항목은 빈 목록으로 고정돼 있으며 점수 산식에도 L3 항이 없다. Benford 계열 신호는 별도로 PHASE1-2 자기 큐가 담당한다([8장](8_PHASE1-2_ANALYTICAL.md)).

L1 안에서도 구분이 있다. 필수 컬럼의 타입·존재·not-null 위반만 치명이고, 값 범위 위반(회계연도가 1999)은 경고다. 실패 정책도 세분화돼 있어, not-null 오류만 있으면 해당 행을 버리고 진행하되(전량이 버려지면 예외), 다른 오류가 섞이면 즉시 중단한다.

검증 점수 산식도 정의돼 있다 — 100점에서 L1 치명 −50 일괄, L1 경고비율×20, L2 불일치 전표비율×15, 날짜 불연속 −5, 중복비율×10. 코드 주석이 설계 이유를 남겼다: "오류 수 × 고정 가중치는 음수가 될 수 있어 비율 기반으로 상한을 보장한다". 다만 이 점수를 만드는 리포트 생성 함수는 **현재 파이프라인에서 호출되지 않으며** 테스트에서만 실행된다.

Benford 판정 자체는 **MAD 기준이고 KS 검정은 참고로만** 쓴다. 첫자리 분포는 이산분포라 KS 검정이 부적합하기 때문이다. MAD 임계 0.006/0.012/0.015는 포렌식 회계의 사실상 표준(Nigrini 2012)을 따랐다.

## 5.6 실증 예시 — 원장 파일이 표준 형태가 되기까지

정상 base(`datasynth_semantic_v1_normal_s10_c001_20260717`)의 `journal_entries.csv`(203.9MB, 73컬럼)를 넣었을 때를 따라간다.

**파일 검증**에서 CSV는 텍스트 계열이라 800MB 상한 안이므로 통과한다. **읽기**는 10MB를 훨씬 넘으므로 5만 행 청크 모드로 들어가고, 앞 64KB로 인코딩을 UTF-8로 판정한다.

**헤더 탐지**는 첫 행에서 `document_id`, `posting_date`, `debit_amount` 같은 알려진 키워드를 다수 발견한다. 키워드 점수가 높으므로 임계 상향 규칙(0.7)은 적용되지 않고, 타입 다양성과 고유성까지 합산해 1행이 헤더로 확정된다.

**컬럼 매핑**에서 이 데이터는 이미 표준 컬럼명을 쓰므로 대부분 정확 일치로 100점 배정된다. 필수 10컬럼이 전부 일치하므로 빠른 경로를 타는데, 이 경로는 스키마에 정의되지 않은 컬럼을 전부 미매핑으로 분류한다 — 스키마 51컬럼 대비 원장 73컬럼이라 배치 ID·변형 이력 같은 생성기 메타 26개가 미매핑으로 남는다. 실제 감사 현장에서 "적요"·"차변금액" 같은 한글 헤더가 들어올 때가 유사도 단계가 일하는 경우다.

**타입 캐스팅**에서 `debit_amount = 45987521`이 숫자로, `posting_date = 2023-10-04 11:22:56`이 타임스탬프로 변환된다. 차대변이 이미 분리된 컬럼이므로 통합 로직 중 경우 A가 적용되고 부호 추정 경고는 뜨지 않는다.

**파생 단계**에서 이 전표에 붙는 값들: 계정 6500 그룹의 금액 Z-score(그룹 표본이 충분해 그룹 기준 적용), 시간대 분류 `normal`(11시 22분), 주말·공휴일 플래그 false, 전기일-증빙일 갭 0일, 가계정 플래그 false(계정과목표 조회 결과), 라운드넘버 false(45,987,521은 유효숫자가 8자리).

**검증 단계**에서 필수 컬럼이 모두 있고 금액이 음수가 아니므로 L1 스키마 검증을 통과한다. 전표 단위로 차변 합과 대변 합이 일치하므로 L2의 치명 조건에도 걸리지 않는다. 이제 이 전표는 탐지기로 넘어갈 자격을 얻었다.

여기서 파생된 값들이 [6장](6_PHASE1-1_RULES.md)의 룰 판정 입력이 된다. 시간대가 `normal`이므로 심야 룰이 침묵하고, 갭이 0일이므로 소급기표 룰이 침묵하는 식이다. **룰이 침묵하는 것도 파생 단계가 만든 결과**라는 점이 중요하다 — 파생이 잘못되면 룰은 조용히 틀린 답을 낸다.
