# 9. PHASE2 — 비지도 VAE와 검증 실패의 기록

## 9.1 파이프라인 내 위치

```
                              ┌─ PHASE1-1 룰
검증 통과 DataFrame ──────────┼─ PHASE1-2 분석신호
                              └─ ▶▶▶ [PHASE2 VAE — 별도 실행·별도 표시] ◀◀◀
```

정상 데이터만으로 "정상의 모양"을 학습한 뒤, 복원이 잘 안 되는 행을 추가 검토 후보로 올리는 표면이다. **이 장은 실패를 기록한다.**

## 9.2 왜 비지도인가

합성데이터에서 지도학습은 순환 논증이다. 부정 라벨은 우리가 규칙으로 심은 것이므로 모델이 학습하는 것은 부정이 아니라 **우리의 주입 규칙**이다. 치트루트 감사가 이를 수치로 확인했다([4장](4_DATASYNTH-QUALITY.md)) — 단일 컬럼만으로 AUROC 0.95 이상이 18건, 시뮬레이션 로지스틱 평균 0.9978.

비지도는 라벨을 쓰지 않으므로 이 함정을 피한다. 정상 분포만 학습하고 그 밖을 표시하는 방식이라 합성데이터에서도 유효할 것으로 봤다. **결과적으로 이 기대는 검증되지 않았다.**

## 9.3 라벨 누수 차단 — 3중 방어

모델이 답을 훔쳐보지 못하게 막는 장치가 세 겹이다.

**1차: 컬럼 deny 목록**(`src/preprocessing/constants.py`)

| 상수                        |                                   개수 | 근거                                                             |
| --------------------------- | -------------------------------------: | ---------------------------------------------------------------- |
| `LABEL_COLUMNS` | 8 | is_fraud, fraud_type, is_anomaly, anomaly_type, sod_violation 등 |
| `SYNTHETIC_ONLY_COLUMNS` | 18 | 위 8개 + mutation 계열 6, 합성 플래그 2, 시나리오 ID, 탐지 힌트 |
| `LEAKAGE_DENY_COLUMNS_BASE` | 15 | 단일 컬럼 AUROC ≥ 0.95 |
| `LEAKAGE_DENY_COLUMNS_V6_BASELINE` | 30 | 치트루트 감사 v6 — 금액 10 · 승인/시점 9 · 시나리오 특화 11 |
| `LEAKAGE_DENY_COLUMNS_V7_DERIVED` | 7 | 치트루트 감사 v7 — 승인권·한도·한도근접 파생 |
| **`LEAKAGE_DENY_COLUMNS`(최종)** | **52** | 위 합집합 — **누적, 해제 없음** |
| `LEAKAGE_DENY_RULES` | 5 | 룰 단위 차단 |

개수는 본 보고서 작성 시 모듈을 직접 로드해 실측했다. 코드 주석이 정량 근거를 남겼다: 24차원 AUPRC 0.397 → 상위 5개 차단 후 0.056(잔존 14.0%).

**2차: 패턴 기반 차단**(`phase2_plan.py`) — label, target, fraud, anomaly, risk, rule, score, model, prediction, probability, export, dashboard 등 11토큰과 단계 출력 접두어를 정규식으로 막는다. 트레이드오프도 기록돼 있다 — "priority 같은 넓은 토큰을 추가하면 payment_priority 등 정상 ERP 필드까지 막힌다".

**3차: 물리적 제거**(`schema_validator.py`) — 8개 컬럼을 아예 DataFrame에서 **drop**한다. "신규 룰이 실수로 참조해도 즉시 실패하게" 만드는 것이 목적이며, 테스트가 AST 수준으로 탐지 코드의 참조 여부까지 회귀 검사한다.

**금액 컬럼은 전량 차단**돼 있다. 사유는 "합성 금액이 너무 깨끗해 지름길 학습을 유발"이다.

## 9.4 평가 설계의 방어 장치

| 장치               | 내용                                                                                                        |
| ------------------ | ----------------------------------------------------------------------------------------------------------- |
| 행 단위 KFold 금지 | 같은 전표의 행이 학습·평가에 나뉘어 들어가는 것을 원천 차단                                                 |
| 분할 누출 게이트   | 무작위 분할과 그룹 분할의 AUC 차이 > 0.05면 사용자 수준 누출 확정, 그룹-시간 차이 > 0.03이면 시간 누출 확정 |
| 순환 라벨 차단     | 라벨 출처가 탐지 점수나 의사 라벨이면 **강제 부적격**                                                       |
| 저신호 폴백        | 양성 50건 미만 또는 비율 1% 미만이면 학습 중단                                                              |
| 승격 게이트        | 9단계(완료 상태·심각 경고 없음·최소 시행·탐색 다양성·실패율 0.5 이하·지표 하한 등)                          |
| 추론 독립성        | PHASE1의 케이스 우선순위·룰 발화는 ML **입력도 게이트도 아니다**. PHASE1은 추론 *후* 덧붙이기에만 사용      |

마지막 항목이 3-surface 비병합의 코드 수준 구현이다. VAE가 PHASE1 결과를 보고 점수를 매기면 두 표면이 독립이 아니게 된다.

## 9.5 학습 구성과 1차 관문 통과

정상 base에서 학습하고, 미학습 정상 구간의 플래그율이 오염 설정 근처인지 먼저 확인했다.

| 항목                     | 값                           |
| ------------------------ | ---------------------------- |
| 학습 표본                | 50,000행 (라벨 미사용)       |
| 보정 표본                | 50,000행                     |
| 피처 수                  | **71**                       |
| 은닉 차원 / 잠재 차원    | 64 / 32                      |
| epoch / batch / 학습률   | 50 / 256(196배치) / 0.001    |
| beta                     | 1.0                          |
| 재구성 손실 / KL 손실    | 0.2300 / 0.3718 (비율 1.616) |
| 임계                     | 0.9900                       |
| 학습 소요                | 460.9초                      |
| 학습 구간 플래그율       | 0.998%                       |
| **미학습 정상 플래그율** | **1.056%**                   |
| 오염 설정                | 1.00%                        |

사전 선언 대역 [0.3배, 3배] 안이므로 **PASS**다. 모델이 정상을 정상으로 보고 있다는 뜻이다.

부수적으로 제품 버그도 하나 잡았다. 결측이 있는 불리언 피처가 전처리 통과 경로에서 numpy 변환에 실패해 터졌다. 불리언 전용 변환기를 신설해 결측을 부동소수 NaN으로 바꾼 뒤 최빈값으로 대치하도록 고쳤다(수치형이 중앙값 대치를 쓰는 것과 같은 결).

## 9.6 2차 관문 — 실패

부정 overlay 4벌의 전 행을 채점하고 문서 단위로 평가했다.

| 지표 (시드 4벌 합산)                                       | 값                        |
| ---------------------------------------------------------- | ------------------------- |
| 문서 AUROC 평균                                            | **0.5187** (무작위 ≈ 0.5) |
| recall@top1% 평균                                          | **0.0038**                |

**판정: 행 단위 VAE 검증 = 실패.** 수치를 완화해 서술하지 않는다.

## 9.7 실패의 원인 분해

이 절이 측정의 실제 산출물이다.

### 모델 결함이 아니라 정보론적 귀결

본 overlay는 지름길 게이트 16종으로 **부정 행의 개별 피처를 전부 정상과 동일하게 강제한 "완전 위장" 시뮬레이션**이다. 부정 전용 계정도, 전용 속성값도, 시나리오를 드러내는 적요도 없다. 행 단위 피처에 정상과의 구분 정보가 **존재하지 않으므로**, 행 단위 비지도 모델이 무반응인 것은 어떤 모델을 써도 피할 수 없다. 시드 4벌에서 일관되게 재현됐다.

### 합성 경로로는 검증 자체가 불가능하다 — 딜레마

```
    흔적을 심는다  ──▶  자기가 심은 것을 자기가 찾음 (자문자답)
                          ↓ 성능 증거 아님
    흔적을 안 심는다 ──▶  정보가 없어 못 잡음 (본 실측)
                          ↓ 성능 증거 아님
```

어느 쪽도 성능 증거가 되지 못한다. 이 딜레마의 인정이 이번 라운드의 결론이다.

실제로 구세대 데이터에서 얻었던 **AUROC 0.93은 흔적을 심은 자문자답 수치였음을 명시적으로 무효화**했다. 좋은 숫자를 폐기하는 결정이었다.

외부 벤치마크(Tritscher ERP-Fraud, 문서 AUROC 0.867~0.999)는 자문자답은 아니지만 홀드아웃당 부정 표본이 6문서에 불과한 시뮬레이션이라 "코드가 동작한다" 이상의 의미를 부여하지 않기로 했다.

### 파생 결정 — 금액 차단 유지

금액 컬럼 차단을 풀지 여부가 열린 질문이었다. **유지로 결정**했다. overlay 금액 설계가 "정상 분포 내 은폐"로 확정된 이상, 금액 피처를 풀어도 부정 신호에 기여할 정보가 없고(원시 금액이 입력에 있던 구성에서도 무반응이었다) **고액 정상 전표 과탐 축만 추가**되기 때문이다.

## 9.8 실패가 남긴 것

**표면의 역할을 축소했다.** VAE는 "학습된 정상 밖 비정형의 추가 검토 후보 제시"라는 보조 역할로만 유지하며, **부정 탐지 성능을 주장하는 표면으로 쓰지 않는다.**

**업계 실무와의 정합성을 확인했다.** "비지도 딥러닝이 실패했으니 구조가 틀렸나"를 확인하려고 빅4 도구를 조사했다.

| 도구           | 방식                                        |
| -------------- | ------------------------------------------- |
| EY Helix GLAD  | 감사인 라벨 기반 지도학습                   |
| KPMG Clara     | MindBridge 앙상블 — 룰 40여 종 위에 ML 보조 |
| PwC Halo       | 룰 묶음 + 의심 점수                         |
| Deloitte Omnia | 기준 기반 분석                              |

네 도구 모두 룰·기준이 주력이고 ML은 보조이며, **비지도 딥러닝 단독 탐지는 실무에 없다.** 본 프로젝트의 구조(결정론 룰이 주 표면, VAE는 병합하지 않는 보조)는 이 관행 안에 있다. 실패한 검증이 도구를 표준에서 벗어나게 만든 것이 아니라, 실무가 왜 그렇게 생겼는지를 실측으로 재확인한 셈이다.

**측정 도구는 유지했다.** 평가 방어 장치(그룹 분할 강제, 누출 게이트, 순환 라벨 차단)는 실데이터가 유입되면 그대로 쓸 수 있다. 실패한 것은 "합성데이터로 비지도 성능을 증명하려는 시도"이지 평가 인프라가 아니다.

## 9.9 실증 예시 — 완전 위장 전표가 VAE를 통과하는 이유

FS08(부적절 자본화) scheme의 부정 전표 한 장을 생각해 보자. 비용으로 처리해야 할 지출을 자산으로 올리는 조작이다.

이 전표의 행 단위 피처를 나열하면: 금액은 정상 분포 안, 계정은 정상이 많이 쓰는 22개 중 하나, 계정 subtype은 base 실측 사전에서 조회한 실재 값, 작성자는 실재 직원, 시각은 근무시간, 소스 분포는 정상과 섞임, 적요는 고빈도 정상 헤더로 정규화됨. **모든 개별 축에서 정상과 구분되지 않는다.**

VAE는 이 행을 입력받아 재구성한다. 재구성 오차가 작다 — 정상 분포 안의 값들이므로 잘 복원된다. 따라서 이상 점수가 낮고 상위 1%에 들지 못한다. 모델이 잘못한 것이 아니라 **입력에 신호가 없다.**

같은 전표를 PHASE1 룰은 잡는다. 비용자산화 룰이 "비용 성격 지출이 자산 계정에 기표됐다"는 **관계**를 보기 때문이다. 이것은 행의 값 분포가 아니라 계정 간 의미론이고, 도메인 지식으로 명시한 규칙만이 볼 수 있다.

여기서 이 프로젝트의 결론이 나온다. 부정은 개별 값의 이상함이 아니라 **맥락의 부적절함**으로 나타나고, 그 맥락을 아는 것은 학습이 아니라 회계 지식이다. 비지도 모델이 잡을 수 있는 것은 "값 자체가 이상한 것"인데, 실제 부정은 값을 정상처럼 만드는 데 공을 들인다. 그것이 은폐의 정의이기 때문이다.

## 9.10 실세계는 이것을 어떻게 하나 — 그리고 왜 이 실패가 구조적인가

행 단위 VAE의 실패를 "데이터가 부족했다" 또는 "튜닝이 모자랐다"로 읽으면 오독이다. 실세계 금융 이상탐지의 작동 방식과 대조하면 이 실패가 **구조적 귀결**임이 드러난다.

핵심 사실: **실무는 원시 전표 한 줄을 모델에 넣지 않는다.** 신호는 원시 행이 아니라 가공한 피처와 맥락에 있다.

| 도메인             | ML/DL이 되는가             | 이유                                                                                                     |
| ------------------ | -------------------------- | -------------------------------------------------------------------------------------------------------- |
| 신용카드 부정      | **잘 된다**                | "카드 30일 평균 대비 금액", "직전 거래와의 거리·시간", 가맹점 위험도 등 **행동·속도 피처** + **대량 실라벨** |
| 자금세탁(AML)      | 보조적                     | **규칙 + 관계망(그래프) + 사람 리뷰**가 주력. 라벨 희소·불확실, ML은 랭킹 보조                            |
| 재무제표·전표 부정 | **가장 어렵다**            | 부정이 정상처럼 위장돼 원시 행엔 신호가 없고, 집계 분석·교차대사·원장 밖 맥락에서만 신호가 나온다          |

신용카드가 되는 이유는 딥러닝이 마법이어서가 아니라, **신호를 담은 행동·속도 피처를 사람이 설계**하고 **막대한 실제 라벨**로 지도학습하기 때문이다. 전표 부정에는 그 세 조건(가공 피처·실라벨·원시 행 신호)이 모두 약하다.

그리고 이 프로젝트는 [1장](1_OVERVIEW.md)·의사결정 ①에서 **단일 법인 원장으로 스코프를 확정하며 거래처 마스터·은행·관계망·외부 대사를 데이터에서 제외**했다. 실무 부정탐지가 기대는 관계·집계·외부 신호가 데이터에 **원리적으로 부재**하도록 스스로 선을 그은 것이다(그 선은 감사 계약이 법인 단위라는 현실에 근거한 옳은 결정이었다). 여기에 합성 라벨의 자문자답 딜레마([9.7](#97-실패의-원인-분해))가 겹친다.

따라서 세 조건이 곱해진다: **가장 어려운 도메인 × 업계도 안 쓰는 방법(원시 행 비지도) × 신호가 사는 데이터를 스코프에서 제외.** 행 단위 VAE의 무반응은 이 셋의 곱이며, 어느 모델을 써도 피할 수 없다.

역으로, 업계가 실제로 쓰는 방식 — **도메인 지식을 규칙으로 명시하고 그 위에 위험 점수를 랭킹** — 은 이 프로젝트의 [PHASE1 룰 엔진](6_PHASE1-1_RULES.md)과 [조합 빌더](7_PHASE1-1_COMBO-BUILDER.md)가 이미 구현한 것이다. 즉 "되는 쪽"은 만들었고, "안 되는 쪽"은 안 되는 이유를 원리로 규명해 기록했다. 후자를 "된다"고 포장하지 않은 것이 이 프로젝트의 규율이다.
