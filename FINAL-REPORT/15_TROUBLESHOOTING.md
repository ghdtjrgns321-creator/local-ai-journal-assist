# 15. 운영 사고와 디버깅 교훈

> **과거형 전용.** 디버깅 기록 8,177줄과 트러블슈팅 문서를 전수 정독해 **사고 91건**(상세 43 + 경미 48)을 추출했고, 그중 피해 규모와 교훈 가치 기준으로 대표 7건을 상술한다.

## 15.2 대표 사고 흐름 — 설정 12섹션 유실

```
[의도] 정상 데이터 생성 설정을 조금 바꾸자
   │
   ▼
[행동] 설정 YAML을 손으로 새로 작성          ← 여기가 분기점
   │      정본은 17섹션, 손작성본은 5섹션
   │
   ▼
[침묵] 12섹션 누락  →  에러 없음
   │      전 필드가 기본값 허용(serde default)
   │      금액 섹션 누락 → USD 기본값 적용
   │      lognormal_mu: 14.0(KRW) → 7.0(USD)
   ▼
[확산] 98만 행 생성 완료, 게이트 일부 통과
   │      매출 중앙값 1,000원 (정본 681,151원)
   │      → e⁷ ≈ 1,097배, 실측 1,223배 차이
   ▼
[발견] 금액 분포가 이상하다는 육안 확인
   │
   ▼
[복구] 정본 설정 파일로 재생성
   │      Rust 코드 수정 0줄 — 코드는 처음부터 옳았다
   ▼
[방지] 규칙 승격: "설정은 정본을 읽어 최소 변경.
        섹션 수를 전후로 확인한다"
```

이 사고가 두 번째였다는 점이 중요하다. 첫 번째는 세금 섹션 누락으로 생성 단계 하나가 통째로 건너뛰어진 사건이었다. 같은 병이 재발했다는 것은 개인의 부주의가 아니라 **설계가 그 실수를 유도한다**는 뜻이다. 설정 스키마가 누락을 에러로 만들지 않는 한 세 번째도 일어난다.

## 15.4 반복된 실패 패턴

91건을 관통하는 패턴이 다섯 가지다.

**① 조용한 기본값.** 누락이 에러가 아니라 기본값으로 채워지는 구조가 최대 피해 사고 두 건을 만들었다. 설정 스키마의 관대함이 편의가 아니라 위험이었다.

**② 스테일 산출물.** 빌드가 no-op이거나 캐시가 남아 **고쳤는데 안 고쳐진 것으로 보이는** 상황이 반복됐다. 교차검증 한 달 사고에서만 5회 낭비됐다.

**③ 측정 도구의 결함.** 측정 하니스가 제품과 다른 경로를 타서 결론이 뒤집혔고, 게이트 4종은 항등식이었으며, 룰 시험 장치가 멀쩡한 룰 3개를 고장으로 판정했다. **검증 인프라도 검증 대상**이라는 결론이 여기서 나왔다.

**④ 완료 선언의 조기화.** 배선이 비었는데 완료로 선언되거나, 11일 만에 폐기될 것이 완료로 기록됐다. "종료 코드로만 완료를 선언한다"는 규칙의 배경이다.

**⑤ 지름길의 예상 밖 출처.** 문서 ID 길이, 소스 필드 분포, 시나리오명이 적요에 노출되는 것 등 **의미 없어 보이는 필드**에서 지름길이 생겼다. 지름길 게이트 16종은 이 경험의 산물이다.

## 15.6 실증 예시 — 룰이 고장 났다고 오판할 뻔한 순간

룰 단위시험 1차 판정에서 세 룰이 자기 표적 전표에서 발화하지 않았다. 26/29였다.

가장 자연스러운 다음 행동은 그 세 룰의 코드를 여는 것이다. 실제로 그렇게 했다면 무결한 코드를 들여다보며 시간을 썼을 것이고, 최악의 경우 "발화하게" 만들려고 조건을 느슨하게 고쳤을 것이다. 그러면 룰은 영원히 잘못된 상태가 된다.

대신 시험 장치를 먼저 의심했고, 세 건 모두 장치 결함이었다.

무효 계정 룰의 경우, 시험 데이터에 "미등재 계정"으로 `999999`를 심었는데 **그 코드가 실제 계정과목표에 등재돼 있었다.** 룰은 정확히 동작했다 — 등재된 계정이니 무효가 아니라고 판정한 것이다. 교정은 미등재 코드를 상수로 박지 말고 런타임에 계정과목표를 읽어 없는 코드를 고르는 것이었다.

업무범위 룰은 더 미묘했다. 이 룰은 설계상 점수를 싣지 않고 검토 채널에만 신호를 보낸다. 판정기가 점수 채널만 읽고 있었으므로 **영원히 0으로 보였다.** 룰도 정상, 데이터도 정상, 판정기만 절반을 못 보고 있었다.

교정 후 재판정은 29/29였고 **룰 결함은 0건**이었다. 만약 첫 판정을 그대로 믿었다면 이 프로젝트는 "룰 3개가 고장 났다"는 잘못된 사실을 안고 갔을 것이다.

이것이 대표 사고 4번(측정 하니스)과 같은 교훈이다. 도구가 제품을 잘못 재면, 제품을 고치라는 잘못된 신호가 나온다.
