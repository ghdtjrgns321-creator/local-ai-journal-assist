# 7. 조합 빌더 — 등급을 선언하지 않는 검토 표면

## 7.1 파이프라인 내 위치

```
생성 → 품질검증 → 수집·파생·검증 → 탐지·정규화 → ▶▶▶ [표면 구성] ◀◀◀ → 적재 → 화면
```

룰 발화를 감사인이 실제로 볼 목록으로 바꾸는 층이다. 이 프로젝트에서 가장 크게 뒤집힌 설계이기도 하다.

## 7.2 왜 자동 등급을 폐기했나

구 설계는 룰 조합에 HIGH/MEDIUM/LOW/CONTEXT 등급을 자동 부여했다. 예컨대 "기말 전표 + 수기 입력 + 승인 생략"이면 HIGH였다. 이 등급의 실증 근거를 검증하려고 금융감독원 감리 지적 사례를 엄격 기준으로 재태깅(731행)한 결과, **지지가 무너졌다.**

| 구 HIGH 조합    | 감리 사례 확정 건수 |
| --------------- | ------------------: |
| 기말 × 추정계정 |               12~22 |
| 나머지 전부     |                 0~3 |

원인은 자료의 장르 한계였다. 감리 문서는 "무엇이 조작됐나"(결과)는 쓰지만 **"전표를 어떻게 입력했나"(수기·승인·시각)는 쓰지 않는다.** 따라서 수단 신호가 낀 조합은 원문으로 영구히 실증할 수 없다. 남는 근거는 조합별 전문가 특례("보강신호 1개면 승격" 같은)뿐이었고, 조합마다 다른 사연은 방어 불가능한 휴리스틱이다.

합성데이터로 메울 수도 없었다. "정상에서 이 조합이 드물더라"는 우리가 생성기에 넣은 파라미터의 메아리이므로 등급 근거가 될 수 없다.

**결정: 등급 자동판정 전면 폐지.** 시스템이 등급을 선언하지 않으면 방어할 휴리스틱이 0이 된다. 조합 선택은 감사인의 판단이며, 감사기준서 240에서도 분개 테스트 항목 선정은 감사인 판단 사항이다.

코드에서 사라진 것들: 등급 4종과 우선순위 밴드 산출, 조합 바닥값 12종, 순위가능 primary 게이트, 조합별 특례 전부. `topic_scoring.py`는 662줄에서 **193줄**로 재작성됐다. 남은 LOW·CONTEXT 두 이름은 위험 등급이 아니라 **"단독으로 큐에 설 자격"을 판정하는 게이트**로만 기능하며, 조합 빌더의 정렬에는 쓰이지 않는다(정렬 키는 분류 점수·시간 심각도·금액·증거 수 4축). 이 게이트가 실제로 문서를 표면에서 배제한다는 사실은 빌더 판정 스크립트가 제외 건수를 별도 기록해 확인한다.

## 7.4 결합 의미론

```python
if strict:
    선택한_몸통 ∪ 선택한_특징  ⊆  이_전표에_발화한_룰       # 전부 발화
else:
    (몸통 미선택 or 몸통 ∩ 발화 ≠ ∅) and (특징 미선택 or 특징 ∩ 발화 ≠ ∅)
```

| 모드 | 조건                                                                    | 미선택 그룹 |
| ---- | ----------------------------------------------------------------------- | ----------- |
| 기본 | (몸통 중 1개 이상) **AND** (특징 중 1개 이상) — 그룹 내 OR, 그룹 간 AND | 무조건 통과 |
| 엄격 | 선택한 룰 **전부** 발화                                                 | —           |

미선택 그룹의 처리가 의도적이다. 특징만 고르면 "모양만 보는 탐색 모드", 몸통만 고르면 "대상만 보는 탐색 모드"가 되고, 둘 다 비면 결과가 없다. 감사인이 어느 축부터 접근하든 막지 않는다.

정렬은 (분류 점수, 시간 심각도, 금액, 증거 수) 순이며 **등급 축이 없다.** 금액은 마지막 동점 처리에만 쓴다 — 고액 정상 거래가 신호 있는 소액 전표를 목록에서 밀어내지 않게 하려는 것이다.

무결성 검증이 세 겹으로 걸려 있다. 몸통·특징 내 룰 중복, 몸통과 특징의 겹침, 프리셋이 어휘 밖 룰을 참조하는 경우 모두 즉시 예외를 던진다. 화면 버그를 조용한 오답이 아니라 크래시로 만든다.

엔진 docstring이 역할을 못 박는다 — "엔진은 전표·흐름 단위 위의 순수 조회다. **등급을 만들지 않는다(판단 주체 = 감사인).**"

## 7.5 프리셋 — 등급이 아니라 시작점

프리셋은 **수정 가능한 체크 상태**이며 위험 선언이 아니다.

| 프리셋           | 몸통                       | 특징                    | 소명                                          |
| ---------------- | -------------------------- | ----------------------- | --------------------------------------------- |
| 추정·관계사 조작 | 추정계정, 관계사           | 승인 계열 4종·수기·소급 | 감리 최다 적발 대상 2종 + 통제 무력화 모양    |
| 수익인식         | 매출과대, 컷오프, 기간귀속 | 전체                    | 기준서 240 문단 26 — 수익인식은 부정위험 추정 |
| 역분개·은폐      | 역분개, 가수금, 이상고액   | 전체                    | 기말 분식 후 되돌림·경과계정 은폐             |
| 비용자산화       | 비용자산화                 | 전체                    | 감리 22건 실증 수법                           |

## 7.6 표면 3분면

```
PHASE1-1 결과 화면
├── ① 조합 빌더        주 검토 표면 — 어휘 20룰 + 프리셋
├── ② 커버리지 큐      룰별 전수 + 드릴다운. 빌더 어휘 밖 활성 룰(L2-01·L2-02·L2-03·L3-12)은 여기서만
└── ③ 데이터정합성     L1-01~03 별도 — 부정 신호가 아니라 데이터 품질 게이트, 빌더 비산입
```

차대 불일치나 필수필드 누락은 **부정 징후가 아니라 데이터가 깨진 것**이므로 위험 목록에 섞으면 안 된다. 별도 패널로 분리해 "먼저 데이터를 고치라"는 신호로만 쓴다.

## 7.7 빌더가 정확한가 — 단위시험 (exit 0)

빌더가 "선택한 조합과 일치하는 전표"를 정확히 집계하는지, 룰 단위시험 데이터를 재사용해 **독립 재구현과 대조**했다.

| 판정 축           | 내용                                                                | 결과                     |
| ----------------- | ------------------------------------------------------------------- | ------------------------ |
| V0a 날조 검사     | 엔진이 제시한 증거가 오라클에 전부 실재하는가                       | **위반 0 / 36,774 unit** |
| V1·V2 결합 의미론 | 몸통10×특징10 + 단독20 = 120셀, 기본·엄격 두 모드 독립 대조         | 각 **120/120**           |
| V3 프리셋         | 저장된 조합 4종의 뷰 정합·절단 규칙                                 | **4/4**                  |
| V4 정답지 하한    | 어휘 20룰 단독 선택 시 자기 표적 문서 일치                          | **20/20**                |
| V0b 표면 커버리지 | 탐지기 발화가 표면에 실제로 오르는가 (합격선 없는 **실측 기록 축**) | 유실 잔존 — 아래         |

V0a가 특히 중요하다. 빌더가 "이 전표는 조건에 맞는다"고 하면서 근거로 드는 룰 발화가 실제로 존재하는지를 전수 확인한다. 화면이 그럴듯한 근거를 지어내면 감사 도구로서 실격이기 때문이다.

**V0b는 통과 기준이 없는 관찰 축이라 exit 0에 포함되지 않지만, 남은 유실을 여기 기록한다.** L2-05를 고친 뒤에도 여러 룰에서 발화가 표면에 오르지 못하는 사례가 남아 있다 — 기말 집중 436건(16,980 중), 심야 110건, 주말 94건, 이상고액 86건, 추정계정 6건, 가계정 5건, 컷오프·소급기표 각 1건으로 **합계 약 741건**이다. 원인은 집계 단계의 위험도 게이트 잔재로 추정되며(전수 규명 미완), 각 룰의 1~3% 수준이다. exit 0은 "빌더 집계가 정확하다"는 증명이지 "발화가 전부 표면에 오른다"는 증명이 아니다.

### 판정이 실제로 잡은 결함 2건

**L3-03 스테일 topic (1차 FAIL 원인).** 룰 메타데이터가 2026-06에 삭제된 topic을 참조하고 있어, 관계사 룰의 발화가 표면 진입 전 게이트에서 **전량 탈락**했다. 점수 레지스트리는 갱신됐는데 메타데이터만 누락된 편측 갱신이었고, 메타데이터가 우선 적용되는 구조라 치명적이었다. 구 등급 체계에서는 이 룰이 보조 신호라 증상이 안 보였고, 빌더 어휘의 몸통(감리 80건)으로 승격되면서 드러났다. 수정 후 증거 보유 unit이 **0 → 199**.

**L2-05 역분개 표면 유실.** 탐지기가 발화한 135문서 중 **109가 표면에 오르지 못했다.** 승격 게이트가 "적요·작성자까지 닮았는가"를 요구했는데, 그런 역분개는 ERP 자동 역분개이고 **수기 은폐형일수록 탈락하는 역선택**이었다. 게이트를 폐기하고 승격 탈락분을 전표 단위로 받는 경로를 추가해 유실 109 → **2**로 줄였다(잔여 2건은 다른 흐름에 흡수된 문서로, 표면에는 존재하나 해당 unit에 표시가 없는 한계로 기록).

탐지기와 표면 빌더가 같은 개념을 각자 구현하면 정의가 어긋난다는 것을, **발화→표면 전수 대조**가 처음 측정했다.

### 프리셋별 — 밀도 차이가 크다

| 프리셋           |   적중 | 표면 규모 |          밀도 |
| ---------------- | -----: | --------: | ------------: |
| 비용자산화       |  20~25 |       232 | **8.7~10.6%** |
| 수익인식         |   6~10 |       260 |      2.3~3.8% |
| 역분개·은폐      | 85~103 |   ~10,800 |      0.8~1.0% |
| 추정·관계사 조작 |   7~11 |     5,505 |    0.13~0.20% |

몸통 전체 OR(특징 미선택)은 적중 173~175(52~53%)에 표면 약 57,800이다. **좁게 고르면 밀도가 높고 넓게 고르면 건수가 많다**는 트레이드오프가 수치로 드러난다. 비용자산화 프리셋은 232건만 보면 그중 20여 건이 부정이고, 몸통 전체를 켜면 5만 건을 봐야 175건을 만난다. 이 판단을 도구가 대신하지 않고 감사인에게 건수로 보여주는 것이 설계 의도다.

| 위장 컴포넌트 (~40%의 다수) | scheme의 "다리" 전표 — 라벨은 부정이지만 전표 단독으로는 규칙 위반이 아님                                                       |
| 매출과대 룰 무반응          | 주입 부정 매출의 전표 단위 금액이 정상 분포 안(총량 조작을 전표로 쪼개는 현실 은폐). 금액 이상치 룰이 침묵하는 것이 도메인 귀결 |
| FS05 순환거래 22%           | 전표 하나하나는 정상 매출·매입·지급 모양. 원환은 거래처 그래프의 영역 — [12장](12_DIFFERENTIATION.md)의 원리적 한계             |
| FS14 유령급여 33%           | 정상 급여 패턴 모사라 전표 모양에 구분점 없음. 급여 마스터 대사 영역                                                            |

측정 중에 결함도 하나 잡았다. **추정계정 룰의 계정 목록 갭** — 계정 체계 동기화가 1개 계정만 반영해 해당 룰이 부정 데이터에서 발화 0이었다. 대손·감가상각·무형자산상각·상각누계 4계정을 추가하자 프리셋 적중이 **0~2 → 7~11**로 회복했다.

## 7.9 실증 예시 — 비용자산화 프리셋으로 한 건을 찾는 과정

감사인이 화면에서 "비용자산화" 프리셋 버튼을 누른다. 몸통에 L2-04 하나가 체크되고 특징은 10개 전체가 켜진다. 결합 의미론상 조건은 **"비용자산화 룰이 발화했고, 동시에 부정 전표의 모양 중 하나 이상이 있는 전표"**다.

화면에 즉시 일치 건수 **232건**이 표시된다. 이 숫자가 판단의 첫 재료다 — 감사인이 실제로 검토 가능한 규모이므로 조합을 더 좁힐 필요가 없다.

목록은 (분류 점수, 시간 심각도, 금액, 증거 수) 순으로 정렬되고, 각 행의 "발화 룰" 컬럼에 그 전표에 걸린 룰 이름이 함께 표시된다. 예컨대 "비용자산화(몸통) + 수기 입력(특징) + 기말 전기" 세 개가 걸린 전표라면, 감사인은 **비용으로 처리해야 할 지출을 자산으로 올렸는데 그것을 수기로 기말에 넣었다**는 이야기를 컬럼에서 읽는다. 다만 행을 펼쳐 룰별 근거 행까지 보는 드릴다운은 **빌더 패널에 아직 구현돼 있지 않다** — 그리드는 선택 이벤트를 소비하지 않고 라벨 문자열만 보여준다([11장](11_DASHBOARD.md)).

이 232건 안에 실제로 심어둔 부정이 20~25건 있다(밀도 8.7~10.6%). 무작위로 232건을 뽑았다면 부정 밀도 0.29%인 모집단에서 기대값은 0.7건이다. 즉 이 프리셋은 **무작위 대비 약 30배 농축**된 목록을 만든다.

그러나 도구는 그 20여 건을 "부정"이라고 표시하지 않는다. 232건 전부가 동등하게 "검토 후보"이고, 어느 것이 실제 문제인지는 감사인이 증빙을 확인해 판단한다. 이것이 등급을 선언하지 않는다는 원칙의 실제 모습이다.
