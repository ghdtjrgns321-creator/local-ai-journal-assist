# 11. 대시보드 — 현재 어디까지 만들어졌나

## 11.1 파이프라인 내 위치

```
생성 → 품질검증 → 수집·파생·검증 → 탐지 → 표면 → 적재 → ▶▶▶ [화면] ◀◀◀
```

**이 장은 미완성 구간을 기록한다.** 분석 엔진과 검증 체계는 완주했지만 화면은 아직 다듬어지지 않았고, 그 상태를 정확히 남기는 것이 이 장의 목적이다.

## 11.2 실제로 도달 가능한 화면 6개

`dashboard/app.py:274-311`이 import하고 호출하는 화면이 전부다.

```
[회사 선택/등록]  page_company.py
        ▼
[업로드 → 매핑 확인 → 파이프라인 실행]  components/data_uploader.py
        ▼
┌─────────────────── 결과 화면 (탭 6개) ───────────────────┐
│ 개요            tab_overview.py         1,115줄          │
│ 회사별 설정     app.py 내부 함수                          │
│ Phase1 결과     tab_phase1.py           4,926줄  ★       │
│ Phase2 결과     tab_phase2.py           4,020줄          │
│ Phase1,2 비교   tab_phase_comparison.py   262줄          │
│ 전기 비교       tab_comparison.py       1,509줄          │
└──────────────────────────────────────────────────────────┘
```

탭 전환은 nav만 담당하고 **활성 탭만 렌더**한다. 6개 패널을 동시에 평가하면 로딩이 느려지기 때문이다.

## 11.3 미배선 모듈 9개 — 파일은 있으나 도달 불가

배선 여부를 grep으로 전수 확인한 결과, 탭 모듈 9개가 `app.py`에서 import되지 않는다. 다른 곳의 참조는 전부 주석·docstring 문구뿐이라 **실행 경로에서 도달할 수 없다.**

| 모듈                  | 줄수 | 상태                                                            | 유일한 참조             |
| --------------------- | ---: | --------------------------------------------------------------- | ----------------------- |
| `tab_review_queue.py` |  573 | 미배선 — 내부 분석 트리거는 no-op 스텁                          | 하위 컴포넌트 docstring |
| `tab_export.py`       |  415 | 미배선 — 코드 품질은 양호하나 도달 불가                         | 패키지 docstring        |
| `tab_data_quality.py` |  236 | 미배선                                                          | 패키지 docstring        |
| `tab_eda.py`          |  145 | 미배선 — `tab_data_quality`와 기능 중복                         | 주석                    |
| `tab_benford.py`      |  137 | 미배선 — 유사 기능이 개요 탭에 내장돼 있음                      | **0곳**                 |
| `tab_summary.py`      |  133 | 미배선 — 개요 탭 후반부와 중복                                  | 주석                    |
| `tab_explorer.py`     |  103 | 미배선 — docstring은 "메인은 tab_findings"라 하나 그것도 미배선 | 주석                    |
| `tab_findings.py`     |   87 | 미배선                                                          | 주석                    |
| `tab_chat.py`         |   17 | 미배선 — 스스로 레거시 no-op 스텁이라 선언                      | 주석                    |

패키지 docstring은 "메인 UX는 개요·데이터품질·Phase1·Phase2·내보내기를 조합해 사용한다"고 적혀 있으나 그중 **데이터품질과 내보내기 2개가 미배선**이다(나머지 3개는 정상 배선). 문서가 코드보다 오래됐다.

미배선 모듈에 딸린 컴포넌트도 함께 도달 불가 상태다. 탐색기 그리드, 전표 상세 패널, 예외처리 CRUD, SHAP 워터폴, 검토 큐 브라우저·워크플로가 여기 해당한다. 코드 자체는 완성돼 있고 테스트도 있지만 **화면에서 갈 수 있는 길이 없다.**

## 11.4 조합 빌더 패널 — 가장 최근에 만들어진 화면

PHASE1 탭의 검토 케이스 영역이 조합 빌더를 호출한다(`components/phase1_combo_builder_panel.py`, 149줄).

| 요소      | 구현                                                                               |
| --------- | ---------------------------------------------------------------------------------- |
| 프리셋    | 버튼 4개 — 클릭 시 세션 상태에 몸통·특징 체크 상태를 주입                          |
| 어휘 선택 | 알약(pills) 2그룹 — 몸통 10 / 특징 10. **몸통 라벨에 감리 사례 건수 표기**         |
| 엄격 모드 | 토글                                                                               |
| 근거      | 접이식 영역에 룰별 발화 근거                                                       |
| 결과      | AgGrid 표 — "발화 룰" 라벨 컬럼. **행 선택 드릴다운은 미구현**(선택 이벤트 미소비) |

몸통 라벨에 감리 건수(추정계정 147, 관계사 80 …)를 노출하는 것이 이 화면의 설계 의도를 보여준다. 시스템이 "이게 더 위험하다"고 말하는 대신 **근거의 양을 보여주고 감사인이 판단하게** 한다.

## 11.5 PHASE1 탭의 상태 — 4,926줄과 미검증

`tab_phase1.py`는 4,926줄이며 최상위 정의가 105개다. 4개 하위 탭(전체 요약 / 데이터 정합성 / 검토 케이스 / 통계 결과)을 담고 있고 검토 케이스가 조합 빌더를 호출한다.

**개별 함수의 동작은 전수 검증되지 않았다.** 등급 폐지 대수술 후 도달 불가 블록 310줄을 삭제했고 관련 테스트는 통과했지만, 105개 함수 각각이 의도대로 동작하는지는 확인되지 않았다. 이것이 "Streamlit을 아직 제대로 만들지 못했다"의 구체적 내용 중 하나다.

## 11.6 무엇이 남았나

| 항목                 | 상태                                                                                   |
| -------------------- | -------------------------------------------------------------------------------------- |
| 화면 흐름            | 회사 선택 → 업로드 → 실행 → 결과 6탭은 동작                                            |
| PHASE1-1 조합 빌더   | 구현·배선 완료                                                                         |
| PHASE1-2 전용 표면   | **미구현** — 신호는 생성·적재되나 화면에 전용 뷰 없음([8장](8_PHASE1-2_ANALYTICAL.md)) |
| 검토 큐 워크플로     | 코드 존재, **미배선**                                                                  |
| 내보내기(Excel·PDF)  | 코드 존재, **미배선**                                                                  |
| 데이터 품질·EDA 화면 | 코드 존재, **미배선**(기능 중복 상태)                                                  |
| 탭 정리              | 중복·레거시 모듈 9개가 정리되지 않은 채 남아 있음                                      |
| 화면 문구            | 주장 범위 규약(등급 비선언·확정 표현 금지)의 UI 반영 미확인                            |
| 함수 단위 검증       | PHASE1·PHASE2 탭의 개별 함수 미검증                                                    |

작업 순서는 정해져 있다. 검증 결과 문서가 UI에 실을 수 있는 주장의 범위를 정의했고, 데모용 데이터셋(부정 330건이 심긴 overlay와 정상 base)이 준비돼 있으며, 조합 빌더 패널이 기준 구현으로 존재한다. 남은 일은 이 재료로 화면을 정리하는 것이다.

## 11.7 실증 예시 — 감사인이 지금 실제로 할 수 있는 일

현재 상태에서 데모가 어디까지 가는지 따라간다.

앱을 실행하면 **회사 선택** 화면이 뜬다. 회사를 만들거나 고르고, 감사연도를 선택하면 그 조합에 해당하는 DuckDB가 열린다(없으면 생성).

**업로드** 단계에서 원장 CSV를 넣으면 3단계 스테이지가 돈다: 업로드 → 매핑 확인 → 파이프라인 실행. 매핑 확인 화면에서 자동 매핑 결과와 애매한 후보가 나뉘어 표시되고, 감사인이 확정한다.

**실행**이 끝나면 결과 화면으로 전환된다. **개요** 탭에서 전체 규모와 KPI, Benford 요약을 본다. **Phase1 결과** 탭으로 가면 네 개 하위 탭이 있고, 검토 케이스 탭에서 조합 빌더를 만난다.

여기서 "비용자산화" 프리셋을 누르면 [7장](7_PHASE1-1_COMBO-BUILDER.md) 7.9절의 흐름이 그대로 화면에서 일어난다 — 몸통 1개·특징 10개가 체크되고, 일치 건수가 표시되고, 정렬된 목록이 "발화 룰" 컬럼과 함께 나온다. 감사인은 알약을 눌러 조합을 바꾸면서 건수 변화를 보고 범위를 조절한다. 다만 개별 행을 펼쳐 룰별 근거 행까지 확인하는 드릴다운은 아직 없다.

**Phase2 결과** 탭에서 VAE 결과를 별도로 볼 수 있다. 점수가 PHASE1과 합쳐지지 않고 따로 표시되는 것이 3-surface 비병합의 화면상 구현이다.

여기까지가 현재 동작하는 범위다. 검토 큐에 판정을 기록하거나, 결과를 Excel로 내보내거나, 분석신호 전용 화면을 보는 것은 **코드는 있으나 화면에서 갈 수 없다.** 데모로 보여줄 수 있는 것과 제품으로 쓸 수 있는 것 사이의 거리가 여기 있다.
