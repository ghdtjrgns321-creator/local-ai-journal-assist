# 1. 개요 — 문제 정의와 접근

## 1.1 어떤 실패를 막는 시스템인가

외부감사에서 전표 테스트(Journal Entry Testing)는 선택이 아니다. PCAOB AS 2401과 ISA 240은 경영진의 통제 무력화 위험 때문에 감사인이 **전 기간의 분개를 대상으로** 부정 위험이 높은 항목을 선별해 검사하도록 요구한다. 문제는 규모다. 중견 제조업 한 곳의 3개년 원장이 35만 행을 넘고, 감사인이 실제로 열어볼 수 있는 것은 수십 건이다.

여기서 두 방향의 실패가 발생한다.

**첫째, 표본이 편향된다.** 금액 상위 100건을 뽑으면 고액 정상 거래가 목록을 채우고, 소액으로 쪼갠 은폐는 통째로 빠진다. 실제 부정은 금액이 아니라 **처리 방식**(수기 입력, 승인 생략, 심야 전기, 기말 집중)에 흔적을 남기는데, 그 축은 정렬 기준에 들어오지 않는다.

**둘째, 자동화 도구가 등급을 대신 선언하면 그 등급을 방어해야 한다.** "이 전표는 HIGH 위험"이라고 도구가 말하는 순간, 감사인은 왜 HIGH인지 설명할 수 있어야 하고, 설명의 근거가 "우리 합성데이터에서 드물었다"면 그것은 감사 증거가 아니다. 이 프로젝트는 실제로 그 등급 체계를 만들었다가 근거가 무너지는 것을 확인하고 폐기했다([14장](14_JOURNEY.md)).

이 도구가 하는 일은 좁고 분명하다. **전수 모집단에서 규칙 위반·정책 위반·이상 징후를 넓게 올려 감사인이 먼저 볼 항목을 만드는 1차 스크리닝**이다. 부정을 확정하지 않고, 확정할 수 있다고 주장하지도 않는다.

## 1.2 왜 이 접근인가 — 대안과 비교

| 접근                             | 방식                                               | 왜 채택하지 않았나 / 어떻게 반영했나                                                                                                                                                                                                             |
| -------------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 지도학습 분류기                  | 부정 라벨로 학습해 확률 출력                       | 합성데이터의 라벨은 우리가 심은 것이라 학습하면 **자기가 심은 규칙을 되찾는 순환 학습**이 된다. 실제로 치트루트 감사에서 단일 컬럼 AUROC 0.95 이상이 18건 나와 NO-GO 판정([4장](4_DATASYNTH-QUALITY.md)). 파이프라인은 인프라로만 두고 휴면 상태 |
| 비지도 딥러닝 단독               | 정상 분포를 학습해 이탈을 탐지                     | 보조 표면으로 채택했으나 **행 단위 검증은 실패**했다(AUROC 0.5187). 실패의 원인과 그것이 원리적 귀결인 이유는 [9장](9_PHASE2_VAE.md)                                                                                                             |
| 자동 위험등급 룰 조합            | 룰 조합마다 HIGH/MEDIUM 부여                       | 구현했다가 **폐기**했다. 감독당국 지적 사례 731행을 엄격 재태깅하니 등급의 실증 근거가 무너졌다([7장](7_PHASE1-1_COMBO-BUILDER.md))                                                                                                              |
| **결정론 룰 + 감사인 조합 선택** | 근거 표기 가능한 어휘만 제공, 조합은 감사인이 선택 | **채택**. 시스템이 등급을 선언하지 않으면 방어할 휴리스틱이 0이 된다. 감사기준서 240에서도 분개 테스트 항목 선정은 감사인 판단 사항이다                                                                                                          |

빅4 회계법인의 저널 테스트 도구를 조사한 결과도 같은 방향이었다. EY Helix GLAD는 감사인 라벨 기반 지도학습, KPMG Clara는 룰·기준 분석에 ML 보조, PwC Halo는 룰 묶음과 의심 점수, Deloitte Omnia는 기준 기반 분석이다. 네 도구 모두 **룰·기준이 주력이고 ML은 보조**이며, 비지도 딥러닝 단독 탐지는 실무에 없다. 본 프로젝트의 실패한 검증(9장)은 도구를 실무 표준에서 벗어나게 만든 것이 아니라, 실무가 왜 그렇게 생겼는지를 실측으로 재확인한 셈이다.

마지막 두 원칙이 이 프로젝트의 성격을 결정한다. 검증 절차가 스스로를 검사하도록 만들었기 때문에, 성공뿐 아니라 **실패도 기록으로 남았다**.

## 1.5 검증 대상 범위와 그 한계

이 도구는 **단일 법인의 총계정원장(GL) 하나**를 본다. 이 경계가 무엇을 가능하게 하고 무엇을 불가능하게 하는지가 보고서 전체에서 반복되는 주제다.

- **가능**: 전표 단위 규칙 위반, 계정·거래처·작성자 단위 집계 신호, 전기 대비 분석적 검토
- **불가능**: 순환거래의 원환 추적(자금 네트워크 전체가 보여야 성립), 가공거래처 확정(거래처 마스터 비보유), 유령급여 확정(인사 마스터 대사 영역)

또한 이 도구는 **이 프로젝트가 생성한 합성 원장(표준 컬럼 스키마) 전용**으로 검증됐다. 다른 ERP의 실제 원장을 넣으려면 사용자가 컬럼 매핑을 직접 손봐야 하며, 무설정 투입 단계는 아니다(상세: [12장](12_DIFFERENTIATION.md) ⑩).

불가능한 것을 "부분적으로 된다"고 말하지 않고 선을 그은 기록은 [12장](12_DIFFERENTIATION.md)과 [9장](9_PHASE2_VAE.md)에 있다.
