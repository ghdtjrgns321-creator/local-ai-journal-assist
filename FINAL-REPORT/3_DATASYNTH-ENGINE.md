# 3. 합성데이터 생성기 (Rust)

## 3.1 파이프라인 내 위치

```
▶▶▶ [생성] ◀◀◀ → 품질검증 → 수집 → 파생 → 검증 → 탐지 → 표면 → 적재 → 화면
```

이 장은 파이프라인의 최상류다. 감사 도구를 검증하려면 원장이 필요한데 실제 기업 원장은 구할 수 없으므로, 회계적으로 성립하는 원장을 생성해 쓴다. 생성기는 `tools/datasynth/`의 Rust 워크스페이스이며 상위 저장소에서 추적하지 않는 별도 벤더 코드다(`.gitignore:68`). 프로젝트가 사용하는 것은 빌드된 바이너리 `datasynth-data`와 설정 YAML뿐이다.

## 3.3 생성 흐름

```
config(YAML 17섹션)
   │  ※ 전 필드가 serde(default) — 섹션을 빠뜨리면 기본값이 조용히 채워진다
   ▼
GeneratorConfig ──▶ validate_config (28개 검증 함수: rate∈[0,1], 분포합 1.0±0.01, 임계 오름차순)
   ▼
EnhancedOrchestrator::generate()
   │
   ├─ 1. CoA 생성 ─── seed_canonical_accounts()
   │        감사룰이 참조하는 실명 계정(가지급금 1190 등)을 자동생성 계정보다 먼저 심어 코드 충돌 방지
   ├─ 2. 마스터데이터 ─ 거래처·고객·자재·자산·직원 생성기를 개별 직접 호출
   ├─ 3. 문서흐름 ──── P2P / O2C 생성기 → 3-way match (가격 5% · 수량 2% · 과납품 10% 허용)
   ├─ 4. OCPM 이벤트 ─ happy 75% / exception 20% / error 5%
   ├─ 5. 전표 생성 ─── JournalEntryGenerator (아래 3.4)
   ├─ 6. 이상치 주입 ─ AnomalyInjector (정상 base에서는 전 비율 0으로 비활성)
   ├─ 7. 잔액 추적 ─── A = L + E + (수익−비용), 허용오차 0.01 실시간 검증
   └─ 8. 출력 ─────── output_writer → journal_entries.csv (73컬럼) + 사이드카
```

설정 스키마의 모든 필드가 `#[serde(default)]`라는 사실이 이 프로젝트에서 가장 비싼 대가를 치른 설계 특성이다. YAML에서 섹션 하나를 빠뜨리면 에러가 아니라 **기본값이 조용히 적용된다**. 기본값은 USD 기준(`lognormal_mu = 7.0`)이고 원화 설정은 14.0이므로, 금액 섹션이 빠지면 전체 금액이 e⁷ ≈ 1,097배 축소된다. 실제로 두 번 발생했다([15장](15_TROUBLESHOOTING.md) 사고 1).

## 3.4 전표 한 건이 만들어지는 경로

```
JournalEntryGenerator
  ├─ ① 회사 선택   WeightedCompanySelector (누적분포 이진탐색)
  ├─ ② 시나리오    BASELINE_SCENARIOS 20종 가중치 룰렛
  ├─ ③ 계정 결정   account_determination.rs::resolve_label
  │                  └ overrides(시나리오·방향·subtype 3중키) 우선 → default
  ├─ ④ 라인 수     LineItemSampler — 2줄 60.68%, 4줄 16.63%, 3줄 5.77% (실증 논문 분포)
  ├─ ⑤ 금액        AmountSampler — LogNormal(14.0, 2.5), 하한 100원, 상한 1,000억
  ├─ ⑥ 일시        TemporalSampler — 월말 2.5배·분기말 4.0배·연말 6.0배, 오전 피크 1.8배
  ├─ ⑦ 적요        DescriptionGenerator (템플릿 2,118줄)
  ├─ ⑧ 승인자      UserPool — 한도 이상 승인권자 중 선택, 생성자 본인 제외
  ├─ ⑨ 의미 게이트 semantic_validator (SEM001~SEM011) — 위반 시 생성 차단
  └─ ⑩ 균형        ensure_balance — 총차변 == 총대변
```

**의미 게이트가 실질적 문지기다.** 시나리오별로 허용되는 차변·대변 subtype 화이트리스트가 있어(`process_gl_mapping.rs`, 1,786줄), 매입송장 시나리오에서 매출 계정이 차변에 오는 식의 조합은 생성 시점에 차단된다. 이 게이트가 없던 시절에는 사후에 원장을 40여 회 덧칠해 고치는 방식이었고, 그것이 이 프로젝트의 가장 큰 아키텍처 전환을 유발했다([14장](14_JOURNEY.md)).

## 3.5 계정결정표 — 무작위 선택을 결정론으로

생성기 초기에는 semantic subtype이 정해진 뒤 계정과목표에서 **맞는 계정을 무작위로 골랐다**. subtype이 다대일로 축약되는 구조(급여·임차료·수도광열·소모품비가 모두 "영업비용"으로 묶임) 때문에, 서로 다른 의미의 subtype이 같은 후보군을 공유했다. 결과는 **임차료 계정에 소모품비가, 부가세예수금이 이연법인세부채(2500)에 기표**되는 원장이었다. 실제 ERP는 여기가 무작위가 아니다.

교정으로 도입한 것이 `account_determination.yaml`이다. 3층 구조를 가진다.

| 층                       | 내용                                                | 건수 |
| ------------------------ | --------------------------------------------------- | ---: |
| `default`                | subtype → 계정 1:1 매핑                             |  40+ |
| `reverse_representative` | 계정 → 대표 subtype 역조회(공유 계정의 모호성 해소) |    5 |
| `overrides`              | (시나리오, 차/대변, subtype) 3중키 예외             |    2 |

주요 매핑:

| subtype                            | 계정     | subtype           | 계정 |
| ---------------------------------- | -------- | ----------------- | ---- |
| AP (매입채무)                      | 2000     | AR (매출채권)     | 1100 |
| GRIR (입고미착)                    | 2900     | CASH              | 1000 |
| INVENTORY                          | 1200     | PREPAID_EXPENSE   | 1300 |
| COGS_MATERIAL                      | 5000     | COGS_DIRECT_LABOR | 5200 |
| OPEX_PAYROLL                       | 6100     | OPEX_RENT         | 6300 |
| **SUSPENSE_RECEIVABLE (가지급금)** | **1190** | RETAINED_EARNINGS | 3200 |
| IC_PAYABLE                         | 2050     | IC_RECEIVABLE     | 1150 |

가지급금 1190은 감사 룰이 이름으로 참조하는 계정이라 자동 생성 계정보다 먼저 심는다. 조회 함수는 모호할 때 추측하지 않는다 — 역조회에서 후보가 2개 이상이면 대표 매핑이 없는 한 `None`을 반환한다. 계약 테스트 2건이 "모든 시나리오의 모든 방향·subtype 조합이 해소되는가"와 "명명된 계정이 왕복 일치하는가"를 강제한다.

`overrides` 2건은 관계사 거래용이다. 관계사 매출의 대변 청산은 채권이 아니라 **채무측(2050)**이라는 회계 실무를 표에 명시했다. 이 어휘는 원래 빠져 있었는데, 표를 처음 만들 때의 시험 데이터가 단일 법인이라 관계사 거래 자체가 없었기 때문이다. 3사 생성 데이터에서 722행을 전수 측정하는 게이트가 이를 적발했다.

## 3.7 fraud overlay — 부정 시나리오 주입

정상 base는 그대로 두고 별도 파일로 부정을 얹는다. 14개 scheme이 각각 3~4단계 구성요소를 가진다(`phase2_scheme_overlay.rs:22-187`).

| id   | 유형            | 심각도 | 흐름 |
| ---- | --------------- | ------ | ---- |
| FS01 | 가공매출        | high   | O2C  |
| FS02 | 진행률 조작     | high   | GL   |
| FS03 | 현금 횡령       | high   | P2P  |
| FS04 | 횡령 은폐       | high   | P2P  |
| FS05 | 순환거래        | high   | O2C  |
| FS06 | 부채 누락       | high   | P2P  |
| FS07 | 재고 과대       | medium | P2P  |
| FS08 | 부적절 자본화   | medium | GL   |
| FS09 | 컷오프 조작     | medium | O2C  |
| FS10 | 대손 회피       | high   | O2C  |
| FS11 | 특수관계자 남용 | high   | IC   |
| FS12 | 충당금 누락     | medium | GL   |
| FS13 | 손상 회피       | high   | IC   |
| FS14 | 유령직원 급여   | medium | H2R  |

FS01(가공매출)의 4단계를 예로 들면: 가공 매출 계상 → 가짜 회수 → 매출채권 할인 차입 → 차기 역분개다. 부정이 전표 한 장이 아니라 **여러 장에 걸친 이야기**로 심긴다는 것이 중요하다. 그중 일부 전표는 그 자체로는 아무 규칙도 위반하지 않는 "다리" 역할이고, 이것이 표면 커버리지가 60% 안팎에 머무는 이유다([7장](7_PHASE1-1_COMBO-BUILDER.md)).

### 지름길 차단

탐지기가 구조가 아니라 흔적을 보고 맞히는 것을 막는 장치가 여러 겹이다.

| 장치               | 내용                                                                                                                    |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| 라벨 컬럼 제거     | 원장 CSV에서 `is_fraud`·`fraud_type`·`is_anomaly`·`anomaly_type` 등 최대 15개 컬럼을 제거                               |
| 헤더 정규화        | 시나리오 설명이 적요에 노출되지 않도록 고빈도 정상 헤더로 치환 — "시나리오명이 원장에서 보이는 지름길 토큰이 되지 않게" |
| 관계 증거 정책     | 직원-거래처 관계를 원장에서 보이는 필드로 표현하지 않음                                                                 |
| source 분포 노이즈 | 소스 필드 분포를 시나리오별로 섞어 라벨이 결정론적으로 예측되지 않게 함                                                 |
| 계정 지름길 폐기   | 부정 전용 신규 계정 도입을 폐기하고 정상이 많이 쓰는 22개 계정으로 전면 교체                                            |

마지막 항목은 s10 세대에서 도입됐다. 이전 overlay는 부정용으로 6자리 확장 계정 23종을 새로 만들었는데, 그러면 "본 적 없는 계정 = 부정"이라는 지름길이 생긴다. 정상 데이터에서 문서 수가 희소한 계정(1300·3200·4500·6300)도 사용을 금지했다.

## 3.8 실증 예시 — 매입송장 전표가 만들어지는 순간

`P2P_VENDOR_INVOICE` 시나리오가 뽑힌 뒤 벌어지는 일을 실제 값으로 따라간다.

시나리오 가중치 300은 20종 중 최대다. 이 시나리오의 허용 차변 subtype 목록과 허용 대변 subtype 목록이 게이트로 걸린다. 계정결정표가 대변 subtype `AP`를 계정 **2000**(매입채무 통제계정)으로, 차변 입고미착 `GRIR`을 **2900**으로 해소한다. 세액이 0보다 크면 라인이 3줄이 된다 — 차변 입고미착(공급가액), 차변 매입부가세(세액), 대변 매입채무(총액). 세액이 없으면 2줄이다.

승인자 배정은 `UserPool`이 담당하는데 조건이 세 개다: 승인권 보유, 승인 한도가 금액 이상, **생성자 본인이 아닐 것**. 마지막 조건이 자기승인(L1-05)이 무작위로 발생하지 않게 막는다. 다만 정상 데이터에도 자기승인이 **있어야 한다** — 실무에서 전결 규정에 따른 자기 결재는 존재하고, 감사인이 봐야 할 항목이기 때문이다. 그래서 별도 파라미터 `self_approval_rate = 0.02`로 승인 문서의 2%에 전결을 부여한다. 설정 주석이 계산을 밝힌다: 승인 3만 2천 문서 기준 약 640건이며, 이는 룰의 정상 대역 [0.01%, 1%]의 중앙부다.

마지막으로 균형 검증을 통과하면 이 전표는 원장에 확정된다. 2.3절에서 추적한 `ef8bb035-…` 전표가 바로 이 경로로 만들어진 것이다.
