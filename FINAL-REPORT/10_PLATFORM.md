# 10. 플랫폼 — 격리·설정·저장·CI

## 10.1 파이프라인 내 위치

```
생성 → 품질검증 → 수집·파생·검증 → 탐지 → 표면 → ▶▶▶ [적재·설정·CI] ◀◀◀ → 화면
```

파이프라인을 떠받치는 기반이다. 회사별 격리, 3계층 설정 해소, DuckDB 저장, 자동 검증이 여기 속한다.

## 10.2 Company-Centric 격리

```
data/companies/{company_id}/
├── company.yaml              회사 메타 + 설정 오버라이드
├── chart_of_accounts.csv     회사 계정과목표
├── keywords.yaml             회사 ERP 헤더 별칭
├── audit_rules.yaml          회사 감사 룰
├── profiles/                 컬럼 매핑 프로파일
└── engagements/{year}/
    ├── engagement.yaml       연도 메타 + 오버라이드
    ├── audit.duckdb          ★ 감사연도별 격리 DB
    └── models/               ML 모델 아티팩트
```

감사는 회사별·연도별로 독립이다. A사 2024년 감사에서 B사 데이터나 A사 2023년 임계가 섞이면 안 된다. DB 파일 자체를 나누는 것이 가장 확실한 격리다.

설정은 **3계층으로 해소**된다: 글로벌 기본값 → 회사 오버라이드 → 연도 오버라이드. 각 계층이 깊은 병합(deep merge)되므로, 회사가 승인한도만 바꾸면 나머지는 글로벌을 따른다.

강제 규칙도 하나 있다. 로컬 우선 정책에 따라 일부 설정(외부 API 사용 계열)은 **회사·연도 설정으로도 켤 수 없게** 최종 강제된다.

## 10.3 설정 파일

| 파일                          | 역할                           | 핵심 값                                                                                                                                                 |
| ----------------------------- | ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `settings.py`                 | 전역 Pydantic 설정 + YAML 로더 | 승인한도 6단계, Z-score 임계 3.0, 심야 22~06시, 기말 여유 5일, Benford MAD 0.012, 역분개 윈도우 45일·허용오차 0.0, PHASE2 학습 상한 5만행, 난수 시드 42 |
| `audit_rules.yaml`            | L1~L4 업무 룰 정본             | L1-07 최소 증거 2건, L2-05 제외계정 3종, L2-04 가계정 5종, L4-03 세전이익 5%·매출 0.5%·중요성비율 0.75, 통화별 소수자리(KRW·JPY 0 / USD·EUR 2)          |
| `sod_toxic_combinations.yaml` | 직무분리 위험 조합 근거        | 7 프로세스 → 4 기능, toxic 10쌍 + 프로세스 내 2 + 제외 4, 이진 플래그(점수 단계 없음)                                                                   |
| `combo_builder.yaml`          | 조합 빌더 어휘 정본            | 몸통 10(감리 확정 건수 병기) × 특징 10 × 프리셋 5                                                                                                       |
| `schema.yaml`                 | 원장 컬럼 스키마 정본          | 51컬럼 정의, **필수 10개**, 라벨 6종은 "탐지 입력 금지" 명시                                                                                            |
| `phase1_case.yaml`            | 케이스·테마 우선순위           | 상위 케이스 50, 테마당 10                                                                                                                               |
| `keywords.yaml`               | ERP 헤더 별칭 사전             | SAP·더존·위하고·영림원·Oracle EBS                                                                                                                       |
| `cleaning.yaml`               | 타입 캐스팅 규칙               | 통화기호 5종, Excel 날짜 일련번호 범위                                                                                                                  |
| `risk_keywords.yaml`          | 위험 적요 키워드               | 상품권·가계정·가수금·가지급·대여금·선급금 등                                                                                                            |
| `presets/`                    | 상황별 오버레이 3종            | 평시 / 결산기(심야 시작 2시, Z-score 4.0) / 건설업(기말 여유 10일, 유효숫자 1자리)                                                                      |

계정 코드를 코드 상수로 박지 않는다는 규칙이 여기서 지켜진다. 가계정 목록도, 역분개 제외 계정도, 위험 키워드도 전부 YAML이다. 회사마다 계정 체계가 다르므로 코드에 박으면 다른 회사에 쓸 수 없다.

설정 파일 두 개는 **자기 한계를 스스로 명시**한다. PHASE2 검토 등급 정책 파일은 "코드에서 직접 로드하지 않는다"고 적혀 있고(상수가 코드에 하드코딩돼 있어 수동 동기화가 필요), 위험 키워드 파일은 "적요 품질 피처는 이 파일을 쓰지 않는다"고 밝힌다. 설정이 있다고 그것이 쓰인다는 보장은 없으므로 이런 표기가 필요하다.

## 10.4 저장 — DuckDB

| 항목         | 내용                                                     |
| ------------ | -------------------------------------------------------- |
| 스키마       | 원장 테이블 약 90컬럼 + 탐지 결과·케이스·사이드카 테이블 |
| 마이그레이션 | v1 → v6 버전 관리                                        |
| 연결         | `ConnectionManager`가 engagement별 파일 경로를 해소      |
| 조회         | 프리셋 쿼리 모음 제공                                    |

DuckDB를 택한 이유는 분석 질의(집계·윈도우 함수)가 주 작업이고, 서버 없이 파일 하나로 격리가 되기 때문이다. 감사인 노트북에서 외부 접속 없이 돌아야 한다는 제약과도 맞는다.

## 10.6 CI — 그리고 그 결함

활성 워크플로 4개와 비활성 초안 1개가 있다.

| 워크플로                     | 트리거                             | 게이트                                        | 강도                                   |
| ---------------------------- | ---------------------------------- | --------------------------------------------- | -------------------------------------- |
| `phase1-kpi-guard.yml`       | PR(경로 8종), main push, 주간 cron | Layer A+B 도메인·부하 가드, Layer C 회귀 관찰 | **A/B = HARD(PR 차단)**, C = SOFT WARN |
| `audit-testing.yml`          | 매일 03:00 KST cron + 수동         | PHASE2 Layer A 가드 + 케이스 계약 회귀        | HARD (단 PR 미차단)                    |
| `docs-audit.yml`             | PR + push                          | 결정 ID 감사, deny 룰 감사                    | HARD                                   |
| `phase2-eval.yml`            | PR + push                          | 평가 리포트 마크다운 검사                     | 명목상 HARD                            |
| `phase2-kpi-guard.yml.draft` | —                                  | 설계만 존재                                   | 비활성                                 |

주 워크플로의 원칙이 주석에 못 박혀 있다 — **"PHASE1 향상은 도메인 정합성으로만 정당화한다. truth recall 향상을 강제하는 가드를 두지 않는다."** 기준선 갱신도 "도메인 정당성 명시 의무, truth recall 향상을 갱신 사유로 사용 금지"다. 성능 숫자를 올리는 방향으로 CI가 압력을 가하면 결국 데이터에 맞춰 룰을 조정하게 되기 때문이다.

SOFT WARN은 연속 3회·6회에서 이슈를 자동 생성하고, 정상 실행 1회로 연속 카운터가 초기화된다.

### 정직하게 기록할 CI 결함 6건

| #   | 결함                                                                                                                                                                                                                                                                                       | 근거                         |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------- |
| C-1 | **`phase2-eval.yml`은 현재 아무것도 검사하지 않는다.** 검사 대상 파일 2개가 모두 부재하고 코드가 부재 시 건너뛴다. 게다가 대상 디렉토리는 gitignore로 통째 제외라 CI 체크아웃에 **존재할 수 없다**                                                                                         | 파일 부재 실측 + 코드        |
| C-2 | **주 KPI 가드의 기준선이 스스로 "퇴역" 상태.** 상태값이 `retired_pending_reestablishment`(2026-07-15). 사유 셋: 원본 데이터셋 2개가 디스크에 없어 재측정 경로 단절, 산출물이 신선도 기준 7일을 31일 초과해 HARD FAIL 중, **기준선은 39룰 기준인데 현행은 26룰이라 분모가 구조적으로 다름** | 기준선 메타 원문             |
| C-3 | 회귀 체크포인트에 케이스 빌더 단계가 없어 해당 테스트가 단언 실패가 아니라 **KeyError로 실패**                                                                                                                                                                                             | 동상                         |
| C-4 | **PHASE2 KPI 가드는 설계만 있고 구현 없음.** 초안이 참조하는 스크립트가 부재하고 기준선도 골격 상태                                                                                                                                                                                        | 파일 부재 실측               |
| C-5 | `audit-testing.yml`은 PR 트리거가 없어 **PR을 차단하지 못한다** — 야간에만 돈다                                                                                                                                                                                                            | 워크플로 원문                |
| C-6 | 그 워크플로가 의존하는 기준 파일이 없으면 테스트가 skip 처리되어 **조용히 통과**한다                                                                                                                                                                                                       | gitignore 체인 + 테스트 코드 |

C-2와 C-6이 이 프로젝트가 반복해서 만난 문제의 전형이다. **"통과"가 "검증됨"을 의미하지 않는다.** 기준선이 퇴역했는데 가드가 돌고 있고, 파일이 없으면 조용히 넘어간다. [4장](4_DATASYNTH-QUALITY.md)의 항등식 게이트 4종을 INFO로 강등한 것과 같은 종류의 문제이며, 그쪽은 조치했고 이쪽은 **미조치 상태로 기록**한다.

## 10.7 배포 구성

| 항목            | 내용                                                                                                  |
| --------------- | ----------------------------------------------------------------------------------------------------- |
| Dockerfile      | 멀티스테이지 — 빌더에서 uv로 의존성 설치 후 런타임 이미지에 `.venv`와 코드만 복사. 비루트 사용자 실행 |
| 설치 그룹       | core + dashboard + ml + nlp + export (**llm·dev 제외**)                                               |
| 실행            | Streamlit 8501 포트, headless, 사용 통계 수집 끔                                                      |
| compose         | app(내부 노출만) + nginx(80 포트, 기본 인증)                                                          |
| `.dockerignore` | **생성기 전체를 이미지에서 제외**, 데이터는 볼륨 마운트만                                             |

이미지에 생성기와 데이터를 넣지 않는 것이 자산 보호 정책이다. 배포되는 것은 분석 도구이지 합성 데이터나 그 생성 수단이 아니다.

## 10.8 실증 예시 — 설정 하나가 3계층을 통과하는 과정

승인한도 임계가 어떻게 결정되는지 따라간다.

**글로벌 기본값**은 `settings.py`에 6단계로 정의돼 있다: 1천만 / 1억 / 10억 / 50억 / 100억 / 500억. 한국 중견 제조업의 전형적 전결 규정이다.

**회사 계층**에서 A사가 `company.yaml`의 오버라이드에 `approval_thresholds: [5000000, 50000000, ...]`를 넣으면 그 회사에만 적용된다. 실제 감사에서는 회사의 전결 규정 문서를 받아 이 값을 채운다.

**연도 계층**에서 2024년에 전결 규정이 개정됐다면 `engagement.yaml`에 그 해의 값만 넣는다. 2023년 감사는 여전히 구 규정으로 판정된다.

**프리셋 오버레이**는 여기에 상황을 얹는다. 결산기 프리셋을 켜면 심야 시작이 22시에서 2시로 내려간다 — 결산기에는 밤 11시 근무가 정상이므로 심야 룰의 기준을 늦추는 것이다. 이 오버레이는 임시 조정이라 저장된 설정을 바꾸지 않는다.

최종적으로 `CompanyContext`가 이 계층들을 병합해 하나의 설정 객체를 만들고, 파이프라인의 모든 단계가 그것을 참조한다. 승인한도 초과 룰(L1-04)은 이 값을 읽어 판정하므로, **같은 원장이라도 회사가 다르면 다른 결과**가 나온다. 그것이 정확한 동작이다 — 승인한도 초과는 절대 금액이 아니라 그 회사 규정 대비 위반이기 때문이다.
