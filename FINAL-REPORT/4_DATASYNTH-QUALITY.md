# 4. 합성데이터 품질 검증과 데이터셋 계보

## 4.1 파이프라인 내 위치

```
생성 → ▶▶▶ [품질 검증] ◀◀◀ → 수집 → 파생 → 검증 → 탐지 → 표면 → 적재 → 화면
```

생성된 원장이 "회계적으로 말이 되는가"를 판정하는 단계다. 이 판정이 느슨하면 이후의 모든 탐지 실측이 무의미해진다.

## 4.2 검증 3겹 구조 — 게이트만으로는 부족하다는 인정

```
┌────────────────────────────────────────────────────────────────┐
│ 1겹  게이트 57종           구현된 축의 자동 검사               │
│                            FAIL은 3분류: ①데이터 결함           │
│                            ②게이트 결함 ③제품 무관 → ①만 수정  │
├────────────────────────────────────────────────────────────────┤
│ 2겹  룰 발화율 대역        룰이 읽는 모든 축의 실질 심판        │
│                            정상에서 룰이 폭주하면 데이터가       │
│                            비현실적이라는 신호                  │
├────────────────────────────────────────────────────────────────┤
│ 3겹  overlay 표면화 실측   심은 이상이 잡히는가 — 최종 증거     │
└────────────────────────────────────────────────────────────────┘
```

프로젝트 문서가 게이트의 한계를 스스로 밝힌다. 현실성 카탈로그 146개 축 중 **게이트로 구현된 것은 49개(34%)**이고, 미구현 97개는 시간 10·분포 5·승인 11·노이즈 7·경제적 타당성 20 등이다. 게다가 구현된 57종 안에서도 네 개(M01·M03·M04·M07)는 **어떤 데이터로도 통과하는 항등식**이었고, ACC01은 판정 방향이 반대였다.

따라서 "게이트 전부 PASS = 현실적"은 성립하지 않는다. 미구현 축을 선제적으로 게이트화하지 않고, **2겹에서 발화율이 대역을 벗어난 룰이 가리키는 축만 역으로 파는** 방식을 택했다. 룰이 읽지 않는 축의 비현실은 결과물에 영향이 없기 때문이다.

## 4.6 폐기된 "덧칠 시대"

이전 세대(r6)는 생성 후 원장을 **40여 회 사후 변형**해 만들었다. 게이트는 44개가 통과했지만 실체는 무너져 있었다.

| 결함                                     | 수치                                           |
| ---------------------------------------- | ---------------------------------------------- |
| 부가세율이 공급가액의 4.12% (10%여야 함) | 중앙값 0.0412, 64,539행                        |
| 계정 체계 붕괴                           | 계정 검사 6~7/9 FAIL, 계정결정표 위반 77,166행 |
| 자기승인 0건 → 해당 룰 검증 불가         | 0 / 376,727                                    |
| 게이트 44 PASS의 신뢰성                  | 네 개가 항등식                                 |

생성기 자체는 건강했다. 정본 설정으로 덧칠 없이 생성한 clean_out은 부가세가 **정확히 0.1000**이었고, 덧칠이 존재했던 이유인 게이트(C06)가 생성 시점에 자연 통과했다. 즉 **덧칠은 처음부터 불필요했고, 오히려 정확했던 것을 깨뜨리고 있었다**. 이 전환의 서사는 [14장](14_JOURNEY.md)에 있다.

## 4.8 외부 데이터셋 검증

합성데이터만으로는 자기 참조를 벗어날 수 없으므로 외부 데이터셋 두 개로 교차 확인했다. 실행 **전에** 사용 계약을 문서로 고정한 것이 특징이다.

| 항목         | 계약 내용                                                                                                     |
| ------------ | ------------------------------------------------------------------------------------------------------------- |
| 허용         | 스키마 인벤토리, 라이선스 확인, 리키지 감사, 매핑 타당성, 그림자 벤치마크                                     |
| 금지         | 활성 모델 승격, 지도학습 승격, run/session/시나리오를 모델 입력으로 사용, 시뮬레이션 라벨을 운영 truth로 취급 |
| 리키지 5범주 | 라벨 계열 / 시나리오 계열 / 실행 이력 / 사후 주석 / 행 식별자                                                 |

**OpenDataPhilly**(필라델피아시 지출 데이터, 238,894행 / 거래처 5,506 / 부서 58)는 기저율 프로파일 확인에 썼다. 골든 리뷰 200건 중 라벨 일치율 84.5%였고, 불일치 31건은 전부 "정상으로 설명 가능"과 "검토 후보" 사이의 경계 사례였다. 지도학습 활성화 판정은 **NO-GO** — 신뢰할 만한 지도학습 양성/음성 표본이 0건이었기 때문이다.

**Tritscher ERP-Fraud**(200,629행 / 17,235문서 / 부정 248건)는 VAE 그림자 벤치마크에 썼다. 문서 단위 평균 AUROC 0.867로, 내부 데이터의 0.930보다 낮고 불균일했다(특정 실행에서 0.4931까지 하락). 다만 홀드아웃당 부정 표본이 6문서에 불과한 시뮬레이션이라 "코드가 동작한다" 이상의 의미를 부여하지 않기로 계약에 명시했다.

## 4.9 치트루트 감사 — 지도학습이 왜 휴면인가

합성데이터로 지도학습을 하면 모델이 부정을 배우는 게 아니라 **생성 규칙을 되찾는다**. 이를 측정하는 감사 프레임이 있다.

| 세대            | 결과                                                                                | 판정                              |
| --------------- | ----------------------------------------------------------------------------------- | --------------------------------- |
| manipulation v4 | 단일 컬럼 AUROC ≥ 0.95가 18건, 상호작용 69쌍, 시뮬레이션 로지스틱 평균 AUROC 0.9978 | **NO-GO**                         |
| v5              | 0.9955                                                                              | NO-GO                             |
| v6              | 0.9998 (악화)                                                                       | NO-GO                             |
| v7 fixed2       | 0.6914 (위험 컬럼 제거 후)                                                          | GO                                |
| v7 patched      | 0.9231                                                                              | 조건부 GO — deny-list 49컬럼 도입 |

산출물이 `LEAKAGE_DENY_COLUMNS`다. 이것은 감사 기록이 아니라 **프로덕션 코드에 살아 있는 배선**으로, `src/preprocessing/constants.py`가 감사 문서를 출처로 명시하며 차단 목록을 정의한다. 원칙도 명문화했다 — **deny-list는 확장만 허용하고, 성능(truth recall)을 올리려는 목적의 완화는 금지**한다.

## 4.10 실증 예시 — 부가세 게이트가 세대를 가른 순간

ACC01(세액 계정 사용) 게이트 하나가 두 세대의 운명을 갈랐다.

구 세대 r6에서 세액을 공급가액으로 나눈 값의 중앙값은 **0.0412**였다. 부가세율은 10%여야 하므로 명백한 오류지만, 놀라운 점은 **생성기가 아니라 사후 덧칠이 이를 깨뜨렸다**는 것이다. 생성기만 돌린 clean_out에서 같은 지표는 **정확히 0.1000**이었다.

그런데 이 결함을 처음에는 게이트가 잡지 못했다. 초기 ACC01은 라인 단위로 판정했는데, 세액이 거래 줄과 부가세 줄 양쪽에 붙는 구조 때문에 정상 데이터에서도 26%가 오탐으로 걸렸다. 게이트가 시끄러우면 아무도 보지 않는다.

2026-07-17 재설계에서 판정 단위를 **문서**로 바꿨다. 세액이 있는 전표에 부가세 계정 줄이 존재하는가, 그리고 문서 단위로 세액 합계를 공급가액으로 나눈 값이 9~11% 범위인가. 이 기준으로 다시 돌리자 r6은 미계상 37,765건으로 **FAIL**, s10은 중앙값 0.1000으로 **PASS**가 나왔다. 게이트가 비로소 두 세대를 구분하게 된 것이다.

이 사례는 게이트 FAIL을 3분류하는 이유를 보여준다. 처음의 ACC01 FAIL은 ①데이터 결함이 아니라 ②게이트 결함이었고, 데이터를 고쳤다면 오히려 정확한 데이터를 망가뜨렸을 것이다. 2.3절의 전표에서 세액 4,598,752.1원 ÷ 공급가액 45,987,521원 = 0.1000이 나오는 것이 이 게이트가 지키는 값이다.
