# 8. PHASE1-2 — 분석적 검토 신호

## 8.1 파이프라인 내 위치

```
                              ┌─ PHASE1-1 룰 (전표 단위)
검증 통과 DataFrame ──────────┼─ ▶▶▶ [PHASE1-2 분석신호 (모집단 단위)] ◀◀◀
                              └─ PHASE2 VAE (행 단위)
```

전표 한 장만 봐서는 안 보이고 **계정·거래처·작성자를 모아야 드러나는** 신호를 만든다. 근거는 감사기준서 520(분석적 절차)이다.

## 8.2 두 가지 운영 방식 — 자기 큐와 배지

```
┌─────────────────────────────────────────────────────────────┐
│ 자기 큐 (별도 목록)                                          │
│   Benford · D01 · D02 · 라운드넘버 밀집도 · 첫등장/희소 거래처│
│   → "이 계정/거래처가 통째로 이상하다"를 독립 표면으로       │
│   → 전표 등급에 미참여(점수 0)                               │
├─────────────────────────────────────────────────────────────┤
│ 배지 (전표 줄의 맥락 꼬리표)                                 │
│   L4-05 심야집중 · L4-06 배치 · L3-12 업무범위               │
│   + 라운드넘버 단건 · 첫등장/희소 전표                       │
│   → 정상 업무에 흔해서 단독 큐는 노이즈                      │
│   → PHASE1-1 전표에 꼬리표로만 표시, 등급 안 바꿈            │
└─────────────────────────────────────────────────────────────┘
```

라운드넘버와 거래처 신호는 **양쪽 모두**다. 밀집도(이 계정에 둥근 금액이 몰렸다)는 자기 큐, 단건(이 전표가 둥근 금액이다)은 배지다. 같은 현상이라도 모집단 수준의 쏠림은 신호이고 개별 건은 흔한 일이기 때문이다.

## 8.3 자기 큐 5종

| 신호               | 단위           | 판정 방식                                                              | 구현                     |
| ------------------ | -------------- | ---------------------------------------------------------------------- | ------------------------ |
| L4-02 Benford      | 계정·월        | MAD 0.012 이하 적합, 0.015 초과 강한 부적합. 표본 500건 미만 그룹 제외 | `benford_detector.py`    |
| D01 활동량 급변    | 계정·전기대비  | 금액합 0.5 + 건수 0.3 + 평균 0.2 가중합                                | `variance_rules.py`      |
| D02 월분포 변화    | 계정·전기대비  | Jensen-Shannon 발산, 4중 게이트(최소 3개월·문서 100건 등)              | `variance_rules.py`      |
| 라운드넘버 밀집도  | 계정·월·작성자 | 이항검정 + 효과크기 하한                                               | `round_density_rules.py` |
| 첫등장/희소 거래처 | 거래처         | 전기 미등장·하위 분위수·휴면 재활성                                    | `partner_signals.py`     |

### 라운드넘버 밀집도의 설계

기준은 PCAOB AS2401 §61(e)의 "끝자리 0이 다수인 금액" 문언이다. 판정에서 두 가지가 중요하다.

**기준선을 데이터에서 뽑는다.** 둥근 금액의 정상 비율은 산업과 회사마다 다르므로, 상수를 박지 않고 **해당 원장 전체의 라운드넘버 비율**을 기준선으로 삼는다. 각 그룹의 라운드 건수를 이항분포 단측검정으로 판정한다.

**표본이 크면 p값만으로 부족하다.** 건수가 많으면 사소한 차이도 통계적으로 유의해지므로, p값 앞에 **효과크기 하한**(기준선 대비 초과분 0.05)을 건다.

**모집단 축은 복합키가 아니라 독립**이다. 계정·월·작성자를 조합하면 그룹당 표본이 급감해 검정력이 죽는다.

실측: 정상 데이터(376,727행, 기준선 5.52%)에서 finding 1건(작성자 한 명, 29.0%), 부정 데이터(381,791행, 기준선 6.03%)에서 3건(계정 3개, 11.8~14.0%).

### 첫등장/희소 거래처 — 이름부터 정직하게

가공거래처(페이퍼컴퍼니)는 감사기준·감리사례·부정조사 3축 모두에서 고위험이지만, 이를 가르는 강한 신호(사업자번호 위조, 직원-거래처 공유 계좌·주소, 등록 직후 거래)는 **거래처 마스터가 있어야 보인다.** 본 프로젝트는 원장 단일 소스로 한정하고 마스터를 의도적으로 비보유한다.

그래서 원장 집계만으로 만들 수 있는 **약한 근사**를 검토 신호로 올린다.

| 신호         | 원장에서 만드는 법                                | 한계                         |
| ------------ | ------------------------------------------------- | ---------------------------- |
| 원장 첫 등장 | 전기 미등장 × 당기 등장 교차                      | 정상 신규 공급처도 첫 등장   |
| 원장 희소    | 거래 건수 하위 10분위                             | 정상 일회성 거래와 경계 모호 |
| 휴면 재활성  | 전기 활동 있음 + 직전 연도 결번 + 180일 이상 공백 | 정상 거래 재개와 구분 불가   |

명명 규율이 명문화돼 있다. **"신규 거래처"가 아니라 "원장 첫 등장"으로 부른다.** 마스터 등록일을 보는 게 아니라 우리 원장에 처음 찍힌 날을 보는 것이므로, "신규 거래처"라는 말은 마스터를 가진 것처럼 오인하게 만든다. 세 신호 모두 "가공거래처 탐지 완료"가 아니라 "검토 후보를 원장 범위 내에서 일부 표면화"로만 표기한다.

실측: 3개년 통합 376,727행에서 거래처 2,966개 중 첫등장 568(19%)·희소 596·휴면 22 → 합집합 **625개(21%)**가 그대로 finding이 된다.

검증 주의사항도 기록돼 있다 — 첫등장을 **단일 연도 파일로 돌리면 거의 모든 거래처가 "첫 등장"이 되어 무의미하다.** 반드시 연도 통합 실행이 필요하다.

## 8.4 검토 목록을 소유하지 않는다

PHASE1-2는 신호만 생성하고, 무엇을 몇 개 어떤 순서로 보여줄지는 화면이 정한다. 따라서 **생성 경로에서 절단하지 않는다.**

과거에는 findings를 정렬해 상위 100건만 남겼는데 제거했다. 이유가 둘이다. 신호 종류마다 점수 척도가 달라(Benford는 0~1, 다른 것은 상한 없는 표준화 점수) 한 리스트를 자르면 **척도가 큰 종류가 칸을 독식**한다. 그리고 백엔드가 미리 버린 신호는 **화면이 되살릴 수 없다.**

산출물 키도 단위별로 분리한다 — 계정·프로세스 단위와 거래처 단위를 합치지 않는다. 정렬 기준이 각각 0~1 점수와 합계 금액(원)이라 섞으면 척도가 어긋나기 때문이다.

예외는 개요 카드용 미리보기 하나뿐이며, 전체 건수를 별도로 노출하고 큐 접근자는 기본이 전량 반환이다. **신호는 안 버리고 요약만 자른다.**

## 8.5 점수 비병합의 3중 강제

분석신호가 전표 점수에 새어 들어가지 않도록 세 겹으로 막는다.

1. 점수 레지스트리에서 해당 룰들의 역할계수를 **0.0**으로
2. 케이스 빌더가 이 룰들을 케이스·토픽·우선순위 계산에서 **집합으로 배제**
3. 탐지기가 행 점수를 **리터럴 0.0으로 대입**

한 겹으로 충분해 보이지만, [6장](6_PHASE1-1_RULES.md)에서 본 함정 때문이다 — 레지스트리에서 지우면 폴백이 primary로 잡아 점수가 오히려 붙는다.

배선상 특이점이 하나 있다. 라운드넘버 밀집도와 거래처 신호는 **탐지기를 거치지 않고 케이스 빌더가 직접 계산한다.** 나머지 탐지 결과와 다른 경로다.

## 8.6 폐기한 신호 — 시계열 당기내 집중

"D01/D02는 전기 대비만 보므로, 당기 안에서 특정 시점에 몰리는 것은 별개 차원"이라는 전제로 자기 큐를 하나 더 만들었다. **구현 후 실측에서 이 전제가 무너졌다.**

| 측정 라운드              | finding | 분기말(3·6·9·12월) 비중 | 12월 |
| ------------------------ | ------: | ----------------------: | ---: |
| 계정·연도 자기 리듬 기준 |     864 |               **70.0%** |  307 |
| 분기말/평월 레인 분리 후 |     595 |               **52.4%** |  185 |

폐기 사유가 네 가지다.

**결산 캘린더를 재발견한다.** 당기 내 리듬과 비교하면 "이 계정은 분기말에 바쁘다"를 계정마다 다시 찾아낸다. 감사인이 이미 아는 사실이고 기말 전표 룰이 행 단위로 이미 커버한다.

**레인을 분리해도 안 된다.** 분기말끼리만 비교해도 12월이 최다다(185건). 감가상각·충당금·법인세가 12월에만 얹히므로 **연말 결산은 분기 결산보다 무겁다.**

**원리적 한계.** 한 해에 연말은 한 번뿐이라 "이번 연말이 연말치고도 이상한가"의 비교 대상이 당기 안에 존재하지 않는다. 판정하려면 전기 연말과 비교해야 하는데, 그것은 "당기 내 기준선"이라는 전제와 정면 충돌한다.

**D02와 중복된다.** "작년 같은 달과 비교"로 가면 D02가 이미 하는 일이다. 통계만 바꾼 중복이 된다.

여기서 생성 데이터를 고치는 선택지도 검토했지만 기각했다. 분기·연말 결산 집중은 **실무상 정상**이며 생성기를 고쳐 없애면 데이터가 오히려 비현실적이 된다.

결정: 자기 큐 6종 → **5종**. 시계열은 D02의 드릴다운으로 재정의(새 룰이 아니라 상세 정보)했고 **현재 미구현**이다. 계산 코드는 재사용을 위해 보존하되 배선은 제거했다.

## 8.7 검증 상태 — 이번 라운드 측정 범위 밖

정직하게 밝힐 사실이 있다. **PHASE1-2는 2026-07 검증 라운드의 측정 대상이 아니었다.** 별도 벤치마크 단위로 분리돼 있고, [6장](6_PHASE1-1_RULES.md)·[7장](7_PHASE1-1_COMBO-BUILDER.md)의 판정(29/29, exit 0, 표면화 59.7~61.8%)은 PHASE1-1 룰 표면에 대한 것이다.

또한 **전용 UI가 미구현**이다. 화면 설계(검토 케이스 / 데이터 정합성 / 분석적 검토 소분류)는 문서에 있으나 [11장](11_DASHBOARD.md)에서 보듯 대시보드는 6화면만 배선돼 있다. 분석신호는 생성돼 DuckDB에 적재되지만 전용 표면으로 노출되지 않는다.

## 8.8 실증 예시 — 거래처 한 곳이 신호가 되는 과정

2.3절 전표의 거래처는 `V-000526`(기업다이렉트(유))이다. 이 거래처가 분석신호 경로를 어떻게 통과하는지 따라간다.

**첫등장 판정.** 전기(2022·2023)에 이 코드가 원장에 있었는지 확인한다. 있었다면 첫등장이 아니다. 이 판정은 반드시 3개년 통합 데이터에서 해야 하며, 2023년 파일만 보면 2022년 거래를 못 보고 잘못 첫등장으로 분류한다.

**희소 판정.** 당기 거래 건수를 전체 거래처 분포와 비교해 하위 10분위 이하인지 본다. 2,966개 거래처 중 596개가 여기 해당한다.

**휴면 판정.** 전기에 활동이 있었고 직전 연도에 결번이며 180일 이상 공백이 있었는지 본다. 전체에서 22개만 해당하는 드문 신호다.

셋 중 하나라도 걸리면 이 거래처는 **거래처 자기 큐**에 오른다. 목록은 절단 없이 전수이며, 거래 내용(계정·문서유형)별로 묶이고 합계 금액과 건수 순으로 정렬된다. 동시에 이 거래처의 전표들에는 **배지**가 붙는다 — 전표 등급은 바뀌지 않고 꼬리표만 달린다.

감사인이 보는 것은 두 화면이다. 거래처 목록에서 "이 거래처가 통째로 검토 대상인가"를 보고, 전표 목록에서 "이 전표에 거래처 배지가 있네"를 맥락으로 읽는다. 어느 쪽도 "가공거래처"라고 말하지 않는다. 그 판정은 사업자등록 확인과 실물 대사를 거쳐야 가능하고, 그것은 도구가 아니라 감사인의 절차다.
