## LLM 자유 판정 — 잔여/신규 이슈

### 잔여 (경미, WARNING 수준)

#### W-01. 5시(AM) 스파이크 — 심야 중 1개 시간대만 돌출

- **봤던 것**: hourly_docs — 5시=1,510건 / 0~4시 평균=565건 (2.7배)
- **왜 이상한지**: 다른 심야 시간대는 550~590건으로 균일한데 5시만 돌출. 이전 데이터에서는 4시 스파이크(1,475건). 매번 시간대 하나가 돌출하는 패턴. temporal generator에서 특정 시간대에 편향이 발생하는 것으로 보임.
- **심각도**: 낮음 — 전체의 0.13%

#### W-02. O2C "매출 에누리" 전표가 부채↔부채 이동

- **봤던 것**: O2C 샘플 — header "매출 에누리 - (주)소비자브랜드", GL 200520 Dr / 200550 Cr
- **왜 이상한지**: 매출 에누리면 Dr 매출에누리(4xxx 차감) / Cr 매출채권(1100) 또는 현금. 부채(200520)에서 부채(200550)로 이동은 매출 에누리가 아니라 부채 재분류.
- **심각도**: 중간 — header↔GL 의미 불일치 잔여. 다만 전수 기준 O2C 수익 비율은 75.9%→현재도 유사 수준으로 PASS.

#### W-03. H2R "주식보상비용 인식" 적요에 "광고선전비"

- **봤던 것**: H2R 샘플 — header "주식보상비용 인식", GL 500560, line_text "광고선전비"
- **왜 이상한지**: 주식보상이면 Dr 주식보상비용 / Cr 주식매수선택권(자본). "광고선전비"는 마케팅 비용으로 주식보상과 무관.
- **심각도**: 중간 — header↔line_text 의미 불일치 잔여. 같은 전표의 다른 라인("급여 비용")은 적절.

#### W-04. TRE "차입금 이자 지급" 전표에 이자비용이 대변(Cr)

- **봤던 것**: TRE 샘플 — GL 7100 Cr 1,200,000, line_text "이자비용"
- **왜 이상한지**: "이자 지급"이면 Dr 이자비용(7100) / Cr 현금이 정상. 대변에 이자비용이면 이자비용 감소(환입)인데 header는 "지급". 차변/대변이 뒤집혀 있음.
- **심각도**: 중간 — 분개 방향 오류. 다만 차대변 합계는 균형.

#### W-05. 적요 "미지급" → 부채 외 GL — R3-02 WARNING 없어짐?

- **봤던 것**: linetext_gl_cross."미지급" → 부채=54,642 / 자산=1,255 / 비용=992 / 수익=895
- **계산**: 불일치 = (1,255+992+895+86+82) / 57,952 = 5.7%
- **이전 대비**: 8.7% → 5.7% (개선). R3-02는 현재 WARNING이지만 경계 수준.

### 다각도 추가 분석에서 발견 (신규)

#### A-01. 프로세스별 전표 금액 중앙값이 거의 동일 — 비현실적

- **봤던 것**: H2R=1,410K, A2R=1,401K, R2R=1,396K, TRE=1,380K, O2C=1,378K, P2P=1,356K (차이 3% 이내)
- **왜 이상한지**: 실무에서는 프로세스별 금액 규모가 크게 다름. H2R(급여)은 중앙값 3,000만~5,000만(월급), TRE(자금)은 억 단위(차입/상환), P2P는 수백만~수천만(자재 구매). 모든 프로세스가 같은 LogNormal에서 샘플링되는 것으로 보임.
- **심각도**: 중간 — Phase 2 ML에서 프로세스별 금액 피처가 무의미해질 수 있음

#### A-02. 프로세스별 월말 집중도가 균일 (36~39%) — 비현실적

- **봤던 것**: R2R=39.2%, TRE=37.2%, O2C=36.6%, H2R=36.6%, P2P=36.5%
- **왜 이상한지**: 실무에서는 R2R(결산)만 월말에 극도로 집중 (60~80%). P2P/O2C는 월 전체에 분산 (월말 20~25%). H2R(급여)은 매월 25일 전후 고정. 월말 가중치(month_end_multiplier)가 모든 프로세스에 균일 적용되는 것으로 보임.
- **심각도**: 중간

#### A-03. 작성자 하루 최대 173건 — 극단값 비현실적

- **봤던 것**: 일평균 1.8건 (P99=11건)은 합리적이나, 최대 173건
- **왜 이상한지**: 사람이 하루에 173건 전표를 수동 입력하는 건 물리적으로 불가능. 20~30건이 현실적 상한. 다만 P99=11이므로 99%의 날은 정상.
- **심각도**: 낮음 — 극단값 1건

#### A-04. 작성자 연속 전표 최소 1초 간격 — 비현실적

- **봤던 것**: 중앙값 15.8시간, P1=85초, 최소=1초
- **왜 이상한지**: 사람이 전표 작성 후 1초 만에 다음 전표를 시작하는 건 불가능. SAP에서 전표 하나 입력하는 데 최소 2~3분. P1=85초(1.4분)도 빠르지만 가능. 최소 1초는 시스템 전표가 사람 ID로 잘못 배정된 것일 수 있음.
- **심각도**: 낮음

#### A-05. fraud의 68%가 automated source — 비현실적

- **봤던 것**: automated=3,254건 (68%), manual=1,051건 (22%)
- **왜 이상한지**: 실제 부정은 대부분 사람이 수동으로 실행. 시스템 자동 전표에서 fraud가 68%는 비현실적. anomaly injector가 source 구분 없이 주입하기 때문.
- **심각도**: 중간 — ML에서 source 피처가 역방향 학습될 수 있음

#### A-06. 회사별 금액 중앙값 동일 — 경미

- **봤던 것**: C001=1,378K, C002=1,380K, C003=1,413K (차이 2.5%)
- **왜 이상한지**: HQ(C001, annual_transaction_volume=hundred_k)와 공장(C002/C003, ten_k)의 중앙값이 거의 같음. HQ가 더 다양한 고액 전표를 처리해야 하지만, max에서 C001=5,640억 vs C002/C003=1,000억으로 극단값에서는 차이 있음.
- **심각도**: 낮음

#### A-07. fraud 작성자 AGI015 편중 (98건) — ML leakage 위험

- **봤던 것**: 사람 fraud 1,040건/184명 (평균 5.7건). AGI015만 98건 (평균의 17배).
- **왜 이상한지**: ML이 "AGI015가 작성 → fraud" 패턴을 학습할 수 있음. 나머지 183명은 19~21건인데 1명만 98건이면 created_by 단독으로 분류 가능.
- **심각도**: 중간 — QG2 L1-01 leakage 스캔에서 잡힐 수 있지만 재확인 필요

#### A-08. cost_center 5개 부서 균등 분포 — 비현실적

- **봤던 것**: 비용 GL의 cost_center — CC1000=39,381 / CC2000=39,708 / CC3000=39,518 / CC4000=39,504 / CC5000=39,539 (차이 0.8%)
- **왜 이상한지**: 실무에서는 제조부(CC1000)가 비용의 40~60%를 차지하고, 관리부(CC5000)는 10% 이하. 5개 부서가 20%씩 균등한 건 랜덤 배정의 흔적.
- **심각도**: 낮음 — 탐지 룰에 cost_center 기반 룰 없음. Phase 2에서 개선 가능.

#### A-09. 금액과 라인 수 무상관 — 경미

- **봤던 것**: 금액 Q1~Q5 전부 평균 3.6~3.9라인 (균일)
- **왜 이상한지**: 실무에서는 고액 전표일수록 라인이 많은 경향 (복합 거래, 세금 라인 추가). 금액과 라인 수 사이 상관관계가 0에 가까움.
- **심각도**: 낮음

#### A-10. 영어 line_text가 정상 전표에 0.2% 혼재 — 경미

- **봤던 것**: 영어 line_text 2,724행 중 정상 2,138행 (78.5%). "Clearing entry — period-end reconciliation" 554건, "GR/IR Clearing — PO match complete" 348건 등
- **왜 이상한지**: 다른 적요는 전부 한글인데 일부만 영어. SAP 표준 텍스트가 번역 안 된 것으로 보임. 감사인이 보면 "이 전표만 왜 영어?"라고 질문.
- **심각도**: 낮음 — 0.2%, C06 탐지에서 영어 키워드로 잡힐 수 있음

#### A-11. 같은 회사 내 reference 중복 187건 — 경미

- **봤던 것**: 중복 reference 20,920건 중 회사 간 중복 20,733건 (각 회사 독립 채번이므로 정상), **같은 회사 내 중복 187건**
- **왜 이상한지**: SAP에서 reference(Belegfeld)는 외부 문서 번호로, 같은 회사 내에서 고유할 필요는 없지만 같은 PO/SO 번호가 여러 전표에 있으면 중복 의심. 다만 입고/송장/지급이 같은 PO 참조하는 건 정상.
- **심각도**: 낮음 — 187건, 대부분 정상 프로세스일 가능성
