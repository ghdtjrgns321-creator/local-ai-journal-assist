## What Changed

`L3-01` is now documented and implemented as:

1. `process_denied_accounts` first
2. conservative `process_disallowed_categories` fallback only when exact deny accounts are not configured for that process
3. optional `process_allowed_keywords` suppression for auditor-confirmed normal exceptions
4. still `review_needed`, not a confirmed-error rule

Default fallback categories were reduced to:

- `O2C -> expense`
- `P2P -> revenue`
- `H2R -> revenue`
- `TRE -> inventory`
- `A2R -> payroll`

## Why

The v29 candidate labels are much closer to Phase 1 intent than the old synthetic set, but most false positives still came from broad category logic rather than clearly wrong account codes.

Observed dominant false-positive buckets in the broad version:

| process | category | false-positive docs |
| --- | --- | ---: |
| P2P | revenue | 10,077 |
| TRE | expense | 3,909 |
| H2R | revenue | 3,184 |
| O2C | expense | 2,808 |
| A2R | revenue | 1,764 |

The synthetic injected cases themselves were concentrated in a much narrower set:

| process | injected target category | label docs |
| --- | --- | ---: |
| O2C | expense | 23 |
| P2P | revenue | 17 |
| H2R | revenue | 7 |
| TRE | inventory | 7 |
| A2R | payroll | 4 |

## Operational Conclusion

- Keep `L3-01` as a Phase 1 review screen.
- Use broad category mismatch only as a conservative fallback.
- Manage `process_denied_accounts` per client CoA as the main precision control.
- Keep `R2R` out of the default L3-01 process scope; it needs separate handling.
