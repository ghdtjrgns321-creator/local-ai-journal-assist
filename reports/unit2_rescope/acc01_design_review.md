# ACC01 게이트 설계 검토 — 게이트가 틀렸다

측정일: 2026-07-15 · 지시: 사용자 ("돌리되 ACC01 게이트도 같이 의심해라")
대상: `tools/scripts/normal_realism_account_checks.py:81-108` `acc01_tax_lines_use_tax_accounts`

## 결론

**ACC01은 정상 회계를 위반으로 잡는다. 게이트를 고쳐야 하고, 계약서 820fd827의 `ACC01 64,713→0`은 개선이 아니라 세금 모듈이 꺼진 것을 잰 수치다.**

## 게이트가 요구하는 것

```python
tax_amt = pd.to_numeric(df["tax_amount"], errors="coerce")
tax_rows = df[tax_amt.fillna(0) > 0]              # 세액이 붙은 모든 행
...
violations = int((~gl.isin(tax_accounts)).sum())   # 세금 계정이 아니면 위반
return ("PASS" if violations == 0 else "FAIL"), metric
```

docstring: *"세액이 붙은 라인은 세금계정에 계상돼야 한다 (부가세법상 대급금/예수금 전용계정)"*

즉 **`tax_amount > 0`인 행은 예외 없이 세금 계정이어야 한다.**

## 반증 1 — 실무 ERP는 그렇게 안 쓴다

매출 세금계산서 전표:

```
차변 1100 매출채권       1,100,000
  대변 4000 매출           1,000,000   ← 공급가액·세액·세금코드가 이 줄의 속성
  대변 2110 부가세예수금     100,000
```

세액은 **그 세액을 발생시킨 거래 줄**에 붙는다. 어느 줄에서 나온 세금인지 모르면 세액을 계산할 수 없다. 부가세 계정 줄은 거기서 파생된 별개 줄이다.

## 반증 2 — 현재 채택된 정상 데이터(r6)가 이 게이트를 불합격한다

| 데이터셋                     | 세액>0 행이 붙은 계정 (상위)                                                                                                     |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **r6 (채택된 normal)**       | `500090` 매출원가 28,262 · `4000` 매출 20,753 · `100030` 8,353 · `6900` 수수료 3,550 · `2000` 매입채무 2,069 — **세금 계정 0개** |
| prod_single (세금 모듈 꺼짐) | `1160` 부가세대급금 46,651 · `2110` 부가세예수금 24,847 — **오직 세금 계정**                                                     |
| clean_out (세금 모듈 켜짐)   | `2110` 56,149 · `4100` 매출 39,620 · `4000` 매출 39,253 · `1160` 14,407 · `6700` 5,031 · `6400` 4,930 — **양쪽 다**              |

**r6은 세액을 거래 줄에만 붙인다(실무 방식).** ACC01은 r6을 **64,714건 위반**으로 판정한다. 현재 정상 데이터가 통과 못 하는 게이트다.

## 반증 3 — 제품 룰은 어느 쪽이든 안 깨진다

제품에 부가세 검증 룰이 있다 — `src/detection/evidence_rules.py:282-286`:

```python
# ── S2: 부가세 검증 ──
if "supply_amount" in df.columns and "tax_amount" in df.columns:
    supply = df["supply_amount"]
    tax = df["tax_amount"]
    # Why: 면세/영세율 거래(tax_amount=0 또는 NaN)는 검증 대상 제외
```

**이 룰은 `supply_amount`와 `tax_amount`가 같은 줄에 있어야 성립한다.** 그리고 공급가액은 부가세 계정 줄이 아니라 거래 줄의 속성이다.

실측 — 두 컬럼이 같은 줄에 있는 행과 그 비율:

| 데이터셋               | 둘 다 있는 줄 | **세액 / 공급가액 중앙값** | 판정                  |
| ---------------------- | ------------: | -------------------------: | --------------------- |
| **r6 (채택된 normal)** |        64,539 |                 **0.0412** | **틀림** (10%여야 함) |
| prod_single            |        71,498 |                 **0.1000** | 정확                  |
| clean_out              |       168,440 |                 **0.1000** | 정확                  |

**부수 발견: r6의 부가세가 공급가액의 4.12%다.** 생성기는 정확히 10%를 만드는데 v42 덧칠이 깨놨다. 제품의 부가세 검증 룰이 r6에서 계산 오류를 터뜨린다. ACC01이 잡았어야 할 진짜 결함은 이쪽인데, ACC01은 세액의 **위치**만 보고 **금액**을 안 본다.

## 반증 4 — prod_single이 통과한 이유

`prod_normal.yaml`에 `tax:` 섹션이 없어 세금 모듈이 꺼졌다(같은 병: `docs/debugging.md:680`). 모듈이 꺼지니 `tax_amount`가 `1160`/`2110` 줄에만 남았고, ACC01은 그 상태를 "정상"으로 통과시켰다.

|                               | prod_single (세금 꺼짐) | clean_out (세금 켜짐) |
| ----------------------------- | ----------------------: | --------------------: |
| ACC01 검사 모집단 (세액>0 행) |                  71,498 |   **168,990** (2.4배) |
| ACC01 위반                    |                       0 |          98,282 (58%) |

**Unit 1은 세금 없는 데이터에서 게이트를 만들고 그 데이터로 검증했다.**

## 계약서 820fd827의 해당 문장

> `ACC01 세금라인 0/69,300` … r6 대비: **ACC01 64,713→0**

**뒤집힌다.** r6이 실무 방식(거래 줄에 세액)을 쓰고 있었고, 새 데이터는 세금 모듈이 꺼져 세액이 세금 계정에만 남아 "0"이 된 것이다. 개선이 아니라 기능 상실이다.

## 수정안

ACC01이 잡으려던 진짜 위험 — *"세액이 붙었는데 부가세 계정이 아예 없는 전표"* (부가세 미계상) — 은 유효하다. 다만 **행 단위가 아니라 전표 단위**로 봐야 한다.

| 항목 | 현행                            | 수정안                                                                           |
| ---- | ------------------------------- | -------------------------------------------------------------------------------- |
| 단위 | 행 (`tax_amount > 0`인 모든 행) | **전표** (`document_id`)                                                         |
| 규칙 | 그 행이 세금 계정이어야 함      | **세액이 붙은 전표에 부가세 계정 줄이 최소 1개 존재해야 함**                     |
| 금액 | 안 봄                           | **`tax_amount ≈ supply_amount × 세율` (면세/영세율 제외)** — r6의 4.12%를 잡는다 |

이러면 r6의 진짜 결함(부가세 4.12%)을 잡고, clean_out의 정상 거래는 통과시킨다.
