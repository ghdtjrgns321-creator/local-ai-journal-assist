# 금액 스케일 수정 결과 — 설정 유실이 원인이었다

측정일: 2026-07-15 · Rust 코드 변경 **0줄**

## 결론

**"생성기가 금액을 1,223배 작게 만든다"는 생성기 갭이 아니라 설정 유실이었다.** 프로젝트 설정(`config/datasynth.yaml`)으로 재생성하니 r6과 같은 대역이 나온다.

**단, 손익 비율(원가율·판관비율)은 설정으로 안 고쳐진다. 그건 진짜 생성기 갭이다.**

## 근본 원인

`tools/datasynth/crates/datasynth-core/src/distributions/amount.rs:47-63`

```rust
impl Default for AmountDistributionConfig {
    fn default() -> Self {
        // WU-DS-02: 기본값은 USD 기준. KRW는 datasynth.yaml에서 오버라이드.
        // je_generator는 항상 with_config()로 YAML 값을 전달하므로,
        // 이 기본값은 단위 테스트나 직접 생성 시에만 사용됨.   ← 이 가정이 깨졌다
        Self {
            min_amount: 0.01,
            max_amount: 100_000_000.0,
            lognormal_mu: 7.0,         // USD: center around ~1000
            decimal_places: 2,
            round_number_unit: 100.0,  // USD: round to nearest 100
            ...
```

YAML에 `transactions.amounts` 섹션이 **없으면** serde 가 이 Default 를 쓴다. 주석의 "je_generator는 항상 YAML 값을 전달"이라는 가정은 **섹션이 있을 때만** 참이다.

|                     |            기본값 (USD) | `config/datasynth.yaml:178-188` (KRW) |
| ------------------- | ----------------------: | ------------------------------------: |
| `lognormal_mu`      | **7.0** (`e^7 = 1,097`) |     **14.0** (`ln(1,200,000) ≈ 14.0`) |
| `min_amount`        |                    0.01 |                                   100 |
| `max_amount`        |             100,000,000 |                       100,000,000,000 |
| `decimal_places`    |                       2 |                0 (원화는 소수점 없음) |
| `round_number_unit` |                     100 |                        10,000 (1만원) |
| `nice_number_unit`  |                       5 |                          1,000 (천원) |

**mu 차이 = 7.0 정확히. `e^7 = 1,097`.** 실측 배율 1,223배(r6 대비)와 정합하며, prod_single 전표 중앙값 **1,000원** = 기본값 주석의 "center around ~1000" 그대로다.

## 남은 것 — 손익 비율은 설정으로 안 고쳐진다

검증기 `_company_year_pnl`(`:639-748`) 과 동일 로직으로 재현. 산출: `<scratchpad>/m11_after.py`

**매출은 연 4,584억~5,104억으로 중견 제조업 대역에 들어왔다.** 비율은 개선됐으나(판관비율 4.5 → 1.3, 영업이익률 −3.86 → −0.47) 여전히 전부 위반.

비율은 스케일 불변이므로 금액 배율로는 안 움직인다. 원인은 별개다:

**`target_gross_margin` 이 죽은 설정이다.** `config.balance.` 전수 검색 결과 소비처가 `datasynth-config/src/validation.rs:551` 의 `validate_rate("target_gross_margin", ...)` — **0~1 범위 검사만 하고 버린다.** je_generator 는 이 값을 읽지 않는다. `target_dso_days`·`target_dpo_days`·`target_current_ratio`·`target_debt_to_equity` 도 동일(`config.balance.` 검색 히트 0).

**즉 아무것도 원가율을 겨냥하지 않는다.** 손익은 시나리오별 행 개수의 부산물이다. r6 의 멀쩡한 비율은 v42 프로파일이 `write_journal`(`normal_coa_v30.rs:348`) **이전에** 원장을 40여 회 변형(`:307-347`)한 결과다 — 사후에 회사인 척 만드는 작업.

**사용자 결정(2026-07-15): 금액 스케일만 고치고 비율은 보류.** 근거 — 비율에 물린 제품 소비처가 없다(`amount_scale_rules.md` 참조). 남은 사유는 포트폴리오 신뢰성뿐이며 별건으로 판단.
