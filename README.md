# Local AI Audit Assistant

감사 전표 테스트 자동화 도구. 전표 35만 건에서 **감사인이 먼저 볼 검토 큐**를 만든다.

> **Streamlit 화면 개발 중🛠️.**

---

## 개요

1. **생성** — 회계적으로 성립하는 합성 원장을 만든다 (K-IFRS 한국 중견 제조업)
2. **탐지** — 결정론 룰 29종이 전표 단위 규칙 위반을 명명해 올린다
3. **표면** — 감사인이 조합을 선택하면 그에 맞는 전표를 정확히 집계한다
4. **보조** — 계정·거래처 단위 분석 신호와 비지도 모델이 별도 표면으로 붙는다

```
   원장 CSV
       │
       ▼
 ┌─ 수집 · 파생 · 검증 　　───────────────────────────────────┐
    헤더탐지 ──▶ 컬럼매핑 ──▶ 타입정제 ──▶ 파생변수 ──▶ 구조검증
    시작행    유사도 3구간   금액·날짜    금액/시간/패턴  통과못하면 중단
 └────────────────────────────┬──────────────────────────────┘
                              ▼
 ┌─ 탐지 · 세 표면 비병합 ──────────────────────────┐

    룰 29종            분석적 검토 신호        비지도 모델
    전표 한 장          계정·거래처를 모아야    정상 분포 밖
    규칙 위반           보이는 신호             비정형
       │                     │                     │
       ▼                     ▼                     ▼
    조합 빌더            자기 큐 5종            검토 후보 제시
    몸통10 × 특징10       + 맥락 배지            (성능 미검증)
    커버리지 큐
    데이터정합성 패널

 └────────────────────────────┬────────────────────┘
                              ▼
                    회사·연도별 DuckDB
                              ▼
                    감사인이 조합을 고르고 판단한다
```

시스템은 위험 등급을 선언하지 않는다. 어떤 조합을 볼지는 감사인이 고르고, 도구는 그 조합에 맞는 전표를 세어 근거와 함께 보여준다.

## 1. 문제 인식

전표 수만 건에서 감사인이 먼저 볼 큐를 만들고 싶었다. 처음에는 금융감독원 감리 지적사례로 룰을 만들고, 이진 분류와 비지도 모델로 이상치를 탐지할 생각이었다.

외부감사에서 전표 테스트는 선택이 아니다. PCAOB AS 2401과 ISA 240은 경영진의 통제 무력화 위험 때문에 전 기간 분개를 대상으로 부정 위험이 높은 항목을 선별해 검사하도록 요구한다.

문제는 규모다.

중견 제조업 한 곳의 3개년 원장이 35만 행을 넘는데, 감사인이 실제로 열어볼 수 있는 분량은 수십 건이다.

### 통상적 접근의 한계

| 구분      | 통상적 접근                    | 본 시스템                           |
| --------- | ------------------------------ | ----------------------------------- |
| 표본 선정 | 금액 상위 N건                  | 처리 방식(수기·승인·시각) 축을 함께 |
| 위험 판정 | 도구가 등급을 자동 부여        | 등급 비선언 — 어휘만 제공           |
| 근거      | 점수의 산출 과정이 불투명      | 어떤 룰이 왜 걸렸는지 표기          |
| 신호 통합 | 여러 신호를 가중합해 단일 순위 | 세 표면 독립, 점수 비병합           |
| 완료 판정 | "검증 완료" 서술               | 스크립트 종료 코드                  |

금액 상위 100건을 뽑으면 고액 정상 거래가 목록을 채우고, 소액으로 쪼갠 은폐는 통째로 빠진다. 실제 부정은 금액이 아니라 처리 방식에 흔적을 남기는데 그 축은 정렬 기준에 들어오지 않는다.

### 등급을 선언하면 그 등급을 방어해야 한다

"이 전표는 HIGH 위험"이라고 도구가 말하는 순간, 감사인은 왜 HIGH인지 설명할 수 있어야 한다. 설명의 근거가 "우리 합성데이터에서 드물었다"뿐이라면 감사 증거가 아니다.

이 프로젝트는 실제로 등급 체계를 만들었다가 근거가 무너지는 것을 확인하고 폐기했다(5장).

## 2. 전체 파이프라인 개요

```
 ━━━ [1] 생성 : Rust 생성기 — 별도 실행, 원장 CSV 산출 ━━━

    설정 17섹션 ──▶ 마스터데이터 ──▶ 문서흐름 ──▶ 전표 생성 ──▶ 출력
                    거래처·직원      P2P/O2C      시나리오 20종   73컬럼
                                                       │
                                          ┌────────────┴────────────┐
                                          ▼                         ▼
                                   정상 base                   부정 overlay
                                   355,786행                   14수법 × 4벌
                                   부정 라벨 0                  밀도 0.29%

 ━━━ [2] 수집 : ERP마다 다른 원장을 표준 형태로 ━━━

    파일검증 ──▶ 헤더탐지 ──▶ 컬럼매핑 ──▶ 타입정제
                 신뢰도       ┳ 정확일치 → 자동 확정
                 가중합       ┣ 유사도   → 감사인 확인
                              ┗ 미매핑   → 목록 표시

 ━━━ [3] 파생 · 검증 : 룰이 읽을 값을 만들고 쓸 만한지 판정 ━━━

    파생변수 ◆ 금액축   계정 그룹 내 이상 정도 (표본 부족하면 계산 안 함)
             ◆ 시간축   근무시간·주말·심야·결산기 보정
             ◆ 패턴축   가계정 여부·둥근 금액·첫자리 숫자
                              │
                              ▼
    구조검증  ─┬─ 통과 ──▶ 탐지로
               └─ 실패 ──▶ 중단 (필수 컬럼·타입이 깨지면 결과가 무의미)
                              │
    회계검증  ─┬─ 경미 ──▶ 경고만 남기고 진행
               └─ 치명 ──▶ 중단 (차대 차액이 금액의 1% 또는 전표의 10% 초과)

 ━━━ [4] 탐지 : 탐지기 5종 병렬 — 결과는 입력 순서로 재정렬 ━━━

    정합성 ─┐
    부정징후 ┤
    이상치 ─┼──▶ 룰별 발화 ──▶ 점수 정규화 ──▶ 증거 단위 구성
    분포   ─┤                  신호강도×심각도    전표 단위 / 흐름 단위
    증빙   ─┘                  ×증거강도×역할     (역분개·중복은 짝이 증거)

 ━━━ [5] 표면 : 세 갈래로 나뉘어 화면에 오른다 ━━━

    ◆ 조합 빌더        몸통(조작 대상) × 특징(부정 전표의 모양)
    ◆ 커버리지 큐      룰별 전수 목록 — 빌더 어휘 밖 룰은 여기서만
    ◆ 데이터정합성     차대 불일치·필수필드 누락 — 부정 신호가 아니라 품질 문제
```

각 단계는 회사·연도별로 격리된 DuckDB에 적재된다. 회사가 다르면 승인 한도가 다르므로, 같은 원장이라도 판정 결과가 갈린다.

## 3. 기술 설명

### 3-1. 합성데이터

실제 기업 원장은 구할 수 없다. 회계적으로 성립하는 원장을 생성해 쓰되, 생성기가 만든 데이터가 "말이 되는지"를 따로 검증하는 체계를 뒀다.

기반은 오픈소스 Rust 생성기 **EY-ASU DataSynth**(Apache 2.0)다. 이를 K-IFRS 한국 중견 제조업 대상으로 개조했다.

| 개조 항목      | 내용                                                    |
| -------------- | ------------------------------------------------------- |
| 통화·금액 분포 | 원화 기준 로그정규 분포로 교체                          |
| 전결 규정      | 한국 중견 제조업 6단계 승인 한도                        |
| 근무 시간      | 오전 피크·점심 감소·퇴근 전 러시, 결산기 야근           |
| 계정 체계      | 계정결정표 신설 — 거래 의미에서 계정을 결정론으로 도출  |
| 거래 종류      | 대손상각·퇴직급여·개발비 자본화 등 저빈도 시나리오 추가 |
| 단일 법인      | 3개 법인 생성 후 본사만 투영                            |

#### 정상 데이터 생성 원칙

정상은 정상이어야 한다. 의도적 이상이나 부정을 섞지 않는다.

| 원칙               | 실제 적용                                       |
| ------------------ | ----------------------------------------------- |
| 부정 라벨 0        | `is_fraud`·`is_anomaly` 전량 거짓               |
| 실무 예외는 남긴다 | 자기 결재 0.11%, 승인 한도 초과 0.03%           |
| 품질 노이즈는 균일 | 승인 공란 0.7%·적요 공란 0.8%를 라벨과 무관하게 |
| 사후 변형 금지     | 생성 시점에 옳게 만들고 나중에 고치지 않는다    |

세 번째가 중요하다. 노이즈가 라벨과 상관되면 모델이 "적요가 비면 부정"이라는 지름길을 학습한다.

네 번째는 이 프로젝트가 큰 대가를 치르고 얻은 원칙이다(5장).

#### 실무 예외를 0으로 만들지 않는 이유

정상 데이터에 자기 결재가 한 건도 없으면 자기승인 룰을 검증할 수 없다. 실무에서도 전결 규정에 따른 자기 결재는 실제로 있고, 감사인이 봐야 할 항목이다.

승인권도 의도적으로 설계했다. 관리자급부터만 승인권을 주면 한도 초과 승인이 10억 이상 문서에서만 가능해 구조적으로 사라진다.

그래서 선임급 이상 전원에게 승인권을 부여했다.

#### 부정 데이터 — 무엇을 섞었나

금융감독원 감리 지적사례에서 실제 수법을 가져와 14개로 정리했다. 정상 base는 그대로 두고 별도 파일로 얹는다.

| ID   | 수법            | 심각도 | 프로세스 |
| ---- | --------------- | ------ | -------- |
| FS01 | 가공매출        | 상     | 판매     |
| FS02 | 진행률 조작     | 상     | 결산     |
| FS03 | 현금 횡령       | 상     | 구매     |
| FS04 | 횡령 은폐       | 상     | 구매     |
| FS05 | 순환거래        | 상     | 판매     |
| FS06 | 부채 누락       | 상     | 구매     |
| FS07 | 재고 과대       | 중     | 구매     |
| FS08 | 부적절 자본화   | 중     | 결산     |
| FS09 | 컷오프 조작     | 중     | 판매     |
| FS10 | 대손 회피       | 상     | 판매     |
| FS11 | 특수관계자 남용 | 상     | 관계사   |
| FS12 | 충당금 누락     | 중     | 결산     |
| FS13 | 손상 회피       | 상     | 관계사   |
| FS14 | 유령직원 급여   | 중     | 급여     |

각 수법은 3~4단계로 심긴다. 가공매출이라면 가공 매출 계상 → 가짜 회수 → 매출채권 할인 차입 → 차기 역분개다.

부정이 전표 한 장이 아니라 **여러 장에 걸친 이야기**로 심기는 것이 핵심이다. 그중 일부는 그 자체로 아무 규칙도 위반하지 않는 다리 역할이다.

#### 지름길 차단

탐지기가 구조가 아니라 흔적을 보고 맞히는 것을 막는 장치를 겹으로 걸었다.

| 장치             | 막는 것                                    |
| ---------------- | ------------------------------------------ |
| 라벨 컬럼 제거   | 원장에서 부정 표시 컬럼 삭제               |
| 헤더 정규화      | 수법 이름이 적요에 노출되는 것             |
| 계정 지름길 폐기 | 부정 전용 신규 계정 도입 금지              |
| 소스 분포 노이즈 | 입력 경로만 보고 라벨을 맞히는 것          |
| 관계 증거 정책   | 직원-거래처 관계가 원장 필드에 드러나는 것 |

계정 항목은 뒤늦게 추가됐다. 초기에는 부정용으로 6자리 확장 계정을 새로 만들었는데, 그러면 "본 적 없는 계정이면 부정"이라는 지름길이 생긴다.

정상이 많이 쓰는 22개 계정으로 전면 교체했다.

#### 생성 결과

| 항목          | 정상 base               | 부정 overlay     |
| ------------- | ----------------------- | ---------------- |
| 규모          | 355,786행 / 113,135문서 | 각 356,446행     |
| 컬럼          | 73                      | 73               |
| 부정 문서     | 0                       | 330 (밀도 0.29%) |
| 시드          | 단일                    | 4벌              |
| 지름길 게이트 | —                       | 16종 전부 통과   |

#### 데이터가 말이 되는지 검증하는 3겹

```
  1겹  현실성 게이트 57종
        복식부기·부가세율·승인권·노이즈율·관계사 방향 등 자동 검사
        실패는 3분류 : ①데이터 결함 ②게이트 결함 ③제품 무관 → ①만 수정

  2겹  룰 발화율 대역
        정상에서 룰이 폭주하면 데이터가 비현실적이라는 신호
        게이트가 못 보는 축을 룰이 실질적으로 심판

  3겹  부정 표면화 실측
        심은 수법이 검토 목록에 오르는가 — 최종 증거
```

게이트만으로는 부족하다는 것을 인정한 구조다. 현실성 항목 146개 중 게이트로 구현한 것은 49개(34%)뿐이다.

게다가 구현된 57종 중 네 개는 **어떤 데이터로도 통과하는 항등식**이었고, 부가세 게이트는 판정 방향이 반대였다. 그래서 통과 개수를 늘리는 대신 무의미한 게이트를 참고 등급으로 강등했다.

| 게이트 판정  | 개수 |
| ------------ | ---- |
| 통과         | 37   |
| 실패         | 7    |
| 판정 불가    | 9    |
| 참고(항등식) | 4    |

남은 실패 7건은 전부 원인이 파악된 보류 항목이다.

### 3-2. 룰 엔진

#### 룰을 어떻게 만들었나

금융감독원 회계포탈의 감리 지적사례 **189건**을 수집해 본문을 직접 읽고 전표 조작 여부를 분류했다. 2011년부터 2025년 상반기까지의 사례이며 원문은 한글·PDF 문서다.

여기서 나온 조작 수법을 전표에서 볼 수 있는 신호로 번역했다. 감사기준서 240의 부정 전표 특징 목록과 대조해 근거를 이중으로 확인했다.

번역이 안 되는 것도 있었다. 감리 문서는 "무엇이 조작됐나"는 쓰지만 "전표를 어떻게 입력했나"는 쓰지 않는다.

#### 주제별 룰 구성

룰은 여섯 개 주제로 묶인다. 주제는 감사인이 던지는 질문의 종류에 대응한다.

**① 데이터 정합성** — 이 원장을 분석해도 되는가

| 룰    | 잡는 것                    | 정상에서의 발화 |
| ----- | -------------------------- | --------------- |


모집단 단위 신호는 역할 계수가 0이라 전표 점수에 기여하지 않는다. 여기에 함정이 하나 있다.

### 3-4. 분석적 검토 신호

전표 한 장만 봐서는 안 보이고 **계정·거래처·작성자를 모아야** 드러나는 신호다. 근거는 감사기준서 520의 분석적 절차다.

```
  ◆ 자기 큐 (별도 목록을 가진다)
     첫자리 분포 · 활동량 급변 · 월분포 변화 · 둥근금액 밀집도 · 첫등장 거래처
     "이 계정이 통째로 이상하다"를 독립 표면으로 올림
     전표 점수에는 기여 0

  ◆ 배지 (전표 줄에 붙는 맥락 꼬리표)
     시간대 집중 · 배치 이상 · 업무범위 + 둥근금액 단건 · 첫등장 거래처 전표
     정상 업무에 흔해서 단독 목록은 노이즈
     전표 등급을 바꾸지 않음
```

둥근 금액과 거래처 신호는 양쪽 모두다. 모집단 수준의 쏠림은 신호이고 개별 건은 흔한 일이기 때문이다.

#### ① 첫자리 분포 (벤포드)

금액의 첫째 자리 숫자 분포를 이론 분포와 비교한다. 계정·월 단위로 검정하고 전표 단위로 내려보내지 않는다.

| 판정        | 기준                    |
| ----------- | ----------------------- |
| 적합        | 평균절대편차 0.012 이하 |
| 중간 부적합 | 0.012 ~ 0.015           |
| 강한 부적합 | 0.015 초과              |
| 검정 제외   | 표본 500건 미만         |

임계는 포렌식 회계의 사실상 표준을 따랐다. 표본이 적은 그룹은 오차가 쉽게 커져 후보를 대량 생성하므로 제외한다.

#### ② 활동량 급변

계정별 거래 활동량을 전기와 비교한다. 금액 합계·건수·평균을 가중 합산해 판정한다.

전기 데이터가 필요하므로 **기본 설정에서는 꺼져 있다**. 전년도 감사 데이터가 있는 기존 고객사에서만 켠다.

#### ③ 월분포 변화

계정의 월별 거래 분포가 전기와 얼마나 달라졌는지 본다. 두 분포의 거리를 재는 방식이다.

잡는 신호는 "전기엔 고르게 발생하던 계정이 당기엔 특정 달에 몰리는 경우"다. 최소 3개월·문서 100건 등 네 겹의 조건을 통과해야 신호가 된다.

#### ④ 둥근 금액 밀집도

부자연스럽게 둥근 금액이 한곳에 몰렸는지 본다. 근거는 감사기준의 "끝자리 0이 다수인 금액" 문언이다.

판정에서 두 가지가 중요하다.

**기준선을 데이터에서 뽑는다.** 둥근 금액의 정상 비율은 산업마다 다르므로 상수를 박지 않고 해당 원장 전체의 비율을 기준으로 삼는다.

**표본이 크면 통계적 유의성만으로 부족하다.** 건수가 많으면 사소한 차이도 유의해지므로 효과 크기 하한을 함께 건다.

정의도 한 번 갈아엎었다. 구 정의는 "100만원의 배수"였는데 원장 금액이 8자릿수에 걸쳐 있어 단일 절대 단위로는 잴 수 없었다.

거래의 64.9%가 100만원 미만이라 구조적으로 판정 대상에서 빠졌고, 38만 행 중 141행만 걸려 검사기가 **어느 데이터에서도 발화하지 못했다**. 지금은 유효숫자 기준의 상대 정의를 쓴다.

#### ⑤ 첫등장·희소 거래처

가공거래처는 감사기준·감리사례·부정조사 3축 모두에서 고위험이다. 그러나 이를 가르는 강한 신호는 거래처 마스터가 있어야 보인다.

본 프로젝트는 원장 단일 소스로 한정하고 마스터를 의도적으로 두지 않는다. 그래서 원장 집계만으로 만들 수 있는 **약한 근사**만 올린다.

| 신호         | 원장에서 만드는 법           | 한계                       |
| ------------ | ---------------------------- | -------------------------- |
| 원장 첫 등장 | 전기 미등장 × 당기 등장      | 정상 신규 공급처도 해당    |
| 원장 희소    | 거래 건수 하위 10분위        | 정상 일회성 거래와 모호    |
| 휴면 재활성  | 전기 활동 후 180일 이상 공백 | 정상 거래 재개와 구분 불가 |

명명 규율을 뒀다. **"신규 거래처"가 아니라 "원장 첫 등장"으로 부른다.**

마스터 등록일이 아니라 우리 원장에 처음 찍힌 날을 보는 것이므로, "신규 거래처"라 부르면 갖고 있지 않은 데이터를 가진 것처럼 오해를 준다.

실측하면 3개년 통합 거래처 2,966개 중 첫등장 568개(19%)·희소 596개·휴면 22개, 합집합 625개(21%)가 신호가 된다.

#### 검토 목록을 소유하지 않는다

이 표면은 신호만 만들고 무엇을 몇 개 보여줄지는 화면이 정한다. 그래서 생성 경로에서 잘라내지 않는다.

과거에는 상위 100건만 남겼는데 제거했다. 신호마다 점수 척도가 달라 한 리스트를 자르면 척도가 큰 종류가 칸을 독식하고, 백엔드가 버린 신호는 화면이 되살릴 수 없기 때문이다.

### 3-5. 비지도 모델 (VAE)

#### 왜 비지도인가

합성데이터에서 지도학습은 순환 논증이다. 부정 라벨은 우리가 규칙으로 심은 것이므로 모델이 학습하는 것은 부정이 아니라 **우리의 주입 규칙**이다.

실제로 측정했다. 단일 컬럼만으로 판별력 0.95 이상이 18건, 시뮬레이션 평균 0.9978이 나왔다.

비지도는 라벨을 쓰지 않으므로 이 함정을 피한다. 정상 분포만 학습하고 그 밖을 표시하는 방식이라 합성데이터에서도 유효할 것으로 봤다. **결과적으로 이 기대는 검증되지 않았다.**

#### 라벨 누수 차단

```
  ◆ 1차   컬럼 차단 목록 — 최종 52개 (누적, 해제 없음)
           라벨 8 · 합성 전용 18 · 판별력 높은 컬럼 15 · 감사 파생 37

  ◆ 2차   이름 패턴 차단 — label, fraud, risk, score, prediction 등 11개 토큰

  ◆ 3차   물리적 제거 — 8개 컬럼을 데이터프레임에서 아예 삭제
           신규 룰이 실수로 참조해도 즉시 실패하도록
```

금액 컬럼은 전량 차단돼 있다. 사유는 "합성 금액이 너무 깨끗해 지름길 학습을 유발"이다.

차단 목록은 **확장만 허용하고 성능을 올리려는 목적의 완화는 금지**한다는 원칙을 명문화했다.

#### 학습 구성과 1차 관문

정상 base에서 학습하고, 학습에 쓰지 않은 정상 구간의 표시율이 설정값 근처인지 먼저 확인했다.

| 항목               | 값                     |
| ------------------ | ---------------------- |
| 학습 표본          | 50,000행 (라벨 미사용) |
| 피처 수            | 71                     |
| 은닉·잠재 차원     | 64 / 32                |
| 학습 반복          | 50회                   |
| 재구성 손실        | 0.2300                 |
| 정보량 손실        | 0.3718                 |
| 미학습 정상 표시율 | 1.056%                 |
| 설정 오염률        | 1.00%                  |

사전 선언 대역 안이라 통과다. 모델이 정상을 정상으로 보고 있다는 뜻이다.

#### 2차 관문 — 실패

부정 overlay 4벌 전 행을 채점하고 문서 단위로 평가했다.

| 지표                                | 값                        |
| ----------------------------------- | ------------------------- |
| 문서 판별력(AUROC)                  | **0.5187** (무작위 ≈ 0.5) |
| 상위 1% 회수율                      | 0.0038                    |
| 룰이 못 올린 부정 중 모델이 건진 것 | 4벌 합계 2건              |

**판정: 행 단위 비지도 탐지 검증 실패.** 수치를 완화해 서술하지 않는다.

#### 왜 실패했나 — 합성데이터의 한계

```
  흔적을 심는다   ──▶  자기가 심은 것을 자기가 찾음 (자문자답)
                          ↓ 성능 증거 아님

  흔적을 안 심는다 ──▶  정보가 없어 못 잡음 (본 실측)
                          ↓ 성능 증거 아님
```

본 overlay는 지름길 게이트 16종으로 부정 행의 개별 피처를 전부 정상과 동일하게 강제한 완전 위장 시뮬레이션이다. 행 단위 피처에 구분 정보가 **존재하지 않는다**.

따라서 무반응은 어떤 모델을 써도 피할 수 없는 결과이며, 시드 4벌에서 일관되게 재현됐다. 모델 결함이 아니라 정보론적 귀결이다.

과거 수치도 무효화했다. 구세대 데이터에서 얻었던 판별력 0.93은 **흔적을 심은 자문자답 수치**였음을 명시했다.

#### 같은 전표를 룰은 왜 잡나

부적절 자본화 수법의 부정 전표를 보면 금액은 정상 분포 안, 계정은 정상이 많이 쓰는 것, 시각은 근무시간, 적요는 정규화된 문구다. 모든 개별 축에서 정상과 구분되지 않는다.

비지도 모델은 이 행을 잘 복원하므로 이상 점수가 낮다. 반면 룰은 "비용 성격 지출이 자산 계정에 기표됐다"는 **관계**를 본다.

부정은 개별 값의 이상함이 아니라 맥락의 부적절함으로 나타난다. 그 맥락을 아는 것은 학습이 아니라 회계 지식이다.

### 정상에서 조용한가 — 발화율 대역

검증의 첫 관문은 부정 탐지가 아니라 **정상에서의 침묵**이었다. 룰이 정상 원장의 30%에 발화하면 검토 목록이 될 수 없다.

핵심 장치는 **측정 전 대역 선언**이다. 실측한 뒤 그 숫자에 맞춰 기준을 정하면 아무것도 증명하지 못한다.

대역 밖은 자동 불합격이 아니라 조사 대상이며, 결과를 ①생성기 결함 ②룰 과탐 ③대역 오선언으로 3분류한다. ③으로 판정하려면 도메인 근거가 필수다.

결과는 **통과 27 / 실패 1**이다. 실패한 절대 고액 룰은 두 원인의 복합이었다.

| 원인        | 내용                                                     |
| ----------- | -------------------------------------------------------- |
| 룰 결함     | 마감분개 식별 키 불일치로 순이익이 잘못 계산돼 임계 붕괴 |
| 생성기 결함 | 비용이 매출의 3.4배인 손익 구조                          |

수정은 별도 라운드로 이연하고 **판정은 실패로 기록**했다. 통과 27/28로 남기고 감추지 않는 것이 이 검증의 규칙이다.

### 죽은 룰 3종을 되살린 라운드

정상에서 0건인 룰이 12종 있었다. 0건은 "정상이 깨끗하다"일 수도 있고 "룰이 죽었다"일 수도 있어 전수 재점검했다.

| 룰            | 0건이었던 이유                                   | 수정                           |
| ------------- | ------------------------------------------------ | ------------------------------ |
| 수익 컷오프   | 통제이전일이 입고 문서 전용이라 매출 전표에 없음 | 출하·입고 기준 통제이전일 복원 |
| 증빙 괴리     | 생성 분포가 ±29일로 잘려 30일 초과가 원천 불가   | 31~90일 지연 꼬리 주입         |
| 가계정 미정리 | 정리 신호 컬럼 자체가 원장에 없음                | 정리 여부 컬럼 신설            |

셋 다 실제 회사에는 있어야 할 현상이 생성기 구멍으로 0이었다. 처음에는 "정상이니까 0이 맞다"고 대역을 재선언하려 했다.

"정상 데이터에 컷오프가 한 건도 없는 게 이상하다"는 지적을 받고 다시 열어 생성기 갭으로 재분류했다. 셋 다 수정 후 사전 선언 대역 안에 들어왔다.

### 쏴야 할 때 쏘는가 — 단위시험

정상에서 조용한 것만으로는 부족하다. 룰이 고장 나 있어도 조용하기 때문이다.

| 항목 | 내용                                          |
| ---- | --------------------------------------------- |
| 배경 | 정상 base의 1개년 슬라이스 119,205행          |
| 주입 | 룰별 최소 위반 사례 74행 / 36문서             |
| 분모 | 탐지기 레지스트리에서 스크립트로 추출해 고정  |
| 판정 | 표적 적중 — "발화했지만 다른 전표에서"는 실패 |
| 결과 | **29 / 29, 룰 결함 0건**                      |

1차는 26/29였다. 미발화 3건을 추적한 결과 **전부 시험 장치의 결함**이었다.

| 룰        | 1차 미발화 원인                                              |
| --------- | ------------------------------------------------------------ |
| 무효 계정 | "미등재"로 심은 계정이 실제로는 계정과목표에 등재돼 있었음   |
| 중복 전표 | 중복 키가 시각까지 비교하는데 1시간 차로 주입                |
| 업무범위  | 점수 비병합 룰이라 점수 채널이 항상 0 — 판정기가 한쪽만 읽음 |

세 번째가 교훈적이다. **점수 0이 침묵을 의미하지 않는다.**

이 룰은 설계상 점수를 싣지 않고 검토 채널에만 신호를 보낸다. 측정 도구가 제품 구조를 모르면 멀쩡한 룰을 고장으로 판정한다.

### 부정이 검토 목록에 오르는가

정상 base는 그대로 두고 부정 330문서를 얹은 데이터셋을 시드만 바꿔 4벌 만들어 측정했다. 측정 항목은 부정 적중률이 아니라 **감사인이 볼 목록에 오르는 비율**이다.

| 시드   | 표면 커버리지     |
| ------ | ----------------- |
| 대표본 | 197 / 330 (59.7%) |
| 시드1  | 203 / 330 (61.5%) |
| 시드2  | 204 / 330 (61.8%) |
| 시드3  | 204 / 330 (61.8%) |

프리셋별로 보면 밀도 차이가 크다.

| 프리셋      |   적중 | 표면 규모 |       밀도 |
| ----------- | -----: | --------: | ---------: |
| 비용자산화  |  20~25 |       232 |  8.7~10.6% |
| 수익인식    |   6~10 |       260 |   2.3~3.8% |
| 역분개·은폐 | 85~103 | 약 10,800 |   0.8~1.0% |
| 추정·관계사 |   7~11 |     5,505 | 0.13~0.20% |

좁게 고르면 밀도가 높고 넓게 고르면 건수가 많다는 트레이드오프가 수치로 드러난다. 비용자산화 프리셋은 232건 중 20여 건이 부정이므로 무작위 대비 약 30배 농축이다.

이 판단을 도구가 대신하지 않고 감사인에게 건수로 보여주는 것이 설계 의도다.

### 표면에 오르지 않은 40%

| 원인               | 내용                                                       |
| ------------------ | ---------------------------------------------------------- |
| 위장 컴포넌트      | 라벨은 부정이나 전표 단독으로는 규칙 위반이 아닌 다리 전표 |
| 매출 이상치 무반응 | 총량 조작을 전표로 쪼개 개별 금액이 정상 분포 안           |
| 순환거래 22%       | 전표 하나하나는 정상 매출·매입 모양                        |
| 유령급여 33%       | 정상 급여 패턴 모사라 전표 모양에 구분점 없음              |

대부분이 설계 의도와 일치한다. 전표 룰이 안 올리는 것이 맞는 경우다.

### 검증이 실제로 잡은 결함

| 결함                        | 잡은 축             | 효과                   |

첫 번째는 이중 레지스트리 중 한쪽만 갱신돼 삭제된 분류를 참조한 경우다. 관계사 룰의 발화가 표면 진입 전에 전량 탈락하고 있었다.

구 등급 체계에서는 이 룰이 보조 신호라 증상이 안 보였고, 빌더 어휘로 승격되면서 드러났다.

## 5. 트러블슈팅 · 핵심 의사결정

### 의사결정 ① 목표를 과하게 잡았다

초기 설계는 룰 기반 탐지와 여러 계열의 구조 분석을 함께 돌리는 구성이었다. 관계사 대사, 순환거래 그래프 분석, 거래처 관계망 분석이 각각 별도 계열로 있었다.

**하나도 제대로 작동하지 않았다.** 원인은 구현 실력이 아니라 전제였다.

| 계열          | 왜 불가능했나                                                 |
| ------------- | ------------------------------------------------------------- |
| 관계사 대사   | 회사를 가로질러야 성립 — 단일 법인 원장에는 상대 장부가 없음  |
| 순환거래 원환 | 자금 네트워크 전체가 보여야 성립 — 은행 계좌망·다사 장부 전제 |
| 거래처 관계망 | 거래처 마스터 필요 — 원장 단일 소스에는 없음                  |

감사 계약은 법인 단위다. 그룹 감사는 별개 절차이고 데이터 접근 권한도 다르다.

**결정: 단일 법인으로 확정하고 해당 계열을 삭제했다.** 주석 보존도 하지 않았다.

대신 원장 집계만으로 만들 수 있는 약한 근사를 남겼다. 거래처 첫등장·희소·휴면이 여기 해당하며, 커버리지로 주장하지 않는다.

부수 효과가 하나 있었다. 통계형 룰 17개 중 **7개가 회사 코드를 그룹 키에 넣지 않고** 있었다.

다법인 데이터에서는 회사별 권한이 합쳐져 실재하지 않는 위험 조합이 만들어진다. 전수 조사해 목록으로 기록했다.

### 의사결정 ② 등급을 매기려 했으나 기준이 설명되지 않았다

룰 조합에 높음·중간·낮음 등급을 자동 부여하는 체계를 만들었다. 근거는 감리 지적사례였다.

검증을 위해 감리 사례를 엄격 기준으로 재태깅했다(731행). **지지가 무너졌다.**

| 구 상위 조합    | 감리 확정 건수 |
| --------------- | -------------: |
| 기말 × 추정계정 |          12~22 |
| 나머지 전부     |            0~3 |

원인은 자료의 장르 한계였다. 감리 문서는 "매출을 과대계상했다"는 적지만 "그 전표를 수기로 심야에 넣었다"는 적지 않는다.

따라서 입력 방식 신호가 낀 조합은 원문으로 **영구히 실증할 수 없다**. 남는 근거는 조합별 전문가 특례뿐이었고, 조합마다 다른 사연은 방어 불가능한 휴리스틱이다.

합성데이터로 메울 수도 없었다. "정상에서 이 조합이 드물다"는 우리가 생성기에 넣은 파라미터의 메아리다.

**결정: 등급 자동판정 전면 폐지, 조합 빌더로 대체.**

| 구분      | 이전               | 이후               |
| --------- | ------------------ | ------------------ |
| 판정 주체 | 시스템이 등급 선언 | 감사인이 조합 선택 |
| 어휘      | 전체 룰 조합       | 근거 있는 20개만   |
| 코드      | 등급 산출 662줄    | 193줄로 재작성     |
| 방어 대상 | 조합별 특례 다수   | 없음               |

시스템이 등급을 선언하지 않으면 방어할 휴리스틱이 0이 된다. 감사기준서 240에서도 분개 테스트 항목 선정은 감사인 판단 사항이다.

### 의사결정 ③ 비지도 모델을 끝내 못 살렸다

여러 구성을 시도했다. 피처 조합을 바꾸고, 차단 목록을 조정하고, 평가 단위를 행에서 문서로 올려봤다.

**판별력 0.5187에서 움직이지 않았다.** 무작위와 같은 수준이다.

원인을 분석한 끝에, 합성데이터 경로로는 이 검증이 **원리적으로 불가능**하다는 결론에 이르렀다. 흔적을 심으면 자문자답이고, 안 심으면 정보가 없어 못 잡는다.

**결정: 실패로 기록하고 역할을 축소했다.** 과거의 0.93도 자문자답 수치로 명시 무효화했다.

이 실패가 구조를 흔든 것인지 확인하려고 빅4 회계법인 도구를 조사했다.

| 도구           | 방식                           |
| -------------- | ------------------------------ |
| EY Helix GLAD  | 감사인 라벨 기반 지도학습      |
| KPMG Clara     | 룰 40여 종 위에 ML 보조 앙상블 |
| PwC Halo       | 룰 묶음 + 의심 점수            |
| Deloitte Omnia | 기준 기반 분석                 |

네 도구 모두 룰·기준이 주력이고 ML은 보조다. **비지도 딥러닝 단독 탐지는 실무에 없다.**

실패한 검증이 도구를 표준에서 벗어나게 만든 것이 아니라, 실무가 왜 그렇게 생겼는지를 실측으로 재확인한 셈이다.

### 운영 사고

디버깅 기록 8,177줄을 전수 정독해 사고 91건을 추출했다. 대표 사고는 다음과 같다.

| 사고                     | 피해                   | 근본 원인                                  | 재발 방지                           |
| ------------------------ | ---------------------- | ------------------------------------------ | ----------------------------------- |
| 설정 손작성 12섹션 유실  | 금액 1,223배 축소      | 섹션 누락이 에러가 아니라 외화 기본값 적용 | 정본을 읽어 최소 변경, 섹션 수 확인 |
| 교차검증 한 달 디버깅    | 재생성 20회            | 덧대기 패치가 다음 사고의 원인이 됨        | 근본 원인 수정 원칙                 |
| 합성데이터 지름길        | 단일 컬럼 판별력 0.998 | 규칙으로 심은 부정이 컬럼에 흔적을 남김    | 학습 입력 차단 목록                 |
| 측정 도구가 다른 경로    | 결론 전면 정정         | 측정 하니스가 제품과 다른 설정을 탐        | 검증 인프라도 검증 대상             |
| 직무분리 룰 과탐         | 110만 건 중 110만 건   | 한국 조직 구조에 안 맞는 조합 정의         | 감리 사례 189건 실증 분석           |
| 계정과목표 누락 3회 재발 | 6,178행 과탐           | 같은 파급 패턴 반복                        | 자동 게이트 상설화                  |
| 원본 파일 파괴           | 복구 불가              | 데이터 폴더가 버전 관리 밖이라 이력 0      | 덮어쓰기 전 추적 여부 확인          |

#### 대표 사고 흐름 — 설정 12섹션 유실

```
  [의도]  생성 설정을 조금 바꾸자
     │
     ▼
  [행동]  설정 파일을 손으로 새로 작성        ◀── 분기점
          정본 17섹션 · 손작성본 5섹션
     │
     ▼
  [침묵]  12섹션 누락 ──▶ 에러 없음
          누락 필드에 기본값이 조용히 채워짐
          금액 분포가 원화(14.0) ──▶ 외화(7.0)
     │
     ▼
  [확산]  98만 행 생성 완료 · 게이트 일부 통과
          매출 중앙값 1,000원 (정본 681,151원)
     │
     ▼
  [발견]  금액 분포가 이상하다는 육안 확인
     │
     ▼
  [복구]  정본 설정으로 재생성 — 생성기 코드 수정 0줄
          코드는 처음부터 옳았다
```

이 사고가 두 번째였다는 점이 중요하다. 첫 번째는 세금 섹션이 빠져 생성 단계 하나를 통째로 건너뛴 사건이었다.

같은 병이 재발했다는 것은 개인의 부주의가 아니라 **설계가 그 실수를 유도한다**는 뜻이다.

#### 반복된 실패 패턴

| 패턴               | 내용                                                      |
| ------------------ | --------------------------------------------------------- |
| 조용한 기본값      | 누락이 에러가 아니라 기본값으로 채워짐                    |
| 스테일 산출물      | 빌드가 무동작이거나 캐시가 남아 "고쳤는데 안 고쳐진" 상태 |
| 측정 도구의 결함   | 측정 하니스·게이트·시험 장치가 각각 틀림                  |
| 완료 선언의 조기화 | 배선이 비었는데 완료로 기록                               |
| 예상 밖 지름길     | 문서 ID 길이, 소스 필드 분포 등 의미 없어 보이는 필드     |

세 번째가 가장 비쌌다. 게이트 4종은 항등식이었고, 룰 시험 장치는 멀쩡한 룰 3개를 고장으로 판정했다.

**검증 인프라도 검증 대상**이라는 결론이 여기서 나왔다.
